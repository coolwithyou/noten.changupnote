# 창업노트 — 매칭 & 스코어 모델 설계

작성일: 2026-06-19 · 역할: `company_profiles` × `grant_criteria` → 적격 판정 + 점수 + 설명 + 다음 질문

## 0. 설계 원칙

1. **단일 만족도 점수 금지.** "만족도 87점" 하나로 뭉치면 *신뢰 가능한 적합도*와 *추측성 선정가능성*을 섞어 오해를 부른다. 점수를 **3분해**해 각각 신뢰도와 함께 보여준다.
2. **결정론(규칙) ↔ 통계(학습) 분리.** 적합도는 규칙엔진(설명가능·확정), 선정가능성은 데이터가 쌓인 뒤 학습. 추측이 확정 출력을 오염시키지 않게 격리.
3. **신뢰도 흡수.** 추출 confidence·프로필 출처(팝빌 vs 자가신고/예비)를 점수에 반영. 모르면 거짓확신 대신 "확인 필요".
4. **모든 점수는 버전 파라미터.** 가중치는 매직상수가 아니라 `scoring_ver`로 관리, 골든셋·피드백으로 튜닝.

---

## 1. 매칭 파이프라인 (회사 1 × 후보 공고 N)

```
Stage 0  후보 검색 (싸게 좁히기)
  grants.f_* 투영 + pgvector로: status=open ∧ (지역 overlap ∨ 전국) ∧ 업종 rough overlap
  → 전수 평가 대신 수십~수백 후보만 다음 단계로

Stage 1  적격 판정 (grant_criteria 순회)
  exclusion 매칭 → 즉시 ineligible (hard stop)
  required: 각각 pass | fail | unknown
     fail 존재 → ineligible
     unknown 존재(겸 fail 없음) → conditional(조건부)
     전부 pass → eligible
  preferred: 충족분 가중치 누적

Stage 2  스코어링  → 적합도 / 선정가능성 / 가치 (아래)

Stage 3  랭킹 + 설명(rule_trace) + 다음질문(정보이득)
```

---

## 2. ㉠ 적합도 점수 (Fit) — *지금 신뢰 가능, 헤드라인*

```
base = eligible    → 100
       conditional → 60~95 밴드 (확정된 필수 비율로)
       ineligible  → 목록 제외 또는 별도 "부적격" 영역

pref_bonus = Σ(충족 preferred.weight) / Σ(전체 preferred.weight)   # 0~1

fit_raw  = w_base·base + w_pref·pref_bonus·scale
fit_score= confidence_weight · fit_raw
```
- **confidence_weight** = 각 criterion 판정에 `min(criterion.confidence, profile_field.confidence)`를 곱해 집계. 자가신고/예비 프로필·낮은 추출 신뢰도면 자연히 하향.
- **base가 pref를 압도**하도록 가중(`w_base ≫ w_pref`) — 가점 많은 *간신히 적격*이 *깔끔한 적격*을 앞서면 안 됨.
- **conditional 밴드**: 확정 하한(현재 충족 필수)과 잠재 상한(unknown 모두 충족 가정)을 함께 산출 → "조건부 78점, 매출 입력 시 최대 92점".

## 3. ㉡ 선정 가능성 (Competitiveness) — *데이터 게이트, v1+*

신청 시 선정될 확률. 초기엔 **데이터 부족 → 표시 보류 또는 "추정" 명시**(거짓확신 금지).
- 입력(가용 시): 경쟁률(지원자/선발 공시), 예산 규모, 우대 충족 강도, **플라이휠 outcome**(유사 기업 선정 이력).
- 초기 휴리스틱 → outcome 누적 후 학습 모델로 승급. 항상 신뢰구간·"추정" 라벨.

## 4. ㉢ 가치 점수 (Value) — *부분 가용*

```
value = norm(support_amount) · 적합도 / 준비부담
   준비부담 ≈ f(첨부서류 수, 접수방법 난이도, 사업기간)
```
- 우선순위 정렬 보조. 지원금은 확정, 준비부담은 휴리스틱(라벨링으로 개선).

## 5. 종합 표현 (UX 계약)

- **헤드라인 = 적합도**(신뢰 가능). 선정가능성·가치는 **보조 배지**로, 추정은 "추정" 라벨.
- 단일 합성점수를 굳이 쓴다면 *적합도 중심 + 보조 약가중*, 그리고 항상 3분해를 펼쳐볼 수 있게.
- `eligible / 조건부 / 부적격`을 명확히 구분 표기.

## 6. unknown → 정보이득 → progressive 다음 질문 *(핵심 연결)*

각 unknown(결측 필드)에 대해 **정보이득**을 추정:
```
gain(field) = Σ over 조건부 공고 g:  (해당 필드가 g의 unknown을 해소할 확률)
                                    · (해소 시 g가 eligible 될 기대)
                                    · (g의 가치·적합도 가중)
```
→ gain 최상위 필드 1개를 다음 질문으로 노출. 토스식 1화면 1질문 + 즉시 가치 피드백의 엔진.
→ 예비창업 수기 입력 순서도 동일 원리로 정렬.

**⚠ 양방향 프레이밍 (검증 반영)**: unknown이 해소되면 fail로 판명돼 ineligible로 *떨어질 수도* 있다. 메시지를 상승 일변도("+3건 확정")로만 하면 "입력했더니 추천이 사라지는" 신뢰 훼손. → **"확정 또는 제외에 도움"** 양방향 표현, conditional 카드에 하방 가능성 명시. `gain`도 부적격 확정의 정보가치(시간 절약)를 포함.

**콜드스타트 prior**: 데이터 없는 초기엔 `gain`의 세 확률을 동일 dimension의 충족 분포(예: 전체 공고에서 해당 조건 충족 비율) prior로 추정 → outcome 쌓이면 갱신.

**match_events 멱등 (검증 반영)**: (company×grant)는 **현재 상태 `match_state`(upsert)** 와 **노출/평가 이력 `match_events`(append)** 로 분리. progressive 재매칭은 match_state upsert. `ruleset_ver` 변경 시 재매칭은 **변경된 criteria·프로필에 영향받는 (company×grant)만** 무효화(전건 N×M 금지 — 비용 가드).

## 7. 설명 (rule_trace → 자연어)

per-criterion 기록: `{dimension, kind, result, source_span, company_value}` → 문장화:
- "업력 7년 이내(필수) — 귀사 4년 2개월 ✅"
- "비수도권(우대) — 귀사 서울 ➖ 미해당"
- "운영사 추천 필요(필수·자동판정 불가) — ⚠ 원문 확인"

설명은 신뢰의 핵심이자 **라벨링 단위**(사용자가 "이 판정 틀렸다" 피드백 → 골든셋).

## 8. 신뢰도 · 임계값 · HITL

- `match_confidence = f(criteria conf, profile conf, 커버리지)`.
- 골든셋으로 임계값 보정 → 미만이면 "확인 필요" 표기 + HITL 큐(필요 시).
- 예비/자가신고는 기본 낮은 신뢰 → "확인 필요" 비중↑.

## 9. 버전 · 튜닝 파라미터

`ruleset_ver`(criteria 평가 로직) + `scoring_ver`(가중치 `w_base,w_pref,scale`, value 계수, 임계값) 를 `match_state`와 `match_events`에 부착 → 현재 상태와 노출/행동 이력 모두 품질 변화 귀속 가능. 가중치는 골든셋 회귀 + 피드백으로 조정.

## 9.5 데이터 누적에 따른 정교화 (학습 구조) ★
**원칙**: 적격 판정(필수/배제)은 **결정론적·설명가능 유지**(블랙박스 금지 — 원칙 #2·#3). 학습은 그 *위*의 랭킹·점수·선정가능성과 추출·매핑 품질에서 일어난다. → "더 똑똑해지되, 적격 판정은 늘 설명 가능."

데이터가 쌓일수록 정교해지는 **4개 경로**:
1. **추출 정확도** ← 골든셋·HITL 라벨 → 프롬프트/few-shot 개선 → grant_criteria 품질↑ → *판정 입력 자체가* 정확해짐.
2. **매핑 자산** ← 라벨링에서 발견된 누락(업종 태그·지역 표현) → `industry_taxonomy` 등 보강 → 분류 정확도↑.
3. **스코어 가중치 (learn-to-rank)** ← 사용자 피드백(적합/부적합)·암묵신호(클릭·저장)·outcome → 적합도 보너스·기업적합·의미유사 **가중치를 데이터로 튜닝**. 초기 규칙기반 → 점진 학습.
4. **선정가능성 모델** ← 신청→선정 outcome 누적 → 경쟁률·적합강도 기반 추정→학습 모델(희소 데이터라 가장 늦게).

모든 개선은 **버전·평가 게이트로 회귀 없이만 반영**(플라이휠). 적격은 안정·설명가능하게 두고, *추천 순서·강조·정밀도*가 시간이 갈수록 좋아진다.

## 10. 평가지표 (플라이휠)

| 대상 | 지표 |
|---|---|
| 추출 | 필드별 precision/recall vs 골든 |
| 적격 판정 | eligible/ineligible/conditional 정확도 vs 사람 라벨 |
| 랭킹 | precision@k, MRR (초기 라벨) → apply rate·선정율 (outcome) |
| 신뢰도 | 임계값별 실제 오류율(calibration) |

## 11. 예비창업 특이사항
- 프로필 전량 자가신고 → `profile.confidence` 낮음 → fit_score·match_confidence 자연 하향, "확인 필요" 비중↑.
- 보강 단계(팝빌) 없이 §6 정보이득 순서로 수기 입력을 유도.
- size=`예비` criteria 사업 중심으로 후보가 좁혀짐(Stage 0).

## 12. 미결정
- `w_base/w_pref` 초기값과 conditional 밴드 폭 → 골든셋 확보 후 1차 튜닝.
- 선정가능성 휴리스틱 도입 시점(outcome 표본 임계).
