# 창업노트 매니저 — 신청 준비 워룸 (Phase 2 플래그십)

작성일: 2026-06-26 · 포지션: 발견·매칭(MVP) *다음*의 "실행/준비" 레이어. deck의 LUMEN(AI초안)+전문가연결(코칭)+신청 준비 시트를 하나로 묶는 작업형 SaaS.

## 0. 한 줄
지원사업 신청 *준비를 끝내는* 캔버스형 워룸. 자동제출이 아니라, 필요 서류·업로드 파일·Q&A·AI 초안·코칭이 **노드로 연결된 작업 공간**에서 "서류 세트를 완성"한다.

## 1. 핵심 설계 결정 (평가 4 보정 — 반드시 준수)
1. **자유 캔버스 ❌ → 자동 레이아웃 *오피니언 그래프*.** 사용자가 노드를 배치하는 화이트보드가 아니다. 시스템이 `회사 → 공고 → 필요서류 → (파일/답변/초안)`을 *완료 상태와 함께 자동 배치*(Sankey/의존 그래프 결). "다 했나?"가 한눈에 읽혀야 함.
2. **멀티 신청 오케스트레이션 파워뷰.** 그래프는 *여러 공고가 많은 서류를 공유*할 때 빛남. 1~2건 준비 유저는 공고별 **신청 준비 시트**(기존)로 자동 전환. 둘 다 제공.
3. **Phase 2 간판 기능 (시퀀싱).** 발견·매칭 MVP 출시 *후*. 여러 선행 조각에 의존(아래 §6).
4. **`document_taxonomy` 선결.** "공통 서류 노드 → 여러 공고"는 *표준 서류 타입*이 있어야 dedupe/공유됨. (데이터표준화 §8.6)

## 2. 노드 타입 & 엣지 의미 (그래프 모델)
| 노드 | 내용 | 데이터 |
|---|---|---|
| 회사 프로필 | 업력·지역·업종·규모 | `companies`·`company_profiles` |
| 지원사업 | 사업명·기관·D-day·적합도·상태 | `grants`·`match_state` |
| 필요 서류 | **표준 서류 타입**(사업자등록증·재무·4대보험·사업계획서…) | `document_types` |
| 업로드 파일 | 사용자가 올린 실제 파일 | `uploaded_files`(R2) |
| Q&A 답변 | 사업개요·문제정의·시장성·기술성·고용 | `qa_answers` |
| AI 초안 | 공고별 신청서/사업계획서 초안 | `ai_drafts`(LUMEN) |
| 코칭 코멘트 | 검토 필요·보완 요청·승인 | `coach_comments` |

**엣지 의미** (색으로 상태):
- 서류 노드 = `document_type` 단위 → **여러 공고로 fan-out**(공통 서류 한 번 준비 = 다수 공고 진행). 핵심 가치.
- 파일 ↔ 서류: 매칭되면 선 활성 + 체크. 미충족 = **amber**, 완료 = **green**, 일반 = **muted blue**.
- Q&A 답변 ↔ AI 초안: 다대다.

## 2.5 그래프 레이아웃 규칙 (자유 캔버스 → 구조화) ★
레퍼런스: `창업노트_매니저_그래프_레퍼런스.html` (라이트 work-zone 보정본)
1. **스윔레인(타입별 열) + 좌→우 의존 흐름**: 내 사업자 → 지원사업 → 필요 서류 → 준비물·검토. 노드는 레인에 *스냅*(자유 배치 금지).
2. **자동 레이아웃**(dagre식 layered) — 시스템 배치, 사용자는 팬/줌·포커스만.
3. **회사 = 단일 루트(좌측 앵커)** → 방향감.
4. **엣지 = 상태색**(완료 green / 누락 amber / 일반 muted), 교차 최소·번들.
5. **포커스 + 접기**: 공고 클릭 → 해당 서브그래프만 강조, 나머지 흐림. 노드 폭주 방지(서류 "5/7" 배지 요약 → 펼치면 상세).
6. **노드 = 콤팩트 상태칩**(큰 이미지 X), 상세는 우측 인스펙터. 정보 밀도.
7. **라이트 work-zone**(은은한 점그리드) + 레인 헤더 = 업무 도구. 다크 AI툴 클리셰·자유 무한캔버스 ❌.
> 핵심: *토폴로지보다 준비 상태가 먼저 읽혀야.* "사업자등록증 1개 → 3개 공고 재사용" 웹이 한눈에.

## 2.6 인터랙션 모델 ★ (직접 조작 캔버스 — 드래그·편집·줌·판단)
캔버스가 리스트보다 나은 이유 = **넓게 보고 한눈에 판단**(전체 준비 상태를 동시 조망). 핵심 구분:
- **계산된 연결(스켈레톤) = 자동.** 공고↔필요서류는 시스템이 아는 사실(추출) → 사용자가 드래그하지 않음. 자동으로 그려짐.
- **사용자 연결(드래그) = 직접 조작의 자리.** 내 자료(업로드 파일·Q&A 답변·AI 초안)를 *슬롯에 연결*하고 **공고 간 재사용 배정**. 노드 포트(핸들)에서 드래그해 잇기. ("사업자등록증.pdf → 서류 노드 → 3개 공고 재사용")

핵심 인터랙션:
1. **드래그 연결**: 자료 → 서류 슬롯, 재사용 배정(포트 핸들·베지어). *계산된 요구 연결은 드래그 대상 아님*.
2. **인라인 편집**: 노드에서 답변 보강·초안 수정(또는 우측 인스펙터에서 넓게).
3. **줌아웃 오버뷰 + 포커스**: 전체 준비 상태 한눈(판단) / 공고 클릭 시 서브그래프만 강조.
4. **자동 레이아웃 = 기본값**, 자유 팬·줌·드래그배정·미세 정렬 허용. 흐트러지면 **"정렬" 버튼**으로 재배치.
5. **판단 앵커 상시 노출**: 상단바 요약(누락 4·마감 2)·완료율은 캔버스 상태와 무관히 항상 표시.
> 자유 무한캔버스(❌)와 빡빡한 자동전용(❌)의 중간 = **"구조화된 스켈레톤 + 조작 가능한 자료"**.

## 3. 화면 구조 (브리프 반영, 자동그래프로 정리)
- **상단 바**: "창업노트 매니저" · 사업자명/번호 · 상태 요약(지원 가능 5·준비 중 3·누락 서류 4·마감 임박 2) · CTA(서류 세트 다운로드·코치 초대·공유).
- **좌측 패널**: "이번 주 할 일" 액션 큐(영향 공고 수·마감 위험 표기) + 필터(전체/적격/확인 필요/준비 필요/마감 임박). ← NPC 액션 큐(매칭 §6.1) 재사용.
- **중앙 캔버스**: 위 자동 레이아웃 그래프.
- **우측 인스펙터**: 선택 노드 상세(예 "2026 초기창업패키지": 준비 중·D-6·서류 5/7·연결 답변 4·초안 상태·버튼[초안 열기/누락 서류/이 공고 ZIP]).
- **다운로드**: 통합 ZIP / 공고별 ZIP / 공통 서류 폴더 / 공고별 AI 초안 / 누락 서류 목록 / 신청 링크·마감 요약 PDF.
- **코칭**: 노드 코멘트 핀 · "코치 검토 필요" 상태 · 코치 아바타/활동.

## 4. 데이터 모델 추가 (DB)
`document_types`(서류 표준 타입·카테고리·source·재사용여부) · `grant_required_docs`(grant × document_type, required) · `uploaded_files`(company·R2 ref·인식 타입) · `file_doc_links`(파일↔필요서류 매칭·검증) · `qa_answers`(company·question·answer) · `ai_drafts`(company·grant·type·상태) · `coach_comments`(node ref·상태·코치) · `applications`(company×grant 준비 상태·진척).

## 5. 가드레일 (원칙 #2·#7와 일치)
- **자동제출 ❌ / 정부사이트 대리조작 ❌** — 딥링크·서류 완성까지만.
- 무질서한 화이트보드 ❌ — 자동 레이아웃·완료 상태 가독.
- 장식 노드 ❌ — 실제 준비 상태가 읽혀야.
- 마케팅 랜딩 ❌ / 다크 AI 툴 클리셰·과한 보라 그라데이션 ❌.

## 6. 의존성 · 시퀀싱 (Phase 2)
선행: ①MVP 매칭(Phase 1) ②`required_documents` 추출 + **`document_taxonomy`**(Phase 1.5) ③파일 스토리지(R2) + 파일↔서류 인식 ④AI 초안(LUMEN, deck IN APP) ⑤코칭(전문가 연결). → 매니저는 이들 위의 통합 뷰.

## 7. 톤
브리프대로 **클린 업무형**(refined-minimal, 블루 중심+민트 보조, 반경 14–16px, 은은한 그리드). **발견(게임적·카드)↔준비(업무적·워룸) 모드 분리** — 작업 화면은 차분·신뢰. 우리 비주얼 톤 결정과 일치.
