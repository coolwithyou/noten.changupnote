## 1. Phase 0 — 검증 스파이크 (가장 먼저, 구현 아님)
> 목적: 가장 큰 불확실성을 *데이터로* 먼저 깬다. 각 스파이크는 숫자/결론 산출.

| 스파이크 | 질문 | 산출 |
|---|---|---|
| ✅ S0-A K-Startup 실응답 | 자격요건이 별도 필드로 구조화? | **완료: 그렇다.** 신엔드포인트 기준 지역·업력·연령·분야·대상·일정 구조화. 추출 低난도. (공고API §4) |
| ✅ S0-B HWP 추출 PoC | HWP 표 자격요건을 보존 변환할 수 있나 | **완료: pyhwp `hwp5html` 채택.** 표 보존 확인, 스캔본만 OCR 잔여 |
| ✅ S0-C1 LLM 추출 PoC | 실공고 텍스트가 tool-use로 안정 추출되나 | **완료: n=5 정성 검증.** nuance·배제·OR·text_only 동작 확인 |
| ⬜ S0-C2 골든셋 P/R | 50~100건 도메인 라벨 대비 정량 정확도는? | **팀 작업 필요:** 추출 P/R %, 원천 결측률 %, 적격판정 P/R |
| 🟡 S0-D 팝빌 연동 | `checkBizInfo` 실제 커버리지·런타임 IP 제약·캐시 가능 범위 | **로컬 호출 성공:** `corpScaleCode/industryCode/establishDate/addr/closeDownState/closeDownTaxType` 채움. 남은 것: Vercel 실행 위치·캐시 약관 |
| ⬜ S0-E dedup | K-Startup↔기업마당 중복 실측 | 중복률, 키 후보, 보수적 임계값 |
| 🟡 S0-F 신청양식 구조 | 정부지원사업은 *주로 어떻게* 신청서를 작성하나? | **실측(K-Startup n=1000, `poc/s0f_application_probe.py`):** 신청방식 온라인 65.7%·이메일 35.2%·방문 5.1%, 공고당 1방식 91.7%. ⚠ **온라인 접수처 225종 파편화(forms.gle 138·구글폼·기관포털 제각각) → 통합 자동제출 불가.** 사업계획서 양식은 API 첨부에 없음(상세페이지 크롤 별도). → **지원서 도움 = 사업계획서 초안 + 프로필 복붙 자동완성만, 자동제출 ❌**(대리신청 금지가 기술적으로도 강제) |
| 🔴 S0-G 수혜이력 소스 | 번호로 수혜이력을 얼마나 자동 파악하나? | **실측: 공고 API 경로 불가 확정.** 공고 3종 + `getBusinessInformation`(연간 사업소개·선정자 아님) + 선정결과 오퍼레이션 4종 타진 → **선정자 명단 없음**(3종 HTTP 400). `prior_award` '제외 조건'만 공고 텍스트 6.1% 존재(누가 받았나 ≠). **NTIS/SMTECH는 키 미발급으로 미실측**(가능성 열림 → 별도 S0-G-2). → **MVP 수혜이력 = 자가신고 우선**, 선정결과 역인덱스 보류 |

**Exit**: S0-A/B/C1로 구현 착수 리스크는 낮아졌지만, 운영 정확도 주장은 S0-C2 골든셋 P/R 전에는 확정하지 않는다. 성립 안 하면 HITL 비중·범위 재설계.
**S0-F/G 실측 결론(2026-06-26)**: ① **자동제출·통합폼은 불가**(접수처 225종 파편화) → 지원서 도움은 *사업계획서 초안 + 복붙 자동완성*으로 한정. `grant_application_spec`은 `method` 분포·`plan_template_type`만 의미 있고 form_fields 자동연동은 폐기. ② **선정결과 역인덱스는 현재 자원으로 불가**(공고 API에 선정자 명단 없음) → `grant_outcomes` 자동수집 스키마는 보류, **수혜이력은 자가신고 우선**. NTIS R&D 보강만 키 발급 후 별도 검증(S0-G-2). *스키마를 박기 전에 실측해서 두 자동화 범위를 정직하게 축소한 것이 이 스파이크의 성과.*

## 2. Phase 1 — MVP (기창업 정부지원사업)
**범위**: K-Startup+기업마당 수집 → 추출 → 규칙매칭 → 번호 기반 조회/로드맵. 정책자금·민간투자 제외.

> **S0-A 반영 — K-Startup 우선 wedge.** K-Startup은 자격요건이 구조화 필드라 **필드 파싱 위주(저위험)**로 먼저 매칭을 세우고, 기업마당(HWP·본문 비정형, 高난도 LLM 추출)은 같은 Phase 내 후속으로. 예비창업은 `biz_enyy`에 네이티브 태깅돼 K-Startup 트랙에서 사실상 무료로 포함.

| 레이어 | 작업 | Exit 기준 |
|---|---|---|
| 데이터자산 | industry_taxonomy·region_hierarchy·size_thresholds 1차, 골든셋 50~100 | 표준값 안정 |
| Ingestion | 어댑터 3종·상태머신·증분·변환·추출·정규화·upsert | 일 1회 자동 수집·발행 |
| 스키마 | grants/grant_criteria/grant_raw/extraction_log/company_profiles/app_refresh_tokens/company_enrichment_cache | 계약 고정 |
| 매칭 | 규칙엔진(필수/우대/배제)·적합도 스코어·rule_trace·정보이득 | 골든셋 적격판정 P/R 측정 |
| 보강 | 팝빌 연동 + 필드별 캐시 + 소유권 검증(국세청 진위확인) + 자가신고 폴백 | 번호→프로필 |
| Serving | Next.js: 번호입력·검증·1차매칭·progressive·결과·로드맵 | 핵심 플로우 동작 |
| 인증 | NextAuth(이메일·Google·Kakao) | 로그인·저장 |
| **계정·권한·개인정보** | companies·user_company·인가/RLS·동의 원장·기본정보 처리방침 | PII 접근통제 동작 |
| 플라이휠 | 로깅·버전태깅·라벨링 큐·평가 하니스 CI | 변경 시 회귀측정 |

**Exit**: 실사용자가 번호 입력→정확도 측정된 매칭+로드맵 확인. 적격판정 P/R 목표치 합의.

## 5. 의존성·순서
```
S0-A(K-Startup 실응답) 완료 ─근거→ 추출 스키마·골든셋  (소스 구조 확인값 위에 골든셋 구축)
S0-D(팝빌 로컬 실커버리지 통과, Vercel 런타임·캐시 약관 잔여) ─게이트→ 보강 자동화 vs 자가신고 폴백
S0 → 표준화 자산+골든셋 → 스키마 고정(grants+company_profiles+계정/회사)
   → Ingestion ─┐
   → 매칭엔진 ──┼→ Serving 플로우 → 플라이휠 가동
   → 보강(팝빌)─┘
   (병행 필수) 계정·회사·권한(RLS/인가)·개인정보 동의  ← 신용정보 유입 전 완료
```
표준화·스키마가 모든 것의 선행. 플라이휠 로깅·버전태깅은 *처음부터*(후행 소급 불가). **계정/권한/개인정보는 매칭과 별개 트랙이 아니라 Phase 1 필수.**

## 8. 성공지표 (측정 가능)
- 추출: 필드 P/R 목표(S0 후 확정), 원천 결측률 추적.
- 매칭: 적격판정 P/R, 랭킹 p@k, 사용자 부적격 신고율↓.
- 커버리지: 활성 공고 수집·발행 비율, 예비 포함 세그먼트.
- 운영: 조회 지연(서빙), 추출 비용/공고, 자동:HITL 비율.
