### 1.4 우리 시스템만의 매칭 알고리즘

한 줄 정의: **AI는 공고를 읽어 구조화하고, 당락 판정은 결정론 규칙엔진이 한다.**

판정 규칙:

```text
1. required 또는 exclusion 중 하나라도 fail -> ineligible
2. fail은 없지만 required/exclusion 중 unknown 존재 -> conditional
3. 모두 pass -> eligible
```

구현 근거: `packages/core/src/matching/match.ts`

| 기능 | 현재 구현 |
|---|---|
| 결정론 판정 | `matchGrantCriteria(criteria, company)`가 criterion별 `pass/fail/unknown`을 산출 |
| 거짓 탈락 방지 | 결측·자동판정 불가는 `unknown`, `text_only`로 남김 |
| 설명 가능성 | `rule_trace`에 dimension, kind, operator, result, message, source_span, company_value 기록 |
| 버전 귀속 | `RULESET_VERSION`, `SCORING_VERSION`을 결과에 포함 |
| 조건부 다음 질문 | unknown 필드 우선순위에 따라 next_question 산출 |
| 점수 | eligible=100, ineligible=0, conditional=60~95 밴드 |

이 알고리즘의 차별점:
- LLM이 "추천/탈락"을 직접 말하지 않는다.
- 판정은 재현 가능하고 감사 가능하다.
- 모르는 것은 떨어뜨리지 않고 조건부로 남긴다.
- 모든 결과가 원문 근거와 회사값을 포함하므로 사용자 신뢰와 라벨링으로 연결된다.

## 2. 슬라이드 초안

### Slide 01. 제목

**공고를 읽는 AI가 아니라, 공고를 판정 가능한 데이터로 바꾸는 엔진**

창업노트는 흩어진 정부지원사업 공고를 표준 계약으로 정규화하고, 회사별 적격 여부·근거·다음 행동을 자동 산출하는 매칭 시스템입니다.

하단 보조 문구:
- 정부지원사업 공고 표준화
- 설명 가능한 매칭 알고리즘
- 데이터가 쌓일수록 정교해지는 성장 플라이휠

### Slide 02. 문제: 공고는 많지만, 판정 가능한 데이터가 아니다

핵심 메시지:
- 현재 열린 공고 유니버스는 약 1,900+건입니다.
- 하지만 공고는 K-Startup, 기업마당, 첨부 HWP, 상세 페이지, 외부 신청 폼으로 흩어져 있습니다.
- 사람이 읽을 수 있는 문서는 많지만, 기계가 판정할 수 있는 데이터는 부족합니다.

넣을 숫자:
- 기업마당 활성 1,588건 전수
- K-Startup open 304건
- K-Startup 누적 29,187+건
- HWP 자격요건 표, 비정형 본문, 신청 링크 파편화

### Slide 03. 해법: 모든 공고를 하나의 표준 계약으로 환원

핵심 메시지:
- 창업노트는 공고를 14개 자격요건 차원과 7개 혜택 패밀리로 정규화합니다.
- 이 계약 덕분에 "회사 데이터"와 "공고 데이터"가 같은 언어로 비교됩니다.

표현:
- 자격요건 14차원: 지역, 업력, 업종, 규모, 매출, 고용, 대표자 연령, 대표자 속성, 인증, 기수혜, IP, 신청대상, 영업상태, 기타
- 혜택 7패밀리: 자금, 융자·보증, 역량, 공간, 시장, 인증·IP, 연결
- 각 조건은 `required / preferred / exclusion`으로 분리

시각화 아이디어:
- 왼쪽: 비정형 공고 원문
- 가운데: 정규화 엔진
- 오른쪽: `grant_criteria[]`, `benefits[]`, `required_documents[]`

### Slide 04. 현재 기준 공고 전수 분석에서 얻은 인사이트

핵심 메시지:
- 지원사업의 가치는 "지원금"만으로 설명되지 않습니다.
- 실제 공고에서는 컨설팅, 판로, 전시, 인증, IP, 융자, 공간 같은 비금전 혜택이 자금만큼 자주 등장합니다.

넣을 데이터:
- 컨설팅·진단·자문 25.3%
- 전시회·해외진출·수출 24.1%
- 마케팅·판로·홍보 23.7%
- 인증·특허·IP 19.9%
- 자금 10.1%

결론 문장:
**그래서 창업노트는 공고를 "얼마 주는가"가 아니라 "무엇을 해금하는가"로 구조화합니다.**

### Slide 05. 수집·정규화 파이프라인: 소스별 난도를 다르게 푼다

핵심 메시지:
- K-Startup은 구조화 필드 중심으로 저비용 파서가 먼저 처리합니다.
- 기업마당은 HWP/본문 비정형이 많아 `HWP -> HTML -> Markdown -> Claude tool-use -> grant_criteria` 경로로 처리합니다.
- dedup, active filter, publish 검증을 통해 최신성과 중복을 관리합니다.

넣을 근거:
- K-Startup 표본 500건에서 지역·업력·연령·분야·대상·일정 100% 구조화
- 기업마당 HWP 표 11개 셀 단위 보존
- raw JSON 파싱 실패 1/3, tool-use 전환 후 실패 0

### Slide 06. 매칭 알고리즘: AI는 읽고, 규칙이 판정한다

핵심 메시지:
- LLM은 비정형 공고를 구조화하는 데 쓰입니다.
- 적격/조건부/부적격 판정은 결정론 규칙엔진이 수행합니다.

판정 로직:

```text
required/exclusion fail -> 부적격
fail 없음 + unknown 존재 -> 조건부
모두 pass -> 적격
```

차별점:
- 블랙박스 추천이 아님
- 원문 근거와 회사값이 함께 남음
- 모르면 탈락시키지 않고 조건부로 남김
- 판정 로직과 스코어 버전을 기록해 회귀 추적 가능

### Slide 07. 실데이터 매칭 데모: 작동하는 엔진

핵심 메시지:
- 설계가 아니라 실데이터에서 엔드투엔드 매칭이 동작했습니다.

넣을 숫자:
- 가상 사업자: 경기, 업력 2년 2개월, 대표 35세
- open 215건 매칭
- 적격 63건
- 조건부 88건
- 부적격 64건

예시:
- 경기 대상, 7년 이내, 대표 연령 충족 -> 적격
- 서울 대상 -> 부적격
- 수도권 제외 -> 경기 기업 부적격
- 업종/규모 상세 불명 -> 조건부

결론:
**조건부 88건은 약점이 아니라 신뢰 설계입니다. 모르는 것을 부적격으로 단정하지 않습니다.**

### Slide 08. 매칭 이후: 다음 질문과 액션 큐

핵심 메시지:
- 창업노트는 공고를 나열하지 않고, 사용자가 지금 무엇을 하면 되는지 정렬합니다.

구현 내용:
- unknown 필드 기반 next_question
- 지원금·마감·노력도를 반영한 action queue
- 신청 준비 시트: 필요한 서류, 원문 첨부, 프로필 복사 필드, 사업계획서 초안 프롬프트

표현:
**추천 목록에서 끝나지 않고, 조건부를 줄이고 신청 가능성을 높이는 다음 행동으로 이어집니다.**

### Slide 09. 운영 신뢰성: 버전·검증·권한까지 포함한 기술 스택

핵심 메시지:
- 창업노트는 단순 PoC가 아니라 운영 가능한 서비스 구조를 갖추고 있습니다.

넣을 근거:
- `match_state`, `match_events`, `feedback`, `golden_set`, `eval_runs`, `versions`
- verify 스크립트 28개
- API route 50개
- NextAuth 이메일/PW·Google·Kakao
- RLS, company role, consent scope

결론:
**모든 추출·판정·피드백은 버전과 로그로 남고, 개선은 골든셋 평가를 통과해야 활성화됩니다.**

### Slide 10. 기술 비전: 기업 성장 데이터 플라이휠

핵심 메시지:
- 창업노트의 장기 해자는 공고 데이터 자체가 아니라, 기업 성장 과정에서 쌓이는 정답 데이터입니다.

플라이휠:
1. 공고 수집·정규화
2. 회사별 매칭·설명
3. 사용자 입력·피드백·신청 행동
4. outcome 라벨 축적
5. golden set 평가
6. 추출·매핑·랭킹 개선

비전 문장:
**우리는 정부지원사업 검색 서비스를 넘어, 스타트업의 성장 조건·기회·행동을 구조화하는 AI 데이터 인프라가 됩니다.**

### Slide 11. IR용 한 문장 요약

**창업노트의 기술력은 "공고를 잘 보여주는 것"이 아니라, 흩어진 공고를 표준화된 계약으로 바꾸고, 설명 가능한 규칙엔진으로 우리 회사의 적격 여부와 다음 행동을 계산하는 데 있습니다.**

보조 문장:
- 현재 공고 유니버스 분석으로 표준화 체계를 만들었습니다.
- K-Startup과 기업마당의 서로 다른 추출 난도를 소스별로 분리해 해결했습니다.
- 매칭 판정은 블랙박스가 아니라 근거·버전·신뢰도를 가진 결정론 엔진입니다.
- 앞으로의 해자는 사용자 피드백과 선정 outcome으로 강화되는 성장 데이터 플라이휠입니다.
