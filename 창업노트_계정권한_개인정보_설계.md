# 창업노트 — 계정·회사·권한·개인정보 설계

작성일: 2026-06-19 · 사유: 독립 검증에서 "데이터 모델의 사용자/회사/권한 절반과 신용정보법 현실"이 누락으로 지적됨. 그 앞단을 메운다.

## 1. 엔티티·관계 모델 (빠져 있던 절반)
```
users (NextAuth 계정: 이메일/Google/Kakao)
  ├─ app_refresh_tokens (Flutter 앱 refresh 회전·폐기)
  └─< user_company >─┐   (소유·권한 N:M)
                     companies (사업자 엔티티)
                        ├── company_profiles (속성: 필드별 source/conf/as_of)
                        ├── consents
                        └── match_state / match_events
external cache: company_enrichment_cache(provider+biz_no+scope)
```
- 한 사용자가 **복수 사업자** 등록 가능, 한 사업자를 **복수 사용자**가(법인 공동) → N:M = `user_company`.
- **예비창업**(번호 없음): `companies.kind='preliminary'`, 자연키 대신 surrogate uuid. 번호 미보유.

### `companies`
`id(uuid) · kind(active|preliminary) · biz_no(nullable, unique) · legal_type(개인|법인) · name · verified(bool) · verified_at · verify_method · created_by · created_at`

### `user_company`
`user_id · company_id · role(owner|member|viewer) · invited_by · created_at` (PK: user_id+company_id)

### `company_profiles` (그동안 미정의 — 매칭 입력의 절반)
프로필은 **필드 단위 레코드**로(출처·신뢰도·시점 추적). grant_criteria와 대칭.
```
company_profiles  (필드 단위 레코드)
  company_id · dimension(region|biz_age|industry|size|revenue|employees
                         |founder_trait|certification|ip|prior_award)
  value(jsonb, 표준화 프로토콜의 canonical 형식)
  source(팝빌|국세청|codef|self_declared|ocr)
  confidence(0~1) · as_of · updated_at
```
- 매칭 comparator는 이 표준값을 grant_criteria와 대조. `source` 우선순위로 충돌 해소(아래).

> 명칭 통일: 전 문서에서 **`company_profiles`(복수)** 로 통일(매칭/플로우 문서의 `company_profile` 단수 표기 정정).

## 2. 소유권 검증 (재설계 — 자기참조 모순 해소)
**문제(검증 지적)**: "대표자명+개업일 일치확인"을 팝빌 보강 앞에 뒀는데, 그 값 출처가 팝빌이라 자기참조. 개업일이 동의 없이 안 나오면 검증 불가.

**재설계**: 검증은 *팝빌이 아니라 국세청과 사용자 입력*으로.
```
① 사용자가 [사업자번호 + 대표자명 + 개업일] 직접 입력
② 국세청 사업자등록 진위확인 API(validate, 3요소 일치) 로 검증   ← 출처=국세청, 자기참조 없음
③ 통과 → companies.verified=true → 이후 팝빌 보강(공개정보)·매칭
   (민감 작업/매출연동 단계는 휴대폰·공동인증 본인인증으로 격상)
```
- 등록증 업로드(OCR)는 보조 수단(입력 자동채움), 단독 검증 근거로는 약함.
- 검증 전에는 비저장 1차 티저만, 저장·정밀조회는 검증 후.

## 2.5 모바일 인증 (NextAuth + 토큰 레이어)
웹은 NextAuth 세션(쿠키). **앱(Flutter)은 NextAuth 세션을 그대로 못 쓰므로 토큰 레이어를 추가**:
- `/api/app/v1/auth/login`(이메일/PW)·`/auth/{provider}`(Google/Kakao OAuth code 교환)·`/auth/refresh` → **access+refresh JWT** 발급, Flutter는 secure storage.
- 웹·앱이 **동일 user/account(Postgres, NextAuth 어댑터)** 공유. 외부 의존 없이 우리 API가 앱 인증 주체.
- access TTL 짧게 + refresh 회전. refresh token은 원문 저장 금지, `app_refresh_tokens(token_hash, device_id, expires_at, revoked_at, rotated_from)`에 해시만 저장해 기기별 로그아웃·탈취 대응·회전 재사용 탐지를 가능하게 한다. (상세: `창업노트_레포전략_배포토폴로지.md` §4)

## 3. 인가·RLS 모델 (전무했음)
Supabase Auth 미사용 → **인가는 BFF(Next.js)가 책임**.
- **접근 규칙**: 모든 company 스코프 요청에 `user_company` 소유 확인 미들웨어. 사용자는 자신이 속한 company 행만.
- **권한 분리**: 
  - 사용자 경로(BFF) = scoped, user→company 검증된 범위만.
  - 워커(GCP, service role) = grants/추출 등 비개인 데이터 + 보강 쓰기. **사용자 PII 읽기 최소화.**
- **이중 방어(권장)**: BFF 인가 + Postgres RLS(앱이 set하는 `app.current_user`/JWT claim 기반)로 DB단에서도 차단. 신용정보 유입(Phase 2) 전 필수.

### 접근 매트릭스(요지)
| 데이터 | 사용자(BFF) | 워커 | 비고 |
|---|---|---|---|
| grants/criteria | 읽기 | 읽기·쓰기 | 비개인 |
| company_profiles | 본인 company만 | 쓰기(보강) | PII |
| match_state | 본인 company만 | 쓰기 | |
| app_refresh_tokens | 본인 토큰만 | 없음 | 원문 토큰 저장 금지 |
| 매출·4대보험(v1+) | 본인+동의 | 위탁조회만 | 신용/과세정보 |

## 4. 개인정보·신용정보법 설계 (누락 — 독립 영역 신설)
데이터를 등급으로 나눠 등급별 처리:
| 등급 | 예 | 취득 | 규율 |
|---|---|---|---|
| 공개정보 | 사업자 상태·상호·업종(공개분) | 국세청/공개 | 경량 |
| 개인정보 | 대표자명·연락처·계정 | 가입·입력 | 개인정보보호법 |
| **과세·신용정보** | 매출·4대보험·신용 | **CODEF 위탁+명시동의** | **신용정보법** |

- **동의 원장(consent ledger)**: 무엇을·언제·어떤 목적으로 동의했는지 레코드(`consents` 테이블), 철회 가능.
- **분리보관·암호화**: 과세·신용정보는 분리 저장·컬럼 암호화·접근로그.
- **위탁 고지**: 팝빌·CODEF는 수탁자 → 처리방침에 위탁사실·항목 고지, 위탁계약(보안).
- **최소수집·보관기간·파기**: 매칭 목적에 필요한 필드만, 기간 만료·탈퇴 시 파기.
- **단계 분리**: 기본정보(경량 동의)와 매출·4대보험(명시 동의·CODEF)을 UX·저장에서 분리.

## 5. 보강 fallback (검증 지적 — "번호만으로 자동보강" 미성립 대비)
S0-D에서 **팝빌 단건 조회로 동의 없이 실제 채워지는 필드**가 적으면:
- 자동보강은 가능한 필드만(상태·공개 업종 등), **나머지는 기창업도 자가신고 경로**로 폴백(예비창업과 동일 입력 UI 재사용).
- 즉 "번호 하나로"는 *가능한 만큼 자동 + 부족분 progressive*로 정직하게.

## 6. 결정
- **본인인증 = PASS(휴대폰)** — 소유권검증 후 민감단계 격상에 사용.
- **RLS** = PII·신용정보 전면, 비PII는 BFF 인가(이중방어).
- **동의 철회** = 해당 동의기반 필드 즉시 파기 + 그 필드로 산출된 매칭은 삭제가 아니라 **unknown으로 재평가**(신뢰도 하향).
- **암호화 키** = GCP KMS, 신용정보 컬럼 암호화·접근로그.
