# 창업노트 — IR 기술 파트 (마스터 문서)

> (주)노튼 · 창업노트(changupnote.com) · 작성 기준일 2026-06-27
> 용도: 기존 IR 자료의 **기술 섹션** 원천 문서. 이 문서에서 슬라이드 장표를 파생한다.
> 원칙: **검증된 사실(실측·구현)과 향후 비전(설계·로드맵)을 명확히 구분**한다. IR 신뢰를 위해 "정확도 검증 완료" 같은 미확정 주장은 쓰지 않는다.

## 0. 한 줄 정의

창업노트는 **흩어진 정부지원사업 공고를 14개 축의 표준 계약(contract)으로 구조화**하고, **설명 가능한 결정론 규칙엔진**으로 "우리 회사 × 모든 공고"의 적격 여부·근거·다음 행동을 자동 산출하는 매칭 엔진이다. AI는 *추출(읽기)* 만, *판정(당락)* 은 규칙이 한다.

---

## 1. 문제 — 왜 기술이 필요한가

| 현실 | 결과 |
|---|---|
| 공고가 K-Startup·기업마당 등 **수십 개 채널에 파편화** | 어디에 뭐가 있는지 모름 |
| 자격요건이 **본문·HWP 표·첨부에 비정형 산문**으로 기재 | 사람이 한 건씩 읽어야 판정 가능 |
| "업력 7년 이내", "수도권 제외", "신산업은 10년 완화" 같은 **조건의 미묘함** | 자동화가 어렵고, 잘못하면 거짓 탈락 |
| 접수처가 **forms.gle 138종 등 225종으로 파편화**(K-Startup n=1,000 실측) | 통합 자동제출 자체가 불가능 |

→ 핵심 난제는 **"비정형 공고 → 신뢰할 수 있는 구조화 + 설명 가능한 판정"**. 이것을 규모(현재 K-Startup 단일 소스만 29,187건)로, 거짓 확신 없이, 데이터가 쌓일수록 정교해지게 푸는 것이 우리의 기술이다.

## 2. 아키텍처 한눈에

```
사용자 ──▶ Next.js (Vercel)  ──매칭조회(빠른 읽기)──▶  Supabase Postgres
            · BFF API / Server Actions                  ▲  정규화 grant 스키마 = 두 레이어의 "계약"
            · 결정론 매칭 엔진(@cunote/core)             │  grants · grant_criteria · match_state · *_log · golden_set
            · Upstash Redis (캐시·레이트리밋·큐)         │
                                                         │ UPSERT(발행)
   ┌──────── GCP / Python 워커 (배치·비동기) ────────┐  │
   │ 수집 → 변환(HWP→MD) → LLM 추출 → 정규화 → 스코어 ─┼──┘
   │ 상태머신·증분·dedup·backfill · R2(첨부 원본)      │
   └──────────────────────────────────────────────────┘
            └ Claude (구조화 추출) · 팝빌(기업 기본정보)
```

**설계의 뼈대 = 2-레이어 분리.**
- **수집(Ingestion) 레이어** — 무겁고 느린 비동기 작업(크롤·변환·LLM 추출). Python/GCP.
- **서빙(Serving) 레이어** — 사용자 경로는 Postgres 정규화 데이터를 **읽기만** 하므로 빠르고 안정적. Next.js/Vercel.
- 둘을 잇는 **단일 계약 = 정규화 `grant` 스키마**. 추출이 아무리 바뀌어도 서빙은 같은 계약만 본다.

> 멀티클라우드(Vercel·Supabase·GCP·Cloudflare) 토폴로지지만, 무거운 데이터 평면은 GCP↔Postgres↔R2로 정렬하고 사용자 경로는 Postgres 읽기로 격리해 비용·지연을 통제한다.

## 3. 핵심 기술 차별점 (Defensible Core)

### 3.1 표준화 계약 — 14개 축으로 환원한 "공고의 공통어"

모든 공고를, 회사와 같은 좌표계에서 비교할 수 있도록 **14개 차원(dimension)**으로 표준화한다.

> 지역 · 업력 · 업종/분야 · 기업규모 · 매출 · 고용 · 대표자 연령 · 대표자 속성 · 인증 · 기수혜이력 · 지식재산 · 신청대상 · 영업상태 · 기타

각 차원은 **5요소 프로토콜**로 정의된다 — ① Canonical(표준 저장형) ② Company resolver(기업데이터→표준) ③ Grant extractor(공고→표준, controlled list만) ④ Comparator(비교규칙) ⑤ Mapping asset(버전 관리되는 매핑 자산).

- 매핑 자산(`industry_taxonomy` KSIC↔정책태그, `region_hierarchy` 권역전개, `size_thresholds` 업종별 임계)은 **코드가 아니라 버전 달린 데이터 자산**이다. 골든셋으로 계속 개선된다.
- 공고가 *주는 것(혜택)* 도 대칭으로 표준화 — **6패밀리 혜택 taxonomy**(자금·융자·역량·공간·시장·인증·연결)로 "받을 수 있는 것"과 부담(obligations)을 함께 구조화한다.

### 3.2 추출(AI) ↔ 판정(규칙) 분리 — "AI는 읽고, 규칙이 판정한다"

이것이 신뢰의 핵심 설계다.

- **추출 = AI(Claude).** 비정형 공고를 `grant_criteria[]`로 구조화. **tool-use(구조화 출력) 강제**, 필드별 `confidence` + **`source_span`(원문 근거)** 동봉, controlled vocabulary만 허용, **근거 없으면 날조 금지(null)**.
- **판정 = 결정론 규칙엔진.** 추출된 조건을 회사 값과 한 줄씩 비교해 `pass / fail / unknown`을 매기고, 단 3줄 규칙으로 **적격 / 조건부 / 부적격**을 확정한다.

```
1. 필수·배제 중 하나라도 fail   → 부적격 (즉시 탈락)
2. fail은 없는데 필수·배제에 unknown → 조건부
3. 전부 pass                    → 적격
```

→ **블랙박스가 당락을 정하지 않는다.** LLM이 틀려도 그 영향은 "추출 입력 품질"에 갇히고, 판정 자체는 늘 결정론적·재현 가능·감사 가능하다. (실제 구현: `@cunote/core`의 `matchGrantCriteria`, `RULESET_VERSION` 버전드)

### 3.3 거짓 탈락 금지 — 결측은 'unknown'

모르는 것을 함부로 '안 됨'으로 단정하지 않는다. 정보가 없으면 **`unknown`(조건부)** 으로 두고 사용자에게 물어본다. 자가신고·예비창업 등 출처 신뢰도가 낮으면 `confidence`가 자연히 점수를 하향한다 — **거짓 확신을 구조적으로 차단**한다.

### 3.4 점수 3분해 — "87점" 같은 거짓 단일점수 금지

신뢰 가능한 적합도와 추측성 선정가능성을 섞지 않는다.

| 점수 | 성격 | 표시 |
|---|---|---|
| **적합도(Fit)** | 지금 신뢰 가능 (규칙엔진 산출) | 헤드라인 |
| **선정가능성(Competitiveness)** | 데이터 누적 후 학습 (outcome 게이트) | "추정" 라벨, 보조 배지 |
| **가치(Value)** | 혜택−부담 합산 | 보조 배지 |

각 점수에 신뢰도(confidence)가 붙고, 조건부는 "지금 78점 → 매출 입력 시 최대 92점"처럼 **확정 하한·잠재 상한을 함께** 보여준다.

### 3.5 정보이득 → 다음 질문 / NPC 액션 큐

- **정보이득(information gain):** 비어 있는 필드 각각이 "몇 건의 조건부 공고를 확정/제외시킬 수 있는가"를 추정해, **가장 가치 있는 질문 하나**만 노출한다(토스식 1화면 1질문). 상승 일변도가 아니라 **"확정 또는 제외에 도움"** 양방향으로 정직하게 프레이밍한다.
- **NPC 액션 큐:** 정보이득을 *모든 행동 유형*(입력·자격취득·신청·보강)으로 일반화해, `(레버리지 × 긴급) ÷ 노력`으로 **"지금 당장 할 일"을 전역 정렬**한다. 무엇을·어디서(발급처)·왜(무엇이 해금되는지)까지 결정론적으로 산출하고, LLM은 표현만 담당한다.

### 3.6 살아있는 월드 — 시간 기반 재매칭(전이 날짜 사전계산)

업력 같은 시간 의존 조건 때문에 매일 "전체 회사 × 전체 공고"를 재평가하는 것은 비효율이다.

- 매칭 시 `eligible_from / eligible_until`(언제 열리고 닫히는지)을 **사전계산**해 `match_state`에 저장.
- 가벼운 일일 잡이 **그날 전이되는 건만** flip + 알림(곧 닫힘 / 새로 해금).
- 프로필·공고 변경은 **영향받는 (회사×공고)만** 이벤트 재매칭(전건 N×M 금지 = 비용 가드).

→ "매일 전수"(무식·비쌈)와 "조회 시 매번 계산"(사전 알림 불가)의 단점을 모두 회피. 게임의 "곧 사라지는 퀘스트" 긴급감을 데이터로 정직하게 구현한다.

### 3.7 지속개선 플라이휠 — 1일차부터 점수·버전·라벨링

모든 추출·매칭 산출물에 **버전 태그**(`model_ver` / `prompt_ver` / `ruleset_ver` / `scoring_ver`)를 박는다 → "무엇이 품질을 바꿨나"를 귀속(attribution)할 수 있다.

```
운영(점수+버전 로깅) → 저신뢰·피드백·outcome → 라벨링 큐 → 사람 정답
  → golden_set 갱신 → CI 평가(P/R·p@k) → 회귀 없으면만 새 버전 활성화
  → backfill(원본 R2로 재추출, 재크롤 0) → 다시 운영…
```

- **CI 평가 게이트:** 골든셋 회귀가 발견되면 **활성화 차단**(조용한 품질 저하 방지).
- **데이터가 쌓일수록 정교해지는 4경로:** ① 추출 정확도 ② 매핑 자산 ③ 스코어 가중치(learn-to-rank) ④ 선정가능성 모델.
- **단, 적격 판정은 늘 결정론·설명 가능하게 둔다** — "더 똑똑해지되, 당락은 언제나 설명 가능."

## 4. 현시점 가장 큰 기술적 특징 (Now)

> **한 문장:** *비정형 공고를 표준 계약으로 환원하고, 설명 가능한 결정론 매칭 엔진을 실데이터에서 엔드투엔드로 실증했다.* 이것이 지금의 해자다.

**왜 이것이 큰가** — 정부지원 매칭은 보통 "LLM에 다 떠넘기는 블랙박스" 아니면 "사람이 수기로 채우는 디렉터리" 둘 중 하나다. 우리는 **둘 다 거부**했다.

실측으로 검증된 사실(설계가 아니라 데이터):

| 항목 | 실측 결과 | 의미 |
|---|---|---|
| **K-Startup 척추 6차원** | 지역·업력·연령·분야·대상·일정 **100% 구조화 필드로 자동 산출** (표본 500 / 전체 29,187건) | **LLM 없이 파서만으로** 견고한 1차 매칭 — 오류 위험·비용 0 |
| **엔드투엔드 매칭 데모** | 가상 사업자 × open 215건 → 적격 63·조건부 88·부적격 64, rule_trace 자연어 설명·수도권 제외 배제까지 동작 | **매칭 엔진 전체가 실데이터에서 작동** |
| **HWP 표 보존** | pyhwp `hwp5html`로 자격요건 표 11개를 셀 단위 보존 → LLM-ready 마크다운 | 비정형 첨부의 **표 유실 리스크 해결**(상용툴 불필요) |
| **LLM 추출 안정성** | tool-use 전환으로 JSON 파싱 실패 **1/3 → 0**, 업력 완화·배제업종·OR 트랙 nuance 포착, 자동판정 불가는 정직하게 보류 | 추출이 **실무 수준으로 작동** |

**그리고 이것은 문서가 아니라 동작하는 코드다.**
- `@cunote/core` — 결정론 매칭 엔진(`matchGrantCriteria`), K-Startup·기업마당 정규화, 팝빌 보강, dedup.
- **use-case 12종 구현** — 매칭 카드 선별, 액션 큐, 로드맵, 티저, 대시보드, 신청 준비 시트, 통계, 상태 갱신/전이 계획 등.
- `@cunote/contracts` — DTO + OpenAPI + `grant_criteria` JSON Schema(웹·앱·워커 공유 계약 seam).
- Next.js 앱(웹+어드민+앱API), Drizzle 마이그레이션·RLS, NextAuth(이메일/PW·Google·Kakao), 골든셋·평가 하니스.

> **정직성 게이트(IR 신뢰):** 도메인 전문가 골든셋 50~100건 기반의 **엄밀 정밀도/재현율(P/R)은 팀 작업으로 진행 중**이며, 그 전에는 "정확도 검증 완료"로 표현하지 않는다. 현재 실증된 것은 *작동·실패모드·구조적 커버리지*다. — 이 절제가 곧 제품 철학(거짓확신 금지)의 일관성이다.

## 5. 향후 비전 (Next)

### 5.1 단기 — 정확도 플라이휠 가동
- 도메인 골든셋 P/R 측정 → 추출 프롬프트·few-shot·매핑 자산 튜닝 → **HITL 자동화 비율 상승**(시간이 갈수록 사람 개입↓).
- 기업마당(HWP·본문 비정형, 高난도) 추출 본격화, 소스 채널 확장(현재 K-Startup 단일 → 다채널 → 1,500+ 공고 규모).

### 5.2 중기 — 학습으로 정교해지는 랭킹
- **learn-to-rank:** 사용자 피드백·암묵신호·outcome으로 적합도 보너스·기업적합·의미유사 **가중치를 데이터로 튜닝**.
- **선정가능성 모델:** 신청→선정 outcome이 쌓이면 경쟁률·적합강도 기반 추정 → 학습모델로 승급(가장 늦게 도착하는, 가장 희소한 데이터).
- 매출·고용 **CODEF 동의연동**으로 규모·정책자금 판정 정밀화, 정책자금(소진공·기보·신보)·민간투자 트랙 확장.

### 5.3 장기 — "기업 성장 데이터 = AI 자산"
- **성장 RPG 메타포:** 지원사업=퀘스트(조건 충족 시 해금), 기업=레벨·업적·투자단계(예비→Seed→A→B→C). 메타포가 실데이터(`rule_trace`·`match_state`)에 1:1로 묶여 **정직**하다.
- **수혜이력 타임라인 + 사업계획서 초안 어시스턴트**(PSST). *단, 대리 제출은 하지 않는다* — 접수처 225종 파편화 실측으로 자동제출이 불가함을 확인했고, 이는 기술적으로도 "프로필 자동완성 + 초안 작성 도움"으로 범위를 정직하게 한정한 결과다.
- 축적되는 **기업 성장 데이터가 (주)노튼의 장기 AI 자산**이 된다 — 플라이휠의 outcome이 곧 데이터 해자.

## 6. 기술 스택 요약

| 영역 | 선택 |

| 프론트/앱 | Next.js(TypeScript)/Vercel · Flutter(iOS/Android, 앱 전용서버 없이 동일 API) |
| DB | Supabase = PostgreSQL 전용 (+ pgvector 후보 축소) |
| 캐시·큐 | Upstash Redis 단일 |
| 워커 | Python / GCP (Cloud Run + Scheduler + Jobs) |
| 엣지·저장 | Cloudflare(DNS·WAF·CDN) + R2(첨부 원본) |
| LLM | Anthropic Claude (구조화 추출, 버전 관리) |
| 기업 기본정보 | 팝빌(기업정보조회) — 제3자 크롤링 배제(ToS·신용정보법 리스크) |

| 마이그레이션 | Drizzle (web 단일 소유) + RLS(PII 접근통제) |

**모노레포 계약 seam:** `@cunote/core`(Next 비의존 도메인 로직) · `@cunote/contracts`(DTO/OpenAPI/JSON Schema) · `apps/web`. 웹·앱·워커가 같은 계약을 공유한다.
