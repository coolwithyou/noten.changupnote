## 결론 요약

**"사업자번호 → 매칭 티저"까지는 30초 안에 매끄럽게 작동한다. 그러나 "매칭 → 내 회사 정보가 반영된 지원서 작성"으로 넘어가는 다리 3개가 전부 끊겨 있다.**

1. **프로필 파이프라인 단절** — 사용자가 입력한 회사 정보가 어느 경로로도 저장·전파되지 않는다 (매칭 화면은 입력 즉시 크래시, 온보딩 폼은 입력을 조용히 버림). 그 결과 "사업자번호 하나로"라는 약속이 이후 모든 화면에서 같은 정보의 반복 입력 요구(최대 4회)로 뒤집힌다.
2. **공고 선택 맥락 유실** — 비로그인 매칭에서 "이 사업 신청 준비하기"를 눌러도 선택한 공고가 어디에도 저장되지 않는다. 가입을 마치면 빈 온보딩 폼에 떨어지고, 그 공고는 사용자가 다시 찾아야 한다.
3. **실제 양식과의 연결 부재** — "AI 초안"은 LLM이 아닌 고정 템플릿이고, 원문 HWP 양식의 실제 문항과 무관하다. 문서 뷰어·필드 매핑·필드 팁은 코드가 존재하지만 사용자 데이터가 0건이라 어떤 공고에서도 나타나지 않는다.

강점도 분명하다: 서류 자동 분류(작성/발급/첨부)가 되는 공고의 준비 시트 구조, 그리고 **"작성 유의사항"(운영 검증 지식) 패널**은 "뭘 조심해야 하는지"를 알려주는 유일하게 실질적인 가이드다. 이 강점을 초안 워크스페이스와 결합하는 것이 다음 개선의 축이 되어야 한다.

## 발견 사항 상세 (심각도순)

### P0 — 여정 차단

**A-2. 매칭 프로필 보강: 타이핑 즉시 전체 페이지 크래시**
- 재현: `/matches?biz=2378603641` → 아무 미입력 카드 "입력하기" → 두 글자 이상 타이핑 → `TypeError: Cannot read properties of null (reading 'value')`, 페이지 전체가 에러 화면으로 교체
- 원인: `apps/web/src/features/matches/MatchesExperience.tsx` — onChange가 **setState 업데이터 함수 안에서 `event.currentTarget.value`를 읽음**. React가 업데이터를 리렌더 시점에 지연 재생(replay)할 때 `currentTarget`은 이미 null. 콤보박스(452행 부근)·숫자 입력(507, 552, 568, 592행 부근) 전 필드 공통 패턴
- 수정 방향: 업데이터 밖에서 `const next = event.currentTarget.value`로 값을 먼저 추출
- 영향: 매칭 정확도를 올리는 핵심 동선("정보를 더 입력하면 매칭 범위가 넓어져요")이 100% 차단. 최근 커밋 "매칭 수동 프로필 보강 흐름 추가"의 기능이 사실상 미출시 상태

**B-1. 온보딩 회사 생성: 입력값 무단 폐기 → 빈 껍데기 회사**
- 재현: 온보딩 폼에 사업자번호+회사명+지역+업종 모두 입력 → 생성 → DB의 companies 행: `name=NULL, biz_no=NULL, verified=false`, company_profiles에 business_status 1건만
- 원인 ①: `apps/web/src/features/onboarding/InitialCompanySetupPanel.tsx:187` — `if (bizNo.length === 10) return { bizNo }` — **번호가 있으면 나머지 입력을 조용히 버림**. UI에는 상호 배타라는 안내 없음
- 원인 ②: 등기 정보가 sparse한 번호면 서버가 만든 회사도 name 없이 생성 → 이후 모든 화면에서 "샘플 기업" 표기, 복붙 프로필·초안 자동채움·매칭 체크리스트 전부 빈 값
- 영향: 첫 인증 사용자 셋업 단계에서 데이터가 소실되는데 성공 메시지가 뜸. 이후 반복 입력 요구의 근원

**A-3. 매칭 → 가입 경계에서 선택 공고 유실**
- 현상: 티저 카드의 "이 사업 신청 준비하기"가 `grantId`를 받지 않음 — `MatchesExperience.tsx:170-184`의 `saveAndContinue`는 bizNo+수동 프로필만 sessionStorage에 저장. 로그인 복귀 후 `createCompanyAndOpenDashboard`는 무조건 `/dashboard`로 이동 (`HomeExperience.tsx`, `LandingExperience.tsx`)
- 영향: "이 공고에 지원하겠다"는 가장 강한 인텐트가 가입 관문에서 증발. 사용자는 기회 맵에서 그 공고를 다시 찾아야 함 (찾을 수 있다는 보장도 없음 — C-3)
- 부수(로컬/프리뷰 한정): NextAuth 리다이렉트가 `NEXTAUTH_URL`(dev.changupnote.com) 기준 절대 URL이라 localhost 세션에서 origin이 바뀌며 sessionStorage 복원 자체가 실행되지 않음

### P1 — 핵심 가치 미완/단절

**C-1. "AI 초안"이 결정적 템플릿 (LLM 미연결)**
- `packages/core/src/documents/draft-generation.ts` — `deterministic-document-draft-v1`. 신청서 "지원 적합성"은 공고명+회사명 치환 1문장, 사업계획서는 입력이 없으면 "~해야 합니다" 지시문이 본문
- "뭘 써야 할지 모르는" 페르소나에게 이 초안은 골격일 뿐 해설·예시가 아님. UI 전반의 "AI 초안" 라벨과 실물 간 기대 격차가 큼
- 참고: 마스터 설계상 LLM 초안은 후속 Phase — 그렇다면 그 전까지 라벨·카피 조정(예: "기본 골격")과 초안 프롬프트/작성 유의사항의 전진 배치가 필요

**D-1. 뷰어·필드 매핑·필드 팁의 사용자 데이터가 전무**
- `grant_application_surfaces` 0 · `document_artifacts` 0 · `grant_document_fields` 0 (dev DB 실측)
- "지원서를 화면에 띄워두고 해설 + 실제 정보가 채워진 제안"이라는 목표 경험의 구현체(Phase 3 뷰어, §8.6 reconciliation)는 존재하나 변환 파이프라인 프로덕션 배선([F2], A7 Cloud Run env)이 안 되어 유저 경로에서 도달 불가
- 뷰어 진입 링크 부재(ApplySheetView)·뷰어 내 내비 부재(D-2)까지 겹쳐 3중 고립

**C-3. 신청 관리가 "현재 매칭 목록"의 부분집합 — 낮은 적합도 지원 시나리오와 정면 충돌**
- `apps/web/src/lib/server/applications/pipeline.ts:69-71` — `matches.filter(eligibility !== "ineligible").slice(0, 80)`에 초안/피드백을 조인. **초안이 있어도 매칭 목록에 없으면 파이프라인에서 소멸**
- 실측: 초안 2건을 만든 [인천] 우수디자인(D-1 마감)이 신청 관리 어디에도 없음, "서류 준비 0건"
- 수정 방향: 파이프라인 모집합 = 매칭 상위 ∪ (초안·피드백·저장 기록이 있는 공고). 부적격이어도 사용자 행동이 있으면 포함

**C-4. 같은 정보 반복 입력 (기업명·소재지·업종 기준 최대 4회)**
- ① 매칭 보강(크래시로 불가) → ② 온보딩(폐기됨) → ③ 신청서 추가 입력 → ④ 사업계획서 추가 입력(문서별 `answerText` 상태 분리, 프로필 저장 안 됨 — `DocumentDraftWorkspace.tsx`)
- 초안용 입력을 company_profiles로 승격(또는 최소한 문서 간 공유)하면 "한 번 입력하면 어디서나"가 됨

**C-6. HWP 간극**
