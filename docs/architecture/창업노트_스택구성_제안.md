## 1. 아키텍처 한눈에

```
                          ┌──────────── Cloudflare (DNS·CDN·WAF) ────────────┐
                          │                                                  │
        사용자 ──▶ [ Vercel: Next.js (TS) ]  ──읽기/매칭조회──▶ [ Supabase Postgres ]
                          │  · BFF API routes                        ▲  (정규화 grant = 계약)
                          │  · Upstash Redis (캐시·레이트리밋·경량큐)  │  grants·company·*_log·golden
                          │                                          │
                          └── R2 (첨부 원본·정적 에셋) ◀── 다운로드 ──┘
                                                                     │
   ┌──────────────── GCP (Python, 배치/독립) ─────────────────┐      │
   │ Cloud Scheduler(cron) → Cloud Run Jobs / Service          │      │
   │  크롤(K-Startup·기업마당) → R2에 첨부 저장                  │ UPSERT
   │  → 변환(HWP/PDF→MD) → LLM 추출(JSON) → 정규화 + 스코어 ────┼──────┘
   │  큐: Upstash Redis 단일 · 재시도·dedup·backfill             │
   └───────────────────────────────────────────────────────────┘
                                                                     │
        [ 어드민(Next.js) ] 라벨링 큐 · 골든셋 · 평가 리포트 ◀──────────┘
```

핵심: 두 실행 환경(Vercel-앱 / GCP-워커)이 **Supabase Postgres = 정규화 grant 스키마**를 공유 계약으로 연결. 무거운 작업은 전부 GCP, 사용자 경로는 Postgres 읽기라 빠름.

## 2. 레이어별 스택

### A. 수집·정제 (Ingestion) — GCP / Python *[최우선 ①]*
| 영역 | 선택 | 비고 |
|---|---|---|
| 실행 | **Cloud Run Service**(상시 API) + **Cloud Run Jobs**(배치 backfill) | scale-to-zero, 컨테이너 |
| 스케줄 | **Cloud Scheduler** → Jobs 트리거 | cron 대체 |

## 3. GCP 워커 구성 (질문 답)

Python on GCP — 적합해. 구체적으론:

- **Cloud Run** (서버리스 컨테이너): 추출/매칭 워커. HWP 파서·LLM 호출 등 장시간 작업을 타임아웃 여유 있게(최대 60분 Jobs). scale-to-zero라 유휴 비용 적음.
- **Cloud Run Jobs**: backfill(전체 재추출) 같은 일괄 작업.
- **Cloud Scheduler**: 매일/매시간 수집 트리거(cron 대체).
- **큐**: Upstash Redis로 통일(네 Redis 결정 활용, Vercel·GCP 공용). Pub/Sub는 운영 복잡도 증가 때문에 MVP에서는 배제.
- **시크릿**: Secret Manager.

### ⚠ 멀티클라우드 주의 (설계에 반영)
Vercel + Supabase + GCP + Cloudflare = 4-벤더 토폴로지. 강력하지만:
- **리전 정렬**: GCP 워커 ↔ Supabase Postgres ↔ R2를 가까운 리전(가능하면 동일 권역)으로. 데이터 평면 왕복이 잦은 곳일수록 중요.
- **이그레스 비용**: 크로스클라우드 전송 과금. 다행히 수집은 비동기 배치라 지연 내성 큼 — 데이터 평면(워커↔Postgres, 첨부↔R2)만 정렬하면 사용자 경로는 영향 적음.
- 경계만 깔끔하면 감당 가능. 초기엔 단순 유지, 과분리 금지.

## 4. 지속개선 플라이휠 (스코어링·라벨링·평가) *[최우선 ③ — 모든 것을 관통]*

핵심 요구: **모든 수집(추출)·매칭이 점수와 버전을 달고 기록되고, 라벨링되어 다시 품질로 환류.** 이걸 1급 계층으로 설계.

### 4.1 모든 산출물에 점수와 버전을 박는다
- **추출 스코어**: 공고→JSON 추출 시 필드별 + 전체 신뢰도(LLM self-eval·휴리스틱). 낮으면 라벨링 큐로.
- **매칭 스코어**: 필수요건 게이트(pass/fail/unknown) + 적합 점수(우대 가점·기업적합·의미유사). 랭킹 산출.
- **버전 태그**: 모든 추출·매칭에 `model_ver / prompt_ver / ruleset_ver` 부착 → "무엇이 품질을 바꿨나" 귀속(attribution) 가능.

### 4.2 신호(피드백) 수집 — 3종
- **암묵**: 노출·클릭·상세조회·저장·신청버튼(딥링크) 클릭.
- **명시**: 추천 카드의 "적합 / 부적합" 버튼.
- **결과(outcome)**: 신청→선정/탈락(사용자 보고·후속). 희소하지만 매칭 품질의 최고 정답.
- **리뷰어 라벨**: 신뢰도 낮은 추출·매칭을 어드민 큐에서 사람이 정정.

### 4.3 환류 루프
```
운영(추출·매칭, 점수+버전 로깅)
   → 신뢰도 낮음/샘플링/사용자피드백 → 라벨링 큐 → 사람 정답
   → golden_set 갱신 → CI 평가(precision/recall, p@k)
   → 추출 프롬프트·few-shot / 매칭 규칙·가중치 개선 → 새 버전 활성화
   → (추출 개선 시) R2 원본으로 backfill 재추출 / (매칭 개선 시) 재랭킹
   → 다시 운영…
```
- HITL 보정 결과가 golden_set·few-shot으로 쌓여 추출이 점점 좋아지고, 사용자 피드백·outcome이 매칭 골든셋을 키움 → **리멤버식 데이터 플라이휠**.
- 버전 활성화 전 **반드시 골든셋 회귀 평가 통과**(조용한 품질 저하 차단).

## 5. 핵심 데이터 모델 (data-of-record)

| 테이블 | 역할 | 주요 컬럼(요지) |
|---|---|---|
| `grants` | 정규화 공고(계약) | 지역·업종·업력·규모·지원금·필수[]·우대[] / 필드별 confidence·source_span / model_ver·prompt_ver |
| `grant_raw` | 원본 스냅샷 | 원문·첨부 R2 ref, 수집시각, 해시(변경감지) |
| `extraction_log` | 추출 이력·스코어 | grant_id, output_json, confidence, status(auto/review/labeled), reviewer, model/prompt_ver |
| `company_profiles` | 기업 보강 데이터 | 필드별 source·confidence·as_of |
| `match_state` | 현재 매칭 상태 | company_id, grant_id, eligibility, match_score, rule_trace, ruleset_ver, scoring_ver, updated_at |
| `match_events` | 사용자/노출 이벤트 로그 | company_id, grant_id, event(surfaced/clicked/saved/apply_click), ruleset_ver, ts |
| `feedback` | 신호 | target, type(impl/expl/outcome), value, actor, ts |
| `golden_set` | 정답 기준셋 | 추출 정답 / (기업×공고) 적격 정답, curated_by, ver |
| `*_versions` | 버전 레지스트리 | model·prompt·ruleset 해시·노트·activated_at |
| `eval_runs` | 평가 결과 | target, version_refs, metrics{}, golden_ver, ts |

## 6. 다음 설계 단계 (코드 아님, 설계 심화)

1. **정규화 grant 스키마 상세** — 두 레이어의 계약. 필드 정의·자료형·필수/우대 표현·결측·신뢰도 메타.
2. **수집 파이프라인 상세** ①
3. **매칭 모델 상세** ② — 자격요건→규칙 표현, 스코어 함수, 결측·우대 처리, 탈락사유 반환.
4. **플라이휠 운영 상세** ③ — 라벨링 큐 정책, 신뢰도 임계값, 평가지표 정의, backfill 트리거.

- **큐**: Upstash Redis 단일.
- **인증**: NextAuth(Auth.js), 외부 의존 최소. 1차 이메일/PW·Google·Kakao, 세션·계정 Postgres 어댑터.
