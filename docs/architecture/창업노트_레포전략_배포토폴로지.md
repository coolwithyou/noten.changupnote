# 창업노트 — 레포 전략 & 배포 토폴로지

작성일: 2026-06-24 · 사유: 클라이언트 확장(웹+Flutter), 단일 Next.js가 앱 API·어드민 겸함, 분리 마일스톤.

## 1. 결정 입력 (확정)
- 클라이언트: **웹(Next.js) + Flutter(iOS/Android)**.
- **앱 전용 API 서버 없음** → 웹(Next.js, 단일 Vercel 프로젝트)이 **앱용 독립 API 엔드포인트**도 제공.
- **Admin도 동일 단일 Next.js**(프론트+백+API+어드민 한 덩어리).
- MVP 안정화 후 **admin·API를 별도 인스턴스로 분리**(지금은 마일스톤, 단 *설계는 분리 가능하게*).
- 인증 **NextAuth, 외부 의존 없음**. **앱 인증도 우리 API** 사용.
- **GCP Python 워커는 별도 repo**.

## 2. 레포 전략 — 3 repo + 계약 seam
| repo | 내용 | 배포 | 언어 |
|---|---|---|---|
| **cunote-web** | 웹 + 어드민 + 앱API + BFF, DB 마이그레이션, OpenAPI/계약 | Vercel(단일 프로젝트) | TS / Next.js |
| **cunote-app** | Flutter iOS/Android, 앱API 클라이언트(코드생성) | App Store / Play | Dart |
| **cunote-ingestion** | 수집·변환·추출·정규화 워커 | GCP Cloud Run | Python |

**계약 seam(원칙 #4 계약 중심)**: OpenAPI(앱 API) + `grant_criteria` JSON Schema + SQL 마이그레이션을 **cunote-web가 단일 소유**, 나머지가 소비.
- 앱: OpenAPI → **Dart 클라이언트 코드생성(CI)**.
- 인제스천: DB 스키마 + criteria JSON Schema 소비(벤더/서브모듈).

> 왜 모노레포 아님: 툴체인(JS/Dart/Python)·배포대상·릴리즈 케이던스가 달라 분리가 자연스럽고, Python 분리는 명시 결정. **드리프트 리스크 → OpenAPI codegen + 계약 버전 + CI 계약테스트로 방어.**

## 3. cunote-web 내부 구조 (분리 가능하게)
pnpm workspace:
```
cunote-web/
  apps/web/                     # Next.js (App Router)
    app/(web)/                  # 공개 웹
    app/(admin)/                # 어드민 (role gate)
    app/api/app/v1/             # 앱용 API (버전드, 토큰 인증)
    app/api/admin/              # 어드민 API
    app/api/web/  (or actions)  # 웹 BFF
  packages/core/                # 비즈니스 로직 (프레임워크 무관) ← 분리의 핵심
  packages/contracts/           # OpenAPI, 타입, criteria JSON Schema
  db/                           # 마이그레이션, RLS
```

**분리 seam (미래 admin/API 분리를 *배포 변경*으로, 재작성 아님):**
1. 비즈니스 로직 전부 `packages/core`(Next 비의존) → 라우트·서버액션은 **얇은 어댑터**.
2. 라우트 그룹 경계(web/admin/api) 명확 + **`/api/app/v1` 버전드**.
3. 인증 컨텍스트 분리(웹 세션 / 앱 토큰 / 어드민 role).
4. 데이터 접근 **repository 레이어** → 미래 독립 API가 그대로 재사용.
→ 분리 시: 어드민은 같은 core를 쓰는 **별도 Next 앱**으로, 앱 API는 core를 쓰는 **독립 서비스**로 추출.

## 4. 모바일 인증 (NextAuth + 토큰 레이어) — 주의 지점
- **웹**: NextAuth 세션(쿠키, JWT 전략).
- **앱(Flutter)**: NextAuth는 웹세션 중심이라 그대로 못 씀 → **토큰 발급 레이어를 명시 추가**. `/api/app/v1/auth/*`가 **access + refresh JWT** 발급, Flutter는 secure storage 보관.
  - 이메일/PW: `/auth/login` → JWT.
  - 소셜(Google/Kakao): 앱이 네이티브/PKCE OAuth → code/token을 우리 API에 전달 → 검증 후 **우리 JWT 발급**(외부 의존 없음).
- 웹·앱이 **동일 user/account 저장소(Postgres, NextAuth 어댑터)** 공유. 우리 API가 앱 인증의 주체.
- refresh token은 원문 저장 없이 `app_refresh_tokens`에 해시로 저장하고, 회전·폐기·재사용 탐지를 서버에서 처리한다.
- ⚠ **NextAuth를 모바일에 그대로 끼우려 하지 말 것** — 웹=NextAuth 세션, 모바일=우리 토큰 엔드포인트로 명확히 분리 설계.

## 5. Python 별도 repo (cunote-ingestion)
- 별도 repo 적절: 런타임·배포(GCP)·케이던스(배치)·CI가 다름. **Supabase Postgres 공유**.
- 계약 = DB 스키마 + criteria JSON Schema(web repo 소유분 소비).
- **마이그레이션 단일 소유 = cunote-web**(이중 마이그레이션 금지). Python은 읽기/쓰기만, 스키마 변경 권한 없음(분리 후에도 동일).

## 6. 배포 토폴로지
**MVP**:
```
[Cloudflare] → [Vercel: Next.js 단일 (웹 + 어드민 + 앱API)] ↔ [Supabase PG]
                                                              ↔ [GCP: Python 워커]
                                                              ↔ [R2] [Upstash]
[Flutter 앱] ──(/api/app/v1, 토큰)──┘
```
**분리 마일스톤(안정화 후)**: 어드민 → 별도 Next 앱/도메인(SSO·IP 게이트), 앱 API → 독립 서비스(스케일·보안 격리). `core` 추출로 **무재작업**.

## 7. 리스크
- **계약 드리프트**(3 소비자) → OpenAPI codegen·버전·CI 계약테스트.
- **어드민이 공개 배포에 동거**(보안 표면) → 강한 role/IP 게이트, 분리 마일스톤으로 해소.
- **모바일 토큰 보안** → refresh 회전·짧은 access TTL·secure storage.
- **core가 Next에 결합되면 분리 불가** → core의 Next 비의존을 **의존성 린트 규칙으로 강제**.
- Vercel 서버리스 실행 제한 → 앱 API는 짧은 작업만(무거운 건 이미 GCP Python).

## 8. 결정
- 마이그레이션 도구 = **Drizzle**(TS-first·경량). 계약 위치 = **초기 web 내 `packages/contracts`**(마찰 시 독립 추출).
- 모바일 소셜 로그인 = **시스템 브라우저 PKCE**(SDK 종속 최소), 카카오만 네이티브 검토.
- 도메인 = **changupnote.com**. 앱 토큰 서명 `JWT_SECRET`, 본인인증 PASS.
