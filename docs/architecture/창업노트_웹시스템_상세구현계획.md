# 창업노트 — 웹 시스템 상세 구현계획

작성일: 2026-06-26 · 기준: 기존 T0/T1/T4 세로 슬라이스 검증 이후, 실제 웹 서비스 구조 착수 계획.

## 0. 결론

이제 웹 시스템 구현을 시작해도 된다. 단, 방향은 "새로 크게 갈아엎기"가 아니라 **현재 동작 중인 라이브 매칭 콘솔을 내부 검증 도구로 보존하고, 그 옆에 서비스용 화면/API/core use-case를 병렬로 세우는 것**이다.

첫 웹 구현 목표는 다음 한 문장으로 고정한다.

> 사용자가 `https://dev.changupnote.com`에 접속해 사업자번호를 입력하면, 확정/조건부를 분리한 티저를 보고, 로그인 전후 흐름을 거쳐 기회 맵·신청 준비 시트·NPC 액션 큐의 최소 동작을 확인한다.

정확도·운영 품질 주장은 아직 하지 않는다. 골든셋 P/R, 팝빌 Vercel 런타임, 캐시 약관, Supabase RLS는 별도 게이트다.

## 2. 현재 코드 기준선

이미 있는 것:

- `apps/web`: Next.js 앱, `pnpm dev:web` 포트 4010, Cloudflare tunnel 개발 접근 가능.
- `apps/web/src/app/api/matches/live/route.ts`: 팝빌 + K-Startup + 기업마당 + LLM을 호출하는 내부 검증 API.
- `apps/web/src/features/live-match/LiveMatchConsole.tsx`: 내부 MVP 콘솔.
- `packages/contracts`: criteria/Grant/CompanyProfile/MatchResult 기본 타입.
- `packages/core`: K-Startup normalize, 기업마당 fetch/LLM 추출, 팝빌 profile, 규칙 매칭 엔진.
- `db/0001_initial.sql`: 통합 DB 초안.

아직 없는 것:

- 서비스용 라우트 그룹 `app/(web)`, `app/(admin)`, `app/api/web/*`, `app/api/app/v1/*`.
- 인터페이스 문서의 DTO: `StatsResult`, `TeaserResult`, `MatchCard`, `ApplySheet`, `RoadmapNode`, `ActionQueue`.
- 화면 단위 core use-case: stats, teaser, dashboard, apply sheet, action queue.
- repository 계층과 DB 연결 추상화.
- NextAuth/권한/RLS/동의 플로우.
- 실제 서비스 UI.

## 3. 구현 원칙

1. **내부 콘솔 보존**: `/api/matches/live`와 `LiveMatchConsole`은 검증 도구로 남긴다. 서비스 화면은 별도 route group으로 만든다.
2. **core 우선**: 화면에서 직접 매칭 로직을 만들지 않는다. `packages/core`가 use-case 결과 DTO를 만들고, Next 라우트는 얇게 감싼다.
3. **계약 먼저**: `packages/contracts`에 화면 DTO를 먼저 둔다. Flutter 앱 API/OpenAPI로 확장될 타입이므로 임의 local type을 남발하지 않는다.
4. **DB는 어댑터 뒤에 둔다**: 초반에는 live external/fixture adapter로 화면을 세우고, Supabase 연결은 repository interface 뒤에 붙인다.
5. **unknown을 가치로 쓴다**: unknown은 오류가 아니라 `NextQuestion`, `ActionQueue`, `conditional` bucket의 원천이다.
6. **PII 출력 금지**: 사업자번호 원문, 대표자명, 상세주소는 화면/로그에 직접 노출하지 않는다.
7. **정확도 표현 제한**: 골든셋 P/R 전에는 "정확도 검증 완료" 또는 선정가능성 확정 표현을 쓰지 않는다.

## 5. 단계별 구현 계획

### W0. 서비스 계약 확장

목표: 화면이 사용할 DTO를 먼저 고정한다.

작업:

- `packages/contracts/src/dto.ts` 추가.
- `StatsResult`, `TeaserResult`, `MatchCard`, `RuleTraceChip`, `ApplySheet`, `RequiredDocument`, `RoadmapNode`, `NextQuestion`, `ActionQueueItem` 정의.
- 기존 `RuleTraceEntry`를 화면용 `RuleTraceChip`으로 변환하는 매핑 계약 정의.
- `Grant.support_amount`와 `required_documents` 타입을 contracts에 추가한다.

Exit:

- `pnpm typecheck` 통과.
- 기존 `/api/matches/live` 타입 깨짐 없음.

### W1. core use-case 계층

목표: UI가 바로 쓸 수 있는 화면 단위 결과를 `packages/core`에서 만든다.

작업:

- `buildStats`: open 공고 수, 총 지원금, 마감임박 수 산출.
- `buildTeaser`: 회사 프로필 + 후보 공고 매칭 → 확정 금액, 조건부 추가 가능 금액, count 산출.
- `buildDashboard`: `MatchResult` + `NormalizedGrant` → `MatchCard[]`, bucket 분류.
- `buildApplySheet`: `rule_trace`를 satisfied/needsCheck/document/preferredMiss로 재분류.
- `buildActionQueue`: unknown, 미충족 조건, eligible+마감임박을 행동 후보로 합산.
- `buildRoadmap`: now/soon/preparable/conditional node 생성. 시간 전이값은 첫 버전에서는 계산 가능한 biz_age만 지원.

Exit:

- fixture 기반 단위 검증 스크립트 또는 테스트로 각 use-case 출력 확인.
- unknown이 `conditional`, `nextQuestion`, `ActionQueueItem`으로 이어지는지 확인.

### W2. 웹 BFF 라우트 스캐폴딩

목표: 서비스 UI가 내부 콘솔 API에 의존하지 않고 화면 지향 API를 호출한다.

작업:

- `GET /api/web/stats`
- `POST /api/web/teaser`
- `GET /api/web/matches`
- `GET /api/web/next-question`
- `GET /api/web/action-queue`
- 초반 adapter는 fixture/live source 중 선택 가능하게 둔다.
- 오류 응답은 웹 내부용 `ActionResult<T>`로 통일한다.

Exit:

- `curl http://127.0.0.1:4010/api/web/stats`가 값 반환.
- `POST /api/web/teaser`가 raw PII 없이 `TeaserResult` 반환.

### W3. 첫 화면 + 티저

목표: 문서상 토스식 첫 화면 훅을 실제 첫 화면으로 만든다.

작업:

- 기존 루트 `page.tsx`는 서비스 첫 화면으로 교체.
- 내부 콘솔은 `/internal/live-match` 같은 별도 경로로 이동.
- 첫 화면: 집계 숫자, 사업자번호 입력, 예비창업 진입 링크.
- 티저: 확정 금액과 조건부 금액을 분리 표시.
- 접근성: 카운트업은 `prefers-reduced-motion`에서 정적 표시.

Exit:

- `https://dev.changupnote.com` 첫 화면 진입 가능.
- 사업자번호 입력 후 티저 확인.
- 원문 사업자번호가 화면에 그대로 남지 않음.

### W4. 기회 맵 + NPC 액션 큐

목표: 목록형 결과가 아니라 `now / soon / preparable / conditional` 맵과 다음 할 일을 보여준다.

작업:

- `features/opportunity-map`: bucket별 매칭 카드.
- `features/action-queue`: 레버리지, 긴급, 쉬움 기준 상위 행동.
- 카드 필수 필드: eligibility, fitScore, 지원금, D-day, rule_trace 요약, 확정/추정 라벨.
- 부적격은 접힌 영역에 사유와 함께 표시한다.

Exit:

- 적격/조건부/부적격이 같은 데이터에서 분리 렌더링.
- unknown 기반 action이 최소 1개 이상 생성되는 fixture 확인.

### W5. 신청 준비 시트 + 로드맵

목표: 공고 상세에서 왜/얼마/언제/어떻게/뭐가 필요한지를 한 화면에 보여준다.

작업:

- `features/apply-sheet`: satisfied, needsCheck, documents, schedule, deepLink.
- `required_documents`가 없으면 날조하지 않고 빈 섹션 또는 원문 확인으로 처리.
- `features/roadmap`: 지금 적격, 곧 적격, 준비하면 적격, 조건부 node 표시.
- `match_events` 연결 전까지 apply click은 route scaffold만 둔다.

Exit:

- 매칭 카드에서 공고 상세 진입 가능.
- rule_trace가 준비 체크리스트로 재프레이밍되어 보임.

### W6. 저장소/인증/권한 연결

목표: fixture/live adapter에서 DB-backed 서비스로 넘어갈 준비를 한다.

작업:

- repository port 정의: `GrantRepository`, `CompanyRepository`, `MatchRepository`, `FeedbackRepository`.
- Supabase/Drizzle 연결 전용 adapter 추가.
- NextAuth skeleton: 웹 세션, company access guard.
- `companies`, `user_company`, `company_profiles` 조회/저장 use-case 연결.
- 앱 API는 OpenAPI/codegen 전에 route skeleton과 envelope만 만든다.

Exit:

- DB 없이도 화면은 유지되고, DB adapter는 interface 뒤에서 교체 가능.
- 권한 guard가 없는 route와 필요한 route가 명확히 분리됨.

### W7. 어드민/플라이휠 최소 골격

목표: 라벨링과 평가를 나중에 붙일 자리를 미리 비워 둔다.

작업:

- `(admin)` route group 생성.
- `extraction_log`, `feedback`, `golden_set`, `eval_runs` 조회 화면은 MVP 이후.
- 첫 구현에서는 admin access gate와 placeholder만 둔다.

Exit:

- 공개 웹/어드민/API route 경계가 코드 구조상 분리됨.

## 6. 우선순위

1. **W0 + W1**: DTO와 core use-case부터 만든다. 이것이 웹/앱/어드민 공통 계약이다.
2. **W2 + W3**: `dev.changupnote.com`에서 보이는 첫 화면과 티저를 붙인다.
3. **W4**: 기회 맵과 NPC 액션 큐를 만든다. 차별화의 중심이다.
4. **W5**: 신청 준비 시트와 로드맵을 붙인다.
5. **W6**: DB, 인증, 권한, 저장을 붙인다. 초반 화면 구현을 막지 않도록 adapter 뒤에 둔다.
6. **W7**: admin/플라이휠은 경계부터 만들고 기능은 골든셋 준비와 함께 확장한다.

### Milestone Web-4: DB/Auth 연결 준비

## 9. 검증 루틴

```bash

pnpm dev:web
```

```bash
curl http://127.0.0.1:4010/api/web/stats
```

브라우저 확인:

- 데스크톱: `https://dev.changupnote.com`

보안 확인:

- 응답/화면/로그에 사업자번호 원문, 대표자명, 상세주소가 노출되지 않는지 확인.
- 실패 응답에 외부 API secret, raw provider payload가 섞이지 않는지 확인.
