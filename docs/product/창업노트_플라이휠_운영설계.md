# 창업노트 — 지속개선 플라이휠 운영 설계 (SOP)

작성일: 2026-06-19 · 역할: 추출·매칭을 측정·라벨링·개선하는 운영 루프의 실행 사양

## 0. 두 개의 골든셋
| 골든셋 | 정답 단위 | 용도 | 초기 규모 |
|---|---|---|---|
| **추출 골든** | 공고 → grant_criteria 필드값 | 추출 정확도 측정 | 50~100건 → 증분 |
| **매칭 골든** | (기업 × 공고) → 적격/부적격 + 근거 | 판정·랭킹 정확도 | 30~50쌍 → 증분 |

- 큐레이션: **2인 라벨링 → 불일치는 도메인 전문가가 조정**(불일치 자체가 spec의 모호점 → 표준화 프로토콜 갱신).
- 버전 관리(`golden_ver`). 라벨 가이드(annotation guideline) 문서화.

## 1. 라벨링 큐 — 진입·우선순위
**진입 조건**
- 추출 confidence < 임계값, 또는 `needs_review=true`
- 사용자 "부적격/틀림" 피드백이 달린 매칭
- 무작위 샘플 X%(편향 방지, 고신뢰도 구간도 점검)
- dead-letter(변환/추출 실패)

**우선순위** = 공고 가치·트래픽 × 영향도(이 라벨이 해소할 conditional·후보 수). 高가치·高노출 공고 먼저.

## 2. 신뢰도 임계값 보정 (calibration)
- 골든셋으로 **confidence ↔ 실제 오류율 곡선**을 차원별로 추정.
- 오류율이 급증하는 지점을 임계값으로 → 미만은 "확인 필요" 표기 + HITL 라우팅.
- 분기별 재보정(모델·프롬프트 바뀌면 재산정).

## 3. 평가 하니스 (CI 게이트)
- **모든** prompt/ruleset/model/taxonomy 변경 → 골든셋 대비 자동 평가.
- 지표: 추출 필드 P/R · 적격판정 P/R · 랭킹 p@k·MRR · calibration.
- **회귀 시 활성화 차단**(조용한 품질 저하 방지). 통과 + 사람 승인만 배포.

## 4. 피드백 수집·해석
| 신호 | 예 | 강도 | 주의 |
|---|---|---|---|
| 암묵 | 노출·클릭·저장·신청버튼 | 약 | 포지션 편향 보정 |
| 명시 | 적합/부적합 버튼 | 중 | 자기선택 편향 |
| 결과(outcome) | 신청→선정/탈락 | 강(희소) | 매칭 골든의 최고 정답 |
| 리뷰어 | 라벨링 큐 정정 | 강 | 추출 골든으로 |

- outcome은 **선정가능성 모델의 학습 라벨**이 되어 §매칭 ㉡를 가동.

### 4.1 결과 추적이 어려운 전제의 1차 폐쇄 루프

신청 결과는 늦고 누락되므로 초기 플라이휠의 중심 라벨로 두지 않는다. 1차 정확도 개선은 **eligibility correction**으로 닫고, outcome은 늦게 들어오는 강한 보조 라벨로 취급한다.

| 레이어 | 수집 신호 | 반영 대상 | 자동 반영 여부 |
|---|---|---|---|
| 행동 | `saved`, `dismissed`, `applied` | 랭킹/UX 우선순위 | 자동 집계 가능 |
| 오류 신고 | `wrong` + `reasonCode` + `correction` | 리뷰 큐, 매칭 골든 후보 | 사람 확인 후 반영 |
| 신청 막힘 | `blocked` + `portal_blocked` | criteria/profile/taxonomy 보정 후보 | 사람 확인 후 반영 |
| 결과 | `selected`, `rejected` | 선정가능성/경쟁력 모델 | 표본 충분 시 반영 |

**저장 계약**
- `feedback.kind`: `saved | dismissed | wrong | applied | selected | rejected | blocked | note`
- `feedback.type`: `selected/rejected/blocked` 또는 명시 outcome은 `outcome`, 나머지는 관련/비관련/암묵 신호로 저장
- `feedback.value.correction`: `{ dimension, criterionId, expectedEligibility, correctedEligibility, correctedResult, note }`
- `feedback.value.reasonCode`: `wrong_high | wrong_low | wrong_condition | profile_wrong | criteria_wrong | taxonomy_gap | portal_blocked | selected | rejected | other`

**운영 원칙**
1. `wrong`과 `blocked`는 곧바로 ruleset을 바꾸지 않고 리뷰 큐로 보낸다.
2. 리뷰어는 `rule_trace`의 한 조건을 기준으로 원인층을 분류한다: 공고 추출 오류, 회사 프로필 오류, taxonomy 누락, 규칙 해석 오류, 포털/기관 재량.
3. 같은 원인층이 반복될 때만 `grant_criteria`, taxonomy, profile enrichment, ruleset/scoring 버전을 바꾼다.
4. `selected/rejected`는 eligibility 정답이 아니라 선정가능성 라벨이다. 선정/탈락만으로 "신청 자격"을 고치지 않는다.
5. 새 버전은 골든셋과 eval run을 통과해야 활성화한다.

**구현 연결**
- `review_queue`는 별도 테이블이 아니라 `feedback`에서 파생한다. 조건: `kind in (wrong, blocked)`, 보정 reasonCode, 또는 `correction` 존재.
- 정정값이 충분한 항목은 `golden_set.kind=matching`, `ref=feedback:{feedbackId}`, `golden_ver=feedback-matching-candidates-v1` 후보로 승격할 수 있다.
- 후보 `gold.expected`는 `correction.correctedEligibility`를 사용한다. 이 값이 없으면 리뷰 큐에는 남지만 골든셋 후보로 저장하지 않는다.
- 후보는 아직 최종 authoritative fixture가 아니다. 리뷰어 승인 후 정식 골든셋 버전으로 편입하고, 이후 `eval_runs`로 회귀 검증한다.
- 관리자 API `GET /api/admin/flywheel/matching-eval?goldenVer=...`는 저장 없이 현재 `match_state` 대비 지표를 계산하고, `POST /api/admin/flywheel/matching-eval`은 같은 결과를 `eval_runs.target=matching`에 저장한다.
- `eval_runs.metrics`는 `accuracy`, `coverage`, `missingState`, `missingGrant`, `eligiblePrecision/Recall`, `conditionalPrecision/Recall`, `ineligiblePrecision/Recall`, `gatePass`를 남긴다. `version_refs`에는 `rulesetVer`, `scoringVer`, `goldenVer`, `runner`를 남겨 품질 변화의 원인을 추적한다.
- 활성화 게이트는 최소 `coverage=1`, `accuracy=1`, class recall pass, 사람 승인 조건을 통과해야 한다. `feedback-matching-candidates-v1`은 후보 버전이므로 정식 배포 게이트에는 별도 curated `golden_ver`로 승격한 뒤 사용한다.

## 5. 버전 레지스트리 & 활성화 워크플로
```
새 버전 제안(prompt/ruleset/model/taxonomy)
  → 평가 하니스(골든셋) → 회귀 없음 + 사람 승인
  → 활성화(activated_at) → 모니터링
  → 지표 악화 시 롤백
```
- 모든 `extraction_log`·`match_state`·`match_events`에 버전 부착 → **품질 변화 귀속**(attribution).

## 6. Backfill 오케스트레이션
- 트리거: prompt/model/taxonomy 메이저 변경.
- 범위: 영향받는 공고만(예: 특정 dimension 추출 변경 → 해당 criteria 보유 공고).
- 예산 캡(LLM 비용) + 오프피크 실행(Cloud Run Jobs). 신·구 `eval_runs` 비교 후 교체.

## 7. 드리프트 모니터링
- 추출 confidence 분포, 적격률, 소스 스키마 해시, 변환 실패율의 시계열 → 이상 변동 알림.
- 사용자 피드백의 정/부 비율 추세.

## 8. 인력 역할
| 역할 | 일 |
|---|---|
| 리뷰어 | 라벨링 큐 처리(추출·매칭 정정) |
| 도메인 전문가 | 골든셋 큐레이션·spec(표준화) 갱신·불일치 조정 |
| 운영 | 드리프트·비용·dead-letter 모니터링 |

## 9. 플라이휠 KPI
- 라벨 커버리지·골든셋 성장, 버전별 품질 추세(P/R·p@k), **자동:HITL 비율**(시간이 갈수록 자동↑), 평균 추출 confidence, 사용자 부적격 신고율↓.

## 10. 미결정
- 무작위 샘플링 비율 X, conditional 노출 정책 세부, outcome 수집 UX(사용자가 선정결과 보고하게 만드는 유인).
