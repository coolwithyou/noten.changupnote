## 0. 이 가이드의 의사결정

오늘은 주말이므로 CODEF·NICE 상담, 공공 마이데이터 이용기관 협의처럼 사람의 승인이나 계약이 필요한 일은 임계경로에서 뺀다.

우선순위는 아래와 같다.

1. 사업자번호로 정확 조회되는 무료 공식 API
2. 사업자번호·법인번호로 정확 조인 가능한 공개 데이터와 배치 색인
3. 사용자가 직접 입력하더라도 자유 텍스트가 아닌 선택형·날짜형·숫자형 구조 입력
4. 평일에만 진행 가능한 기관 승인·유료 계약
5. CODEF는 데모 검증과 보조 경로로 유지하되, 자동채움의 유일한 전제로 두지 않음

현재 22축 중 `premises`, `export_performance` 예약축과 자유입력 `other`를 제외하면 운영상 구조화 대상은 19축이다. 이 가이드의 목표치는 다음과 같다.

## 2. 공통 완료 기준

각 STEP은 아래 다섯 항목을 모두 만족해야 완료로 본다.

- [ ] 원천이 실제로 해당 필드를 제공한다는 공식 근거 또는 실응답이 있다.
- [ ] 값이 없는 경우와 API 실패를 구분한다.
- [ ] `live/cache/pending/failed/n-a/self-declared` 상태가 정확히 표시된다.
- [ ] 원천·기준일·confidence가 저장 또는 표시된다.
- [ ] fixture 테스트와 웹 타입체크가 통과한다.

결격·제재 데이터는 추가 조건이 있다.

- 명단에서 발견된 경우 `present`로 판단할 수 있다.
- 명단에 없다는 사실을 `없음`으로 확정하려면 해당 명단이 소진적이고 최신이라는 근거가 있어야 한다.
- 비소진적 명단은 부재를 통과 근거로 쓰지 않고 객관식 질문으로 보완한다.

- 조회 경로: `devServiceDataMonitor.runRegistryConnector`
- 현재 적재 방식: CSV 파싱 후 `replaceBySource()`가 한 트랜잭션에서 해당 소스의 기존 행을 삭제하고 새 행을 삽입

> **구현 갱신 (2026-07-12)**: 아래 버전형 모델과 `/registry-imports` ops 화면을 구현했고 마이그레이션 `0044_boring_stellaris.sql`을 적용했다. 기존 `replaceBySource()`는 레거시 CLI 호환용으로 남아 있지만 ops 발행 경로에서는 사용하지 않는다.

현재 방식은 개발 검증에는 충분하지만 운영 배치에는 다음 위험이 있다.

- 빈 파일·잘못된 인코딩·헤더 변경으로 파싱 결과가 0건이어도 기존 데이터를 모두 삭제할 수 있다.
- 어떤 CSV 파일을 언제 적재했는지, 체크섬·행 수·파서 버전·실패 원인이 별도 이력으로 남지 않는다.
- 이전 정상 버전으로 즉시 되돌리는 rollback 포인터가 없다.
- `hasSource()`는 행이 한 건만 있어도 소스가 존재한다고 판단하므로, 오래된 조달청 스냅샷으로 `명단 부재=제재 없음`을 확정할 수 있다.
- 중대재해 CSV의 `재해발생연도`가 유효기간으로 모델링되지 않아 과거 사건이 영구히 현재 상태처럼 보일 수 있다.

따라서 운영 반영은 `CSV → registry_index 직접 교체`가 아니라 다음 파이프라인으로 바꾼다.

```text
[원본 CSV 확보]
  → [원본 보관 + SHA-256]
  → [인코딩/헤더 판별]
  → [파싱·정규화]
  → [import run별 staging 적재]
  → [품질 gate]
  → [active run 포인터 원자적 전환]
  → [dev 조회 검증]
  → [이전 run 보존/필요 시 즉시 rollback]
```

### DB 반영 모델 — 버전형 적재

기존 `registry_index`를 유지하되, 다음 두 메타 테이블과 컬럼을 추가한다.

#### `registry_import_runs`

CSV 한 번의 반입을 나타낸다.

| 컬럼 | 용도 |
|---|---|
| `id` | import run UUID |
| `source` | `data.go.kr:15137996` 같은 소스 식별자 |
| `status` | `staged`, `validated`, `published`, `failed`, `superseded` |
| `filename` | 원본 파일명 |
| `sha256` | 같은 파일 중복 반입·변조 확인 |
| `raw_object_key` | R2 원본 보관 경로. 로컬 검증이면 null 가능 |
| `source_published_at` | 포털에 표시된 파일 기준일/수정일 |
| `downloaded_at` | 파일 확보 시각 |
| `parser_version` | 어댑터 버전 |
| `schema_signature` | 정규화한 CSV 헤더의 해시 |
| `raw_row_count` | 원본 데이터 행 수 |
| `parsed_row_count` | 정상 변환 행 수 |
| `rejected_row_count` | 누락·형식 오류로 거부한 행 수 |
| `exact_key_count` | 유효 사업자번호 또는 법인번호 보유 행 수 |
| `active_row_count` | 적재 시점에 유효한 행 수 |
| `error_summary` | 실패 원인. 원문 사업자번호·민감값은 저장하지 않음 |
| `created_at`, `completed_at` | 수행 시각 |

`(source, sha256)` unique 제약으로 동일 파일을 중복 적재하지 않는다.

#### `registry_source_state`

소스별 현재 공개 버전을 가리키는 작은 포인터 테이블이다.

| 컬럼 | 용도 |
|---|---|
| `source` | PK |
| `active_run_id` | 조회에 사용할 성공 import run |
| `last_success_at` | 마지막 성공 시각 |
| `fresh_until` | 부재를 신뢰할 수 있는 기한 |
| `last_error_at`, `last_error_code` | 운영 상태 표시 |

#### `registry_index` 추가 컬럼

- `import_run_id`: 어느 CSV 버전에서 온 행인지 나타내는 FK
- `source_record_key`: 소스 내 행의 안정 식별자 또는 canonical 행 해시
- `source_year`: 중대재해처럼 연도 스냅샷인 데이터의 기준연도

조회는 `registry_source_state.active_run_id = registry_index.import_run_id`인 행만 사용한다. 새 run을 발행할 때 대량 행을 삭제하지 않고 포인터 하나만 트랜잭션으로 바꾼다. rollback은 `active_run_id`를 직전 성공 run으로 되돌리는 것으로 끝난다.

### 원본 CSV 보관 정책

- 로컬 개발: 저장소 밖 임시 경로를 사용한다. CSV 원본은 git에 커밋하지 않는다.
- 운영: R2의 `registry-raw/<source>/<yyyy-mm-dd>/<sha256>.csv`에 원본 그대로 보관한다.
- 원본과 별도로 UTF-8 변환본을 만들 수 있지만, 원본 체크섬은 변환 전 파일을 기준으로 한다.
- 원본 보관기간은 기본 1년, DB import run과 직전 성공 원본 2개 이상은 항상 남긴다.
- 공개 명단이어도 원본을 웹에서 직접 노출하지 않는다.

### CSV 검증 gate

아래 항목을 모두 통과해야 `published`로 전환한다.

1. 파일 크기가 0이 아니고 허용 상한 이내다.
2. UTF-8, CP949, EUC-KR 중 인코딩을 판별하고 UTF-8로 안전하게 변환한다.
3. 필수 헤더가 존재하고 `schema_signature`가 알려진 버전과 일치한다.
4. 파싱 결과가 0건이면 무조건 실패한다.
5. 새 행 수가 직전 성공 run 대비 소스별 허용 범위 안에 있다.
   - 기본: `-30% ~ +100%`
   - 연간 스냅샷은 최초 적재 후 실제 변동폭을 보고 조정
6. 필수 필드 결측률과 날짜 파싱 실패율이 허용치를 넘지 않는다.
7. 사업자번호가 있는 소스는 10자리 형식과 체크섬을 검증한다.
8. 중복 `source_record_key` 비율이 허용치를 넘지 않는다.
9. 로그 샘플은 사업자번호를 마스킹한다.
10. `known_on_absence` 소스는 **완전한 전량 스냅샷임이 확인되고 fresh_until 이내일 때만** 부재를 확정한다.

검증 실패 시 staging run만 `failed`로 남기고 현재 `active_run_id`는 바꾸지 않는다.

### 소스별 해석 정책

| 소스 | DB 조인 | polarity | freshness/유효기간 정책 |
|---|---|---|---|
| 조달청 부정당 `15137996` | 사업자번호·법인번호 exact | `known_on_absence` | 전량 스냅샷·최신성 gate 통과 시에만 부재 확정. 행 자체는 제재 시작·종료일로 active 계산 |
| 벤처확인 `15084581` | 상호+지역 fuzzy | `present_only` | 확인서 시작·종료일 안에 있는 행만 현재 보유 후보. exact API `15125362`가 있으면 그 값 우선 |
| 중대재해 `15090150` | 상호+지역 fuzzy | `present_only` | 영구 active 금지. `재해발생연도`를 보존하고 공고가 요구하는 최근 N년 조건에서만 판정 |

중대재해 행은 회사 자체보다 `사업장명(현장명)`일 수 있으므로 자동 확정하지 않고 사용자 확인 후보로만 노출한다.

### 배치 명령 인터페이스

최종적으로 다음 네 명령으로 운영할 수 있게 한다.

```bash
# 1. DB 미접촉 검증
pnpm registry:validate --source procurement-debarment --file <csv>

# 2. staging 적재 + 품질 gate + 발행
pnpm registry:import --source procurement-debarment --file <csv>

# 3. 소스별 마지막 성공·행 수·freshness 확인
pnpm registry:status

# 4. 직전 성공 run으로 복귀
pnpm registry:rollback --source data.go.kr:15137996 --run <run-id>
```

기존 `scripts/registry/build.ts --dry-run`은 `registry:validate` 내부에서 재사용하고, 현재 `replaceBySource()`는 버전형 publish가 완성될 때까지 개발용으로만 유지한다.

현재 구현된 로컬 무DB 검증 명령은 다음과 같다.

```bash
pnpm exec tsx --tsconfig apps/admin/tsconfig.json scripts/registry/preview-upload.ts \
  --source venture-confirmation --file <csv>
```

운영에서는 ops `/registry-imports` 화면이 R2 직접 업로드, 미리보기, 기존 활성 행 비교, 발행, 이력, rollback을 제공한다.

### 갱신 실행 방식

#### 1단계 — 지금: 수동 반입

1. 사용자가 data.go.kr에서 최신 CSV를 내려받는다.
2. 저장소 밖 경로에 둔다.
3. Codex가 `registry:validate`를 실행하고 품질 리포트를 확인한다.
4. 통과하면 `registry:import`로 staging·publish한다.
5. dev 하네스에서 알려진 hit·non-hit 표본을 검증한다.

파일 다운로드 URL이 안정적이지 않거나 로그인이 필요해도 이 단계는 동작한다.

#### 2단계 — 직접 다운로드 URL 확인 후: 예약 배치

- 안정된 다운로드 URL과 이용 조건이 확인된 소스만 주기 작업으로 전환한다.
- 작업은 `다운로드 → R2 원본 보관 → validate → import → status 알림` 순서로 실행한다.
- 다운로드·검증 실패는 기존 active run을 유지한다.
- 갱신주기는 포털의 실제 수정 주기에 맞춘다. 임의로 매일 호출하지 않는다.

#### 3단계 — 운영 모니터링

- `fresh_until` 임박, 최근 실패, 행 수 급변을 운영 화면에 표시한다.
- `known_on_absence` 소스가 stale이면 `제재 없음` 대신 `확인 필요`로 강등한다.
- 성공·실패 요약만 알림으로 보내고 CSV 원문이나 사업자번호는 보내지 않는다.

# STEP 2 — API로 남은 빈칸을 최소 객관식으로 보완

## 2.1 질문 설계 원칙

- 질문은 현재 매칭 결과를 바꿀 가능성이 있는 것만 보여준다.
- `예/아니오`만으로 불확실한 질문에는 반드시 `모름`을 둔다.
- 민감하거나 법적 의미가 있는 질문은 “없음”을 기본 선택하지 않는다.
- API 값이 있으면 질문을 숨기되 사용자는 `값 수정 요청`을 할 수 있다.
- 범위형 답변이 공고 임계값을 가로지르면 그때만 정확값을 추가로 묻는다.
- 한 화면에 모든 질문을 나열하지 않고, 정보이득이 큰 질문부터 한 장씩 보여준다.
- 자유 텍스트는 `other`와 프로그램명 검색처럼 선택지를 만들기 위한 검색 입력에만 제한한다.

# STEP 3 — 표본 측정과 자동채움 ROI 결정

## 표본 설계

첫 측정은 최소 30개, 안정화 판단은 100개 이상을 사용한다.


| 층             | 최소 표본 |
| ------------- | -----: |
| 개인 일반과세       | 10    |
| 개인 간이·면세      | 10    |
| 소형 법인         | 10    |
| DART 공시 법인    | 5     |
| 휴·폐업 또는 경계 사례 | 5     |


중복 가능하므로 전체 수는 조정할 수 있다. 실기업 번호는 사용 권한이 있는 데이터만 사용한다.

## 사용자 작업

- [ ] 테스트 가능한 사업자번호 표본을 제공하거나 사용 가능한 공개 표본 범위를 승인한다.
- [ ] 정답 대조가 가능한 사업자 3~5곳의 실제 값을 확인한다.

## Codex 작업

- [ ] 사업자번호 원문을 결과 리포트에 노출하지 않고 익명 표본 ID로 기록한다.
- [ ] 소스별 응답률, 필드별 정확도, 호출비용, latency를 측정한다.
- [ ] 다음 지표를 코호트별로 산출한다.
  - 사업자번호만으로 채워진 축
  - 객관식 질문 수
  - 권위 원천으로 확정된 축
  - 잘못 채운 값과 충돌률
  - 질문 1개당 해소된 공고 unknown 수

## Go/no-go 기준

- API 응답률이 20% 미만이고 특정 고가치 코호트도 없다면 기본 호출에서 제외한다.
- 정확 조인 API는 응답률이 낮아도 비용이 거의 없으면 조건부 호출로 유지한다.
- 퍼지 소스는 precision 95% 미만이면 자동 확정하지 않고 사용자 확인 후보로만 쓴다.
- 질문은 응답률이 낮거나 매칭 결과를 거의 바꾸지 않으면 온보딩에서 제거한다.

- STEP 1D의 버전형 import run, active pointer, ops 발행·rollback은 이미 구현되어 있다. 1D는 적재 인프라가 아니라 exact/fuzzy 해석과 사용자 확인 UI만 남았다.
- `scripts/verify-datago-connectors.ts`는 1A 라이브 스모크의 초안이다. 단, 성공·정상 빈값·전제 미충족·API 실패를 기계 판독 가능한 결과로 남기도록 보강해야 한다.

## 2. 구현 순서와 중단 조건

### Phase A — 측정 계약 고정 (STEP 0)

**목표:** 커넥터를 더 붙이기 전에 어떤 값을 API 확정으로 세는지 단일 계약으로 고정한다.

1. `FieldSourceRef`와 물리 provider를 분리한 `EvidenceSourceKind` 타입을 공용 계층에 추가한다.
   - 허용값: `authoritative_api`, `public_registry`, `auth_supplied`, `self_declared`, `derived`
   - provider는 `nts`, `codef`, `kcomwel`, `fsc`, `registry` 등 실제 조회처를 별도 필드로 유지한다.
2. `FieldCoverageRow`에 `sourceKind`, `asOf`, `axisCompleteness`를 추가한다. 하위 플래그 행은 부모 축 커버리지를 직접 증가시키지 않는다.
3. 순수 함수 `measureAutofillCoverage(rows, grantWeights?)`를 core에 둔다.
   - 분모: 운영 구조화 대상 19축
   - `authoritative_axis_coverage`: `authoritative_api | public_registry`이며 부모 축이 판정 가능한 경우만 계산
   - `total_answered_coverage`: 위 원천과 `auth_supplied | self_declared | derived` 포함
   - `grant_weighted_coverage`: 활성·검수된 공고 criterion 빈도를 dimension별 가중치로 적용
4. dev 모니터 요약을 `API 확정` / `전체 판정 가능` / `공고 가중` 세 값으로 교체하고, 각 행에 원천 종류·provider·기준일을 노출한다.
5. CODEF의 생년월일·성별은 `auth_supplied`로 고정하고 `founder_age`, `founder_trait`을 API 확정 분자에서 제외한다.

**주요 수정 지점**

- `packages/contracts/src/dto.ts` 또는 자동채움 전용 core 모듈
- `apps/web/src/lib/server/devServiceDataMonitor.ts`
- `apps/web/src/features/dev/ServiceDataMonitor.tsx`
- 신규 `packages/core/src/autofill/coverage.ts` 및 단위 테스트

**게이트**

- CODEF fixture에서 `founder_age` 행이 채워져도 `authoritative_axis_coverage`는 증가하지 않는다.
- `sanction.participation_restricted` 하나만 알려져도 `sanction` 부모축은 전체 판정 가능으로 세지 않는다.
- 19축의 분모와 가중치 합이 테스트에서 고정된다.

### Phase B — 기존 공공 API 재검증 (STEP 1A)

**목표:** 이미 있는 NTS·근로복지공단·금융위 커넥터의 실제 상태를 재현 가능하게 기록한다.

1. `.env.local` 파서를 이용해 `CUNOTE_DATA_GO_KR_SERVICE_KEY` 존재·빈 값 여부만 출력하는 preflight를 추가한다. 키 값과 인코딩된 키는 로그에 남기지 않는다.
2. 현재 개별 키 환경변수와 공용 키의 resolution 순서를 `resolveDataGoKrServiceKey`로 단일화한다.
3. `verify-datago-connectors.ts`의 결과를 `found`, `empty`, `skipped`, `failed`로 구분한다. 인력 식별자는 화면·JSON 결과에 마스킹하고 raw payload는 저장하지 않는다.
4. 법인번호 없는 법인과 개인사업자에 대한 금융위 호출을 `skipped` 분기의 단위 테스트로 고정한다.
5. 근로복지공단 502 등 외부 장애는 전체 조회를 실패시키지 않고 해당 필드만 `failed`로 남긴다.

**게이트**

- fixture 테스트와 마스킹 테스트가 키 없이 통과한다.
- 키가 있을 때만 라이브 스모크를 수행하며, 장애나 정상 빈값을 성공 데이터로 위장하지 않는다.
- 라이브 표본이 없으면 Phase B는 `fixture complete / live pending`으로 기록하고 Phase C의 fixture 구현을 계속한다.

### Phase C — exact 무료 소스 추가 (STEP 1B, 1C)

#### C1. KIPRISPlus

1. core에 `businessNo -> applicant/customerNo`와 `customerNo -> IP holdings`를 분리한 커넥터 계약을 만든다.
2. IP 종류를 `patent`, `utility_model`, `trademark`, `design`으로 정규화하고, 건별로 `application/publication/registration/expiration` 상태·기준일을 보존한다.
3. `ip`의 표시값과 evidence 목록을 분리한다. 매칭은 정규화된 보유 유형을 소비하고 UI는 상태별 건수를 표시한다.
4. 캐시 TTL은 30일 이상으로 하고 `(provider, scope, bizNoRef)` 단위로 동일 요청을 합친다. 월 호출 한도를 넘기기 전에 호출을 중단하는 미터를 추가한다.
5. exact miss는 `not_found`로 남기되 `IP 없음`으로 일반화하지 않는다. API가 제공하는 대상·상태 범위가 소진적임이 확인된 경우만 부재 증거로 쓴다.

#### C2. 창업기업확인서

1. `brno`를 우선하고 법인번호가 있을 때 `crno`를 보조키로 사용하는 exact 커넥터를 추가한다.
2. 응답을 `active`, `expired`, `scheduled`, `not_found`, `failed`로 구분한다.
3. `active`인 경우만 `certification`에 `창업기업확인서`를 병합하고 발급일·만료일을 evidence에 남긴다.
4. exact API 결과를 동일 인증의 registry fuzzy 후보보다 우선하되, 기존 evidence는 삭제하지 않는다.

**공통 수정 지점**

- 신규 `packages/core/src/kipris/*`, `packages/core/src/startup-confirmation/*`
- `packages/core/src/index.ts`
- `apps/web/src/lib/server/devServiceDataMonitor.ts`
- `apps/web/src/lib/server/serviceData.ts`는 아직 연결하지 않고 Phase F에서 승격한다.

**게이트**

- 성공·빈 결과·만료·스키마 변경·타임아웃 fixture가 있다.
- 키 없음은 `pending`, 전제 미충족은 `pending/skipped`, 호출 실패는 `failed`, exact miss는 `n-a/not-covered` 또는 `not_found`로 구분된다.
- 키와 사용 권한이 준비되면 사용자가 실행한 dev 서버의 `/dev/service-data`에서 표본을 검증한다.

### Phase D — registry 후보 확인 경로 (STEP 1D 잔여)

**목표:** exact hit와 fuzzy candidate를 데이터 계약과 UI 둘 다에서 명확히 나눈다.

1. `matchRegistry`의 결과를 `exact_confirmed`, `fuzzy_candidate`, `no_match`, `source_unavailable`로 정규화한다.
2. 조달청은 새 CSV에 유효한 사업자번호 또는 법인번호가 있을 때만 exact 판정을 활성화한다. 새 파일이 오기 전에는 현재 active run을 변경하지 않는다.
3. 벤처·중대재해 fuzzy candidate는 profile이나 `known_flags`에 즉시 병합하지 않고, 후보 ID·원천·기준년도·유사도를 포함한 확인 작업으로 내린다.
4. 사용자 선택은 `confirmed`, `rejected`, `unknown`으로 저장하고 원본 후보와 분리한다. 선택 변경 시 매칭을 재계산한다.
5. 중대재해는 `source_year`를 공고의 최근 N년 조건과 비교할 수 있을 때만 판정한다. 기간 조건이 없으면 확인 후보로만 남긴다.

**주요 수정 지점**

- `packages/core/src/registry/matcher.ts`, `packages/core/src/registry/types.ts`
- `apps/web/src/lib/server/devServiceDataMonitor.ts`
- 대시보드 확인 카드 또는 `ProgressiveQuestionCard` 연결
- 확인 응답을 위한 API·repository 포트

**게이트**

- fuzzy hit만으로 `certification` 보유나 `sanction` 해당을 자동 확정하지 않는다.
- exact/fuzzy가 `sourceKind`, confidence, 표시 문구에서 구분된다.
- 조달청 전량 exact CSV가 없어도 나머지 Phase의 fixture·UI 구현은 계속한다.

### Phase E — DART 법인 오버레이 (STEP 1E)

1. DART `corpCode.xml`을 주기적으로 가져와 `bizr_no -> corp_code, jurir_no`로 정규화한 역색인을 버전형으로 교체한다.
2. 역색인 miss는 `not_covered`로 즉시 종료하고 직원·재무 API를 호출하지 않는다.
3. 회사 개황에서 확보한 `jurir_no`를 금융위 커넥터의 법인번호 브리지로 넘긴다.
4. 직원·재무 evidence에 `businessYear`, `reportCode`, `statementType(consolidated/separate)`, `asOf`를 저장한다. 단순 최신 fetchedAt로 값을 덮어쓰지 않는다.
5. DART·금융위·팝빌·NICE 값은 evidence history에 모두 보존하고, 필드별 source priority resolver가 표시값을 선택하게 한다.

**게이트**

- 공시 법인 fixture에서 법인번호 브리지가 동작한다.
- 비공시 법인은 `failed`가 아니라 `n-a/not_covered`다.
- 키가 없으면 fixture까지만 완료하고 live gate를 보류한다.

### Phase F — 조건부 Q&A 제품 연결 (STEP 2)

**목표:** 현재 질문 선정기를 재사용하되, 자유 입력과 `미응답 = 모름`을 제거한다.

1. dimension별 문구·선택지·단위·정밀값 후속 조건을 `QuestionDefinition` 데이터로 정의한다. `question-planner.ts`는 질문을 직접 작성하지 않고 definition ID와 필요한 criterion threshold를 반환한다.
2. `unknown` 응답을 profile 값과 분리한 answer-state로 저장한다. 이로써 같은 질문을 반복하지 않되, 매칭 엔진에는 값을 알려지 않는다.
3. API가 해당 축을 권위 원천으로 확정하면 질문을 숨긴다. fuzzy candidate나 하위 플래그 일부만 있는 경우는 숨기지 않는다.
4. 매출·근로자는 우선 구간으로 묻고, 현재 후보 공고의 임계값을 가로지를 때만 정확 숫자를 후속 질문으로 낸다.
5. 민감·결격 문항은 `해당 있음 / 없음 확인 / 모름`을 모두 제공하고 기본값을 두지 않는다.
6. 저장 API는 evidence source를 `self_declared`로 기록하고, 성공 후 scoped match refresh를 수행한 뒤 영향받은 공고 수 변화를 반환한다.
7. 한 번에 최대 5개를 노출하되 기본 UI는 가장 점수가 높은 한 장씩 진행한다.

**주요 수정 지점**

- `packages/core/src/matching/question-planner.ts`
- 신규 `packages/core/src/questions/definitions.ts`
- `packages/core/src/company/update-profile-field.ts`
- `packages/contracts/src/dto.ts`
- `apps/web/src/features/dashboard/ProgressiveQuestionCard.tsx`
- web/app `next-question`, `profile/field` API와 repository 적재 계약

**게이트**

- `모름`을 저장한 질문은 정책적 TTL 또는 관련 profile/evidence 변경 전까지 반복 노출되지 않는다.
- API 확정값을 self-declared 응답이 자동으로 덮어쓰지 않는다.
- 질문 저장 전후의 match trace를 비교해 실제 unknown이 줄었는지 테스트한다.

### Phase G — 표본 측정과 ROI 리포트 (STEP 3)

1. 입력 CSV/JSON의 사업자번호를 서버 내 HMAC으로 익명 표본 ID로 변환한다. 리포트에는 원문·상호·대표자명을 쓰지 않는다.
2. 소스 호출마다 duration, cache hit, 결과 분류, 추정 비용을 수집하되 raw payload는 수집하지 않는다.
3. 코호트별로 Phase A의 세 커버리지, 소스별 응답률, 필드별 정확도·충돌률, 질문 수, `질문 1개당 해소된 unknown`을 계산한다.
4. 정답 대조가 없는 필드는 accuracy 분모에 넣지 않고 `unverified`로 따로 집계한다.
5. 결과는 기계 판독 JSON과 사람용 Markdown 둘 다로 만들고, 문서의 go/no-go 기준을 자동 표시한다.

**게이트**

- fixture로 익명화·분모 제외·비용·latency 집계를 먼저 검증한다.
- 최소 30개 실표본과 정답 대조 3~5개가 없으면 `measurement harness complete / sample pending`으로 종료한다.
- 퍼지 소스 precision이 95% 미만이면 자동 확정 경로로 승격하지 않는다.

### Phase H — 프로덕션 승격 (STEP 5)

Phase A~G를 모두 기다려 일괄 승격하지 않는다. 각 소스는 다음 조건을 독립적으로 통과한 뒤 `serviceData.ts`에 연결한다.

- fixture·타입체크 통과
- dev 하네스의 성공·빈값·장애·캐시 상태 통과
- 실표본의 정확도와 범위 확인
- fail-open·타임아웃·호출 예산 가드 통과
- evidence history와 source priority 정책 통과
- 사용자에게 원천·기준일·자가응답 여부 노출

호출 순서는 `무료 캐시 → 무료 공식 라이브 → 동의 필요 소스 → 실제 criterion이 요구하는 조건부 유료 소스`로 고정한다. 신규 소스 장애는 해당 evidence만 없는 상태로 남기고 매칭 전체를 실패시키지 않는다.

## 3. 권장 커밋 단위

현재 작업 트리가 크게 변경된 상태이므로 다음처럼 작은 종결 단위로 나눈다.

1. `autofill: coverage source taxonomy and metrics`
2. `datago: classify connector outcomes and masked smoke report`
3. `kipris: add exact applicant and holdings connector`
4. `startup-confirmation: add exact active-certificate connector`
5. `registry: separate exact evidence from fuzzy candidates`
6. `questions: persist unknown and render structured conditional inputs`
7. `autofill: add anonymized cohort measurement report`
8. `service-data: promote verified connectors with fail-open policy`

각 커밋은 해당 기능의 fixture 테스트와 문서 체크리스트 갱신을 같이 포함한다. 사용자의 다른 미커밋 변경은 포함하지 않는다.

## 4. 검증 매트릭스

| 구간 | 필수 검증 | 라이브/브라우저 조건 |
|---|---|---|
| Phase A | coverage 순수 함수 테스트, CODEF 원천 분류 테스트, web 타입체크 | 불필요 |
| Phase B | NTS/KCOMWEL/FSC fixture, skip/fail 분기, 마스킹 검증 | 키가 있을 때 CLI 라이브 스모크 |
| Phase C | 소스별 parser·상태·캐시·schema drift fixture | 키+권한+실표본이 있을 때 `/dev/service-data` |
| Phase D | exact/fuzzy matcher, present-only, 중대재해 기간 테스트 | 실행 중인 web 서버가 있을 때 후보 확인 UI |
| Phase E | DART index/company/employee/finance fixture, not-covered 분기 | 키가 있을 때 라이브 표본 |
| Phase F | question planner, answer normalization, unknown persistence, scoped refresh | 사용자가 서버를 띄운 후 대시보드 시나리오 |
| Phase G | 익명화, 집계 분모, go/no-go 판정 fixture | 사용 권한 있는 30+ 표본 필요 |
| Phase H | 소스별 fail-open, TTL, priority, payload 비노출 회귀 | 승격 대상별 실사업자 E2E |

기본 정적 검증은 다음 순서로 수행한다.

```bash
pnpm --filter @cunote/core typecheck
pnpm --filter @cunote/web typecheck
pnpm verify:company-enrichment
pnpm verify:service-data
```

변경 범위에 맞는 단위 테스트를 먼저 실행하고, 위 회귀 묶음은 각 Phase 종료 시 실행한다. 전체 `pnpm test`는 변경 범위가 넓거나 프로덕션 승격 전에 수행한다.

## 5. 사용자 입력이 필요한 지점

Codex는 다음 순서로 외부 입력 없이 계속 진행하고, 해당 지점에서만 라이브 gate를 보류한다.

- KIPRIS·창업기업확인서·DART 키 미준비: fixture·커넥터·dev 배선까지 구현하고 live 검증만 보류
- 조달청 exact CSV 미준비: fuzzy 안전장치·UI를 구현하고 exact import 검증만 보류
- 개발 서버 미실행: 정적·fixture·CLI 검증까지 완료하고 브라우저 검수만 보류
- 실표본 미제공: 측정 하네스를 완성하고 ROI 실측만 보류
- 민감 질문 필수 여부·구간 문구 미승인: 스키마·저장·조건부 노출을 구현하되 기본값 없는 선택 질문으로 유지

따라서 바로 시작할 첫 구현 단위는 **Phase A — STEP 0 측정 계약 고정**이다. 이 단계가 완료되기 전에는 신규 API 값을 커버리지에 포함하지 않는다.
