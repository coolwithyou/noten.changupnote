## §0. 한 장 요약

**무엇인가**: 매칭 화면의 `conditional`(확인 필요) 상태를 대화로 해소하는 LLM 에이전트. 직무는 세 가지 — ①**수사관**: 결측 프로필 사실을 대화로 확보해 규칙 엔진에 공급, ②**통역(요건)**: 구조화되지 못한 공고 요건(`other`축 text_only)을 격리된 해석 호출로 읽어 질문과 draft candidate를 만들고, 사용자 확인을 **검수 수요 신호**로 전환, ③**통역(판정)**: 규칙 엔진의 rule_trace를 자연어로 설명.

**무엇이 아닌가**: LLM이 자격(hard eligibility)을 판정하는 구조가 아니다(D2, 재론 없음). 판정 변화는 언제나 "사용자가 확인한 새 사실을 규칙 엔진이 재평가한 결과"이며, LLM 출력이 판정·tier를 직접 바꾸는 경로는 존재하지 않는다. 또한 사용자 확인은 **검수 라벨이 아니다** — 라벨 확정자는 D5의 창업자 reviewer뿐이고, 사용자 산출물은 draft candidate와 검수 우선순위 신호까지다.

**왜 지금 설계해 두는가**: 현황평가가 확정한 병목은 엔진이 아니라 연료다(사람 검수 정답 0건, 즉시판정 가능률 ~40%, recommendable 0.61%). 이 에이전트는 그 연료 두 가지 — 회사 사실(프로필 결측)과 검수 처리량(공고 요건) — 의 생산을 사용자 대화에 연결한다. 18장 지식루프의 수요주도 유입과 같은 구조를 매칭 요건으로 확장하는 것이다.

## §1. 결정 프레임

### 1.1 판정은 왜 끝까지 규칙인가 (D2 재확인)

"LLM에게 공고+프로필을 주고 판정시키자"는 제안은 마스터 실행 문서 D2가 닫은 결정이고, 이 설계는 그 결정을 전제로 한다. 근거 네 가지를 이 문서의 전제로 다시 기록한다:

1. **비용의 차수가 다르다.** 공고 요건 추출은 공고당 1회로 상각된다(실측 ≈$4.6/1,477건, 2026-07-13 스냅샷 — 현황평가). LLM 판정은 O(회사×공고×갱신)이다. `match_state`는 현재 147,139행이고, 프로필 변경·공고 개정마다 무효화된다. 판정을 LLM화하면 비결정론적이라 버전 관리가 안 되는 판정 캐시를 재발명하게 된다.
2. **rule_trace가 곧 제품이다.** 공고 상세의 자격 매칭 아코디언(`apps/web/src/features/grant-overview/EligibilityMatchAccordion.tsx`)은 축별 판정과 근거(회사값이 있으면 회사값, 없으면 공고 원문 span)를 보여준다. "왜"를 추적 가능한 근거로 답하는 것이 신뢰 표면이며, LLM 판정은 이를 "모델이 그렇게 판단했다"로 대체한다.
3. **측정이 무너진다.** 골든 라벨 0건인 현재, LLM 판정으로 전환하면 지금 엔진보다 나은지조차 측정할 수 없다. CI의 매칭 회귀 게이트(`827a07a`)도 무력화된다.
4. **토큰이 자산을 안 남긴다.** 해자는 ①검수된 정답 데이터 ②소싱 배선 ③선정 결과 데이터다(마스터 실행 문서 §2). LLM 판정은 매 조회마다 토큰을 태우고 아무것도 축적하지 않는다. 이 설계는 반대로, 태운 토큰이 ①번 자산의 생산 속도를 높이는 위치에만 LLM을 둔다.

도메인 프레임: **자격(eligibility)은 규칙이 명시적으로 존재하는 rules-as-code 도메인**이라 결정론이 항상 우위다. LLM이 우위인 곳은 규칙이 없는 영역 — 사실 확보 대화, 자유서술 해석, 설명 — 이고, 이 설계는 그 영역만 담당한다.

### 1.2 D4와의 관계 — 새 자리를 만들지 않는다

D4는 LLM의 자리를 3곳으로 제한한다: ①공고 추출(현행) ②골든셋 사전 라벨 ③conditional 해소 보조(후순위). 이 설계의 세 직무는 전부 그 안에 있다:

| 이 설계의 직무 | D4의 자리 |
|---|---|
| ①슬롯필링(프로필 사실 확보 대화) | ③ conditional 해소 보조 |
| ②`other`축 text_only 해석·질문 생성·검수 수요 신호 수집 | ③ (해소 보조) + ②의 입력 보강(draft 사전 생성·검수 큐 가속) |
| ③rule_trace 자연어 설명 | ③ "판정을 바꾸지 않는 참고용 레이어" |

초판에 있던 "recommendable 버킷 내 LLM 정렬 보조"는 리뷰 지적(노출 순위는 그 자체로 제품 판정 표면 — D4 범위 밖)을 수용해 **삭제**했다. 정렬이 필요해지면 D4 개정 + 결정론 fallback + 노출 편향 지표를 갖춘 별도 결정으로 승인받는다(§6).

## §2. 직무별 설계

### 2.1 직무① — 슬롯필링: apply-experience-v2 §7.5의 계승

결측 프로필 축을 묻고 답을 저장하는 파이프라인은 이미 완성돼 있고, 매칭 채팅의 tool 계약도 apply-experience-v2 §7.5에 설계 기준선이 있다(`update_profile_field` tool — core 검증 경유, tool 호출은 사용자 발화가 있는 턴만, 재평가는 스트림 종료 후 비동기). 이 설계는 그 기준선을 그대로 계승하며 다음을 구체화한다:

- **질문 대상은 운영 19축의 structured unknown뿐이다.** `question-planner.ts`의 `isProfileResolvableCriterion`이 이미 이렇게 동작한다 — `text_only`·`needs_review`·`other`·`premises`·`export_performance`는 프로필 질문 대상에서 제외된다. 에이전트도 같은 게이트를 소비하며, 예약 2축은 G0B/G6 승인 전까지 질문·저장·해소율 분모에서 모두 제외한다(`update-profile-field.ts`가 두 축의 입력을 이미 거부한다).
- **질문 선택**: `question-planner.ts`의 정보가치 순서 + §7.5의 GATE 패턴(`review_gate.reasons` 상위 항목 주입, 배치당 1~2개, 선택지 우선)을 그대로 따른다. 에이전트가 질문 우선순위를 재발명하지 않는다.
- **파싱 스키마는 신규 작업이다.** `QuestionDefinition`(`questions/definitions.ts`)은 prompt/inputType 등 **UI 메타데이터이지 파싱 타깃이 아니다**. dimension별 값 변환은 현재 클라이언트 헬퍼(`features/profile-questions/questionAnswer.ts`의 `parseQuestionValue`·결격/수치그룹/수혜이력 빌더)에 분산돼 있다. 필요한 것: **definitionId별 discriminated answer schema + 서버 canonicalizer**. LLM은 발화를 이 schema로 파싱하고, 확인 카드는 LLM 문구가 아니라 **canonicalize된 값에서 렌더**하며, 확정 시 동일 canonical 값이 `updateCompanyProfileField`로 전달된다.
- **확인 후 저장 원칙**: LLM이 파싱한 값은 곧바로 저장하지 않는다. 확인 카드("연매출 5억 원으로 기록할게요 — 맞나요?")로 사용자가 확정한 값만 저장한다(P5-3의 E2E 수용 기준과 동일한 흐름). provider는 클라이언트 입력으로 열지 않는다 — 현재 `/api/web/profile/field`는 `cunote_profile_question`으로 고정돼 있으므로, 매칭 채팅 전용의 인증된 서버 액션이 `cunote_matching_chat`을 지정한다.
- **"모름" 처리**: 기존 `question_answer_state`의 TTL 억제(30일)를 그대로 따른다.

판정 변화는 이 경로에서만 발생하며, 그것은 자가신고 사실에 대한 규칙 엔진의 재평가다 — ProgressiveQuestionCard의 기존 정책과 동일하고, LLM은 통역만 했다.

### 2.2 직무② — `other`축 text_only 해석: 신뢰 경계를 분리한 2-call 구조

대상: 활성 공고 604건의 `other`축 criterion 789개 — **전부 text_only이며** 그중 589개 공고에서 hard 조건으로 쓰인다(`docs/research/2026-07-13-service-data-매칭입력-필드-문제인식.md`). 문제인식 문서의 결론대로 이는 "사용자 입력으로 풀 문제가 아니라 공고 원문 추출·검수 품질 문제"다. 이 직무는 그 검수의 **처리량과 우선순위**를 사용자 대화로 끌어올리는 장치이지, 사용자 입력으로 판정을 여는 장치가 아니다.

#### 신뢰 경계 (리뷰 BLOCKER 1 반영 — 이 절이 이 문서의 핵심 설계 결정)

apply-experience-v2 P9 원칙은 matching 컨텍스트에 외부 문서(공고 markdown)를 그라운딩에 넣지 않는다고 확정했다(§7.5). "문서 내 명령 무시" 프롬프트는 방어층일 뿐 신뢰 경계가 아니므로, **공고 원문을 읽는 능력과 프로필을 쓰는 능력을 같은 LLM 호출에 두지 않는다**:

1. **문서 해석 호출 (격리)**: 공고 `rawText/sourceSpan` 접근 가능. **tool 없음, 프로필 쓰기 없음, 대화 이력 없음.** 산출물은 draft criterion 후보와 질문 초안뿐. 운영 추출 파이프라인과 동일한 서버 검증을 전부 통과해야 한다 — schema(`buildGrantCriteriaToolSchema`)만이 아니라 normalize → evidence gate(근거 span 실재성) → canonicalize → contract assertion 전체(kstartup 경로가 현재 이렇게 동작한다). 검증 실패 draft는 폐기하고 검수 큐 부스트 대상에서도 제외한다. candidate에는 source revision/hash, evidence span, parser version, **stable criterion signature**를 필수로 묶는다. 이 호출은 공고당 결과가 캐시·상각된다(대화마다 재해석하지 않는다).
2. **대화 호출 (쓰기 경계)**: 공고 원문을 **받지 않는다.** 보는 것은 프로필, `review_gate.reasons`, 그리고 서버가 검증한 typed candidate(ID 참조)와 현재 사용자 발화뿐 — P9의 기존 경계 그대로다. 저장은 사용자의 명시 확인(confirm action/nonce) 이후 **별도 서버 액션**이 수행한다.

이 구조는 P9의 "외부 문서 미주입" 원칙을 대화 호출에서 유지하면서, 원문 해석을 쓰기 권한이 없는 격리 호출로 옮긴 것이다. **apply-experience-v2 P9/§7.5에 이 확장(격리된 문서 해석 호출의 존재)을 먼저 등재해야 착수할 수 있다**(§5 선행 조건).

#### 사용자 응답의 이중 경로

**(a) 회사 사실이 운영 19축에 사상되는 경우** ("이노비즈 인증 있어요" → certification, "정부 R&D 수혜 이력 있어요" → prior_award): 직무①의 확인 카드 경로로 self_declared 저장, 판정 반영. 기존 정책 그대로.

**(b) 축에 사상되지 않는 요건 해석의 확인** ("이 공고의 '지역특화업종 우대'는 내 업종이 해당하는 것 같아요"): **판정 반영 금지.** 처리는 두 갈래:
- **검수 수요 신호**: 검증 통과 draft criterion에 대한 사용자 확인 이벤트를 창업자 검수 큐(D5·D6)의 우선순위 피처로 기록한다. draft가 이미 만들어져 있으므로 검수는 확정/수정만 하면 된다.
- **본인 한정 참고 오버레이**: 확인한 사용자에게만 "검수 전 해석 — 판정 미반영" 배지를 노출한다. 최소 레코드 키 `(user_id, company_id, grant_id, source_revision, criterion_signature)`, 상태 `draft/confirmed/expired`, TTL, 공고 개정 시 revision 불일치로 자동 무효화. UI에는 rule_trace와 **분리된 별도 prop**(예: `unreviewedOverlay`)으로 전달해 판정 근거처럼 보이지 않게 한다(`EligibilityMatchAccordion`은 현재 satisfied/needsCheck 두 그룹만 받는다 — 신규 슬롯 필요). 전역 발행은 이 오버레이가 아니라 V3 annotation으로 변환된 뒤 기존 reviewer 경로(`reviewed-publication.ts`)만 사용한다.

#### 순환성·어뷰징 방어

사용자는 자기에게 유리한 해석을 확인할 인센티브가 있다. 방어는 두 겹:
- **전역 오염 차단**: 사용자 확인을 needs_review 해제에 직결 금지 — Gate 1의 "AI 라벨 무검수 golden 승격 금지"와 같은 논리. 확인은 검수의 대체가 아니라 **감사 가능한 약한 demand signal**이다.
- **큐 어뷰징 차단**: (source revision + criterion signature)당 사용자 1회 dedupe, 회사 단위 cap, rate limit, 시간 감쇠, 부스트 상한, 그리고 unboosted/오래된 항목의 처리 quota(부스트가 순회 큐를 독점하지 못하게).

### 2.3 직무③ — 설명: 비판정 레이어

rule_trace(축별 pass/fail/unknown + 근거)를 자연어로 풀어 답한다. 근거는 기존 grounded chat과 동일하게 인용 기반 — 인용 없는 서술은 "일반 안내"로 시각 구분(기존 `ChatPanel` 규약). 버킷·tier·점수·노출 순서에는 관여하지 않는다(§1.2 — 정렬은 이 설계에서 제외).

## §3. 아키텍처 배선 — 기존 인프라의 예약석과 실제 신규 작업량

| 구성 요소 | 재사용 대상 | 신규 작업 (리뷰 반영으로 초판보다 구체화) |
|---|---|---|

## §4. 안전 불변식

HANDOFF(service-data 트랙) §4.4의 불변식을 계승하고 확장한다. 1~2는 계승, 3~7은 이 설계가 추가하는 착수 전 필수 조건이다:

1. 값이 부족·불명확하면 `unknown` 유지. 해소를 위해 pass로 바꾸지 않는다.
2. **AI는 hard eligibility를 직접 판정하지 않는다.** 판정 변화의 유일한 경로는 "사용자 확정 사실 → 규칙 엔진 재평가".
3. **grounding은 userId 스코프 필수.** 현재 `buildGrantGrounding`은 company ID만으로 전 프로필 행을 읽는다 — 한 회사에 복수 사용자가 연결되면 결격·신용·세금 자가신고가 다른 구성원의 컨텍스트에 들어갈 수 있다. repository의 shared+same-user 해석(`repositories/drizzle.ts`)을 grounding에도 적용한 뒤에만 매칭 채팅을 연다. 2인 동일회사 격리 테스트를 수용 기준에 포함한다.
4. **민감 답변은 원문 영속화 전에 처리한다.** 현재 chat route는 LLM 처리 전에 사용자 원문을 `chat_messages`에 저장하고 이 테이블에는 expiry/redaction이 없다. 결격 3축 등 민감 dimension 답변은 서버에서 마스킹 후 저장하거나 구조화 확인 이벤트로만 기록한다. 보존기간·삭제/내보내기·관리자 접근·로그 redaction은 착수 전 확정(§9).
5. **외부 원문 접근과 프로필 쓰기 권한을 같은 LLM 호출에 두지 않는다**(§2.2 신뢰 경계).
6. **LLM 파싱 값은 사용자 확정 전 저장 금지**(§2.1 확인 카드). 사용자의 요건 해석 확인은 전역 발행 금지 — 검수(D5) 통과 후에만(§2.2).
7. **D9 presentation gate.** `buildReviewGate`는 profile evidence 출처를 입력받지 않으므로, 채팅이 결격 답변을 self_declared로 채우면 D9("결격 3축 자가신고 동안 '지원 가능성 높음' 문구 금지")를 표현 계층에서 강제할 수 없다. resolved profile evidence를 소비하는 별도 presentation gate를 두어, 결격 3축 중 하나라도 self_declared이면 tier와 무관하게 "지원 가능/높음" 계열 문구를 차단한다. 참고 오버레이 배지는 "검수 전 · 판정 미반영" 문구 고정 + 충족 영역 밖 배치.

## §5. 선행 조건·타이밍

이 설계의 가치는 아래가 갖춰진 뒤에 발생한다. 역순으로 착수하면 안 된다:

1. **G1 트랙(service-data)의 CompanyProfile 필드 SSOT + typed update 계약(G2A/B)** — 에이전트의 쓰기 경로와 answer schema(§2.1)가 소비할 계약. 병렬 세션에서 진행 중.
2. **WS-B 공고 검수 게이트 가동** — 검수 수요 신호가 흘러들 큐가 실제로 돌아야 §2.2가 의미를 가진다.
3. **recommendable 해금**(공고 검수 → reviewed 발행) — conditional이 화면에 의미 있게 노출된 뒤에야 성립한다.
4. **apply-experience-v2 Phase 5 착수 게이트**(채팅 v1 안정화, trust gate 실데이터, `matching_chat` featureCode 등재) + **P9/§7.5 개정**: 격리된 문서 해석 호출의 존재와 그 경계를 P9 원칙에 등재. 이 개정 전에는 직무②를 열 수 없다(직무①·③만으로 1차 개방 가능).

## §6. 비목표

- LLM 직접 판정, LLM 점수·정렬로 tier/버킷/노출 순서 변경 (D2 + 리뷰 MAJOR 11 — recommendable 버킷 내 정렬 포함 전면 제외)

- 신규 평가 인프라 (D8) — §8의 telemetry는 평가 하네스가 아니라 운영 계측 확장이다(마스터 등재 시 이 구분을 명시)

## §7. 비용 모델

- **차수**: 토큰 소모는 O(사용자가 시작한 대화 세션) + O(공고당 1회 문서 해석·캐시)다. LLM 판정의 O(회사×공고×갱신)과 달리 사용자가 능동적으로 시작한 상호작용에만 변동비가 발생한다.
- **턴별 분해식(고정 봉투 대신)**: 입력 = grounding 문서(현행 24k cap은 markdown에만 적용됨을 유의) + 동적 컨텍스트(criteria candidate·rule_trace·프로필 — **신규 hard cap 필요**) + 누적 대화 이력(현행 세션은 매 턴 전체 이력 재전송) + 시스템. 출력·cache write/read는 별도 합산. 일일 예산(`CHAT_DAILY_TOKEN_BUDGET` 300,000)은 cache read/write까지 합산하는 내부 통제치이지 provider 원가가 아니다 — 둘을 혼동하지 않는다.
- **산정 방식**: 고정 "5~8턴" 가정을 쓰지 않는다. 개방 후 `usage_events`의 세션당 토큰 p50/p95와 동시 요청 overshoot를 측정해 budget·estimate 기본값을 정한다. matching bundle(동적 컨텍스트) 전체에 hard cap을 둔다.
- **과금 경로**: 처음부터 `withCreditMetering`으로 배선하되(stream-aware 계약 — §3), 초기 요율은 `feature_free`로 시드해 무과금 개방 + 원가 수집. 유료 전환은 요율 행 "변경"이 아니라 **append-only 규약**(free 행 `effectiveUntil` 마감 + 새 `model_token` 행 insert — 크레딧 설계 문서의 요율 이력 규칙)을 따르고, `matching_chat`을 seed·표시 사전·estimate 기본값에 함께 등록한다. 일일 토큰 예산은 남용 차단기로 병행 유지.

## §8. 성공 지표와 측정 계약

리뷰 확인 결과 **기존 테이블만으로는 아래 KPI를 계산할 수 없다** — `profile_question_events`는 provider/user/grant 없이 집계값만 저장하고, `match_events` enum은 노출·클릭 계열뿐이다. 따라서 착수 시 **원문을 저장하지 않는 최소 telemetry 계약**을 함께 정의한다: origin/provider, chat session ID, dimension, 판정 before/after counts, 검수 큐 이벤트 timestamp. (D8 관계는 §6 참조.)

| 지표 | 측정원 | 기준선 |
|---|---|---|
| 세션당 unknown 해소 축 수 | 신규 telemetry(origin 구분) | 폼 플래너 단일 관측 n=1, 14/19≈0.737 — **상시 기준선 없음**, 개방 후 자체 기준선 수립 |
| draft candidate 확인 유입 | 확인 이벤트 건수/주 | 0 (현재 수단 없음) |
| 검수 가속 | 부스트 공고의 검수 완료 리드타임 (큐 이벤트 timestamp) | 순회 큐 기본 순서 대비 |
| 판정 전이 | 세션 후 conditional→eligible/ineligible 건수 (before/after counts) | — |
| 단위 경제 | 해소 1건당 크레딧 (`usage_events`) | — |

## §9. 열린 결정 사항 (착수 시 확정)

1. **민감 답변 처리의 세부**: §4-4의 원칙(영속화 전 마스킹 또는 구조화 이벤트만) 아래에서 — 마스킹 범위(결격 3축만 vs 재무 포함), `chat_messages` 보존기간, 삭제·내보내기 절차, 관리자 접근 정책.
2. **참고 오버레이 배지의 문구·위치 확정**: §4-7의 제약("검수 전 · 판정 미반영" 고정, 충족 영역 밖) 아래에서 구체 카피·컴포넌트 위치.
3. **과금 전환 시점**: feature_free → model_token 전환 기준(세션 수·p50/p95 원가 실측).
4. **오버레이·확인 이벤트의 물리 저장소**: 계약(§2.2 레코드 키·상태·TTL)은 확정 — 테이블 신설 vs 기존 확장은 G2A/B typed update 계약 확정 후.
5. **매칭 채팅의 스코프 단위**: 단일 공고 진입(`/grants/[grantId]`) vs 후보 목록 진입(`/matches` CTA — apply P5-2) 중 1차 개방 범위.
