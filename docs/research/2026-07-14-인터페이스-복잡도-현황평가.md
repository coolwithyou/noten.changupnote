# 인터페이스 복잡도 현황 평가 (2026-07-14)

> **목적**: "시스템 복잡도와 무관하게 인터페이스는 극단적으로 단순해야 하며, 사용자가 원하는 것만 즉시 보여주는 것이 비즈니스의 핵심"이라는 명제를 기준으로 현재 4개 화면군(랜딩·매칭 입력·매칭 결과·지원서 가이드)을 평가한다.

> **범위**: 평가만. 구현 없음.

## 0. 요약 판정

|---|---|---|

| 지원서 워크스페이스 | 🔴 심각 | 필드 1개당 정보 9종 + 뱃지 4 + 버튼 6이 필드 수만큼 반복. 필드 20개면 결정 지점 130+ |
| 신청 관리(applications) | 🔴 심각 | 8단계를 통계카드·토글·레인으로 3중 반복 노출 |
| 공고 요약(grant overview) | 🟢 양호 | 이미 미니멀 재설계 완료(5섹션 상한이 코드 주석에 계약으로 명시). **전사 기준으로 삼을 만한 모범** |

**핵심 결론**: 복잡도의 원인은 "화면이 많아서"가 아니다. 세 가지 구조적 패턴이 원인이다.

1. **시스템의 정직성이 전부 push로 노출된다** — 룰 트레이스, 근거, 점수, 상태 뱃지, 면책문을 기본 화면에 전시한다. 정직성 자체는 자산이지만, 결론보다 근거가 먼저 보인다.
2. **같은 질문에 답하는 표면이 중복 존재한다** — "어떤 공고를 노릴까"에 답하는 UI가 4개(`/matches` 카드, OpportunityMap, Roadmap, ActionQueue), 상태 어휘가 각각 다르다.
3. **다음 행동이 하나로 수렴하지 않는다** — 매 화면에서 사용자가 할 수 있는 일이 수십 개 병렬로 열려 있고, "지금 이것 하나를 하세요"가 없다.

## 1. 평가 프레임

### 1.1 사용자의 질문은 3개뿐

이 서비스의 사용자(소상공인·초기 창업자, 공고 문법에 익숙하지 않음)가 어느 화면에서든 갖는 질문은 셋으로 수렴한다.

- **Q1. 나 받을 수 있어?** (판정)
- **Q2. 그래서 지금 뭘 하면 돼?** (다음 행동)
- **Q3. 언제까지?** (마감)

화면의 모든 정보 요소는 ①이 3개 질문 중 하나에 즉답하거나 ②요청 시 공개(pull)로 강등되거나 ③삭제 대상이다. 이 프레임으로 각 화면을 채점한다.

### 1.2 복잡도 측정 지표

- **정보 클러스터 수**: 첫 뷰포트에 동시 노출되는 독립 정보 덩어리
- **결정 지점 수**: 화면에서 사용자가 선택 가능한 분기(버튼·토글·펼침)
- **상태 어휘 수**: 사용자에게 노출되는 상태 라벨의 가짓수
- **병렬 표면 수**: 같은 사용자 질문에 답하는 별개 UI의 수

## 2. 화면별 실태

### 2.1 랜딩 (`apps/web/src/app/(marketing)/page.tsx`)

- **구성**: Hero → SocialProof → HowItWorks → Features → TrustStats → FinalCta → Faq → Footer, **8개 섹션 · 14개 파일 · 약 1,253줄**
- **첫 뷰포트(Hero)**: 뱃지 1 + 3줄 헤드라인 + 서브카피 + 입력 폼 + 신뢰 칩 3개 = **6~7개 정보 클러스터** (`landing-hero.tsx:11-58`)
- **CTA 12종**. 그중 실제 전환 목표는 단 하나(사업자번호 제출)이고, Hero와 FinalCta의 폼은 동일 컨트롤러 공유
- **밀도 과잉 지점**:
  - SocialProof: 공고 카드 3장 × 카드당 4~5 데이터 = **최대 15개 데이터 포인트** (`marketing-sections.tsx:80-137`)
  - Features: 카드 3장 × (아이콘+태그+제목+본문+불릿 2) = 아이콘 9개 + 불릿 6개 동시 노출 (`marketing-sections.tsx:167-210`)
  - TrustStats의 `activeCount`는 Hero 뱃지와 **중복 노출** (`marketing-sections.tsx:213-235`)
- **평가**: 매칭 입력 자체(1필드, 30초)는 이미 가치제안 그대로다. 문제는 "사업자번호만 넣으면 된다"고 말하는 페이지가 그 말을 8섹션에 걸쳐 반복 증명하려 한다는 것. 단순함을 **주장**하는 카피와 단순함을 **체감**시키는 구조가 어긋난다.

### 2.2 매칭 입력 플로우

- 입력 필드 **1개**(사업자번호), 체크섬 검증 → 로딩 → 확인 다이얼로그(뱃지 3 + 버튼 2) → `/matches?biz=` (`use-biz-lookup.ts:105-142`, `biz-lookup-dialog.tsx:61-107`)
- **평가**: 🟢 이 구간은 토스식 "한 화면 한 가지 일"에 이미 부합한다. 유일한 보완점은 확인 다이얼로그가 모달이라는 점 정도이며 우선순위 낮음.

### 2.3 매칭 결과 (`/matches`, `features/match-results/`)

결과 페이지는 ResultsHero → ProfileSection → Programs 3블록. 문제는 각 블록 내부다.

**(a) ResultsHero** — 헤드라인 + **StatTile 3개**(총 지원 가능액 / 마감 임박 / 최고 조건 확인도) + 컨텍스트 노트 (`ResultsHero.tsx:45-83`). Q1에 답하는 것은 헤드라인 한 문장이면 충분한데 지표 3개가 병렬 경쟁한다. "총 지원 가능액" 합산치는 실제 수령 가능액이 아니어서 오해 위험도 있다.

**(b) ProfileSection** — 결과 페이지 한가운데에 **입력 카드 11개 그리드**(상호·소재지·업력·기업규모·업종·영업상태·사업자 유형·대표자 연령·인증·상시근로자·연 매출) + 충족도 Progress + 질문 카드 (`ProfileSection.tsx:135-197`, `serviceData.ts:1273-1352`). "사업자번호 하나로"라고 약속한 직후, 결과를 보기도 전에 11개의 입력 표면이 펼쳐진다. **가치 전달 앞에 노동 요구가 배치된 역순 구조**.

**(c) Programs 카드** — 접힘 상태에서 이미 뱃지 4종(적격/마감/작성지원/관련성) + 제목 + 기관·금액 + fitScore Progress + 조건 요약 (`Programs.tsx:135-295`). 펼치면 룰 트레이스 N행(행당 최대 5정보) + 추천 근거 3줄 + 면책문 + 작성지원 안내 + **피드백 버튼 4개**(저장/제외/오류/신청함) + CTA. **첫 카드는 기본 펼침**이라 진입 즉시 최고 밀도 화면이 열린다. 카드 수 제한·페이지네이션 **없음** — 전량 세로 나열 (`Programs.tsx:41-42`).

**(d) 매직 루프의 매몰** — 이 제품에서 가장 가치 있는 상호작용은 실측으로 검증돼 있다: **질문 1개 답변 → 확인 필요 19건 중 14건 확정** (마스터 실행 문서 WS-D 근거 실측). 그런데 그 질문 카드는 ProfileSection 내부, 11개 필드 그리드와 충족도 바 사이에 끼어 있다. 현황평가 문서(§3.2)도 `needs_profile_input`(답하면 확정)과 `needs_core_review`(대기)가 한 섹션에 뭉뚱그려져(`logic.ts:543-545`) "확정까지의 경로"가 매몰됐다고 이미 지적했다. **제품의 심장이 화면 위계상 3순위다.**

### 2.4 대시보드 (`features/opportunity-map/`, `features/dashboard/`)

- OpportunityMap = `/matches`와 **별개로 구현된 두 번째 매칭 결과 UI**. 상태 필터 7종 + 정렬 4종 + 버킷 4개 × 카드 4장 + 부적격 disclosure + 카드마다 피드백 4버튼 (`OpportunityMap.tsx:34-201`)
- 같은 대시보드에 RoadmapStrip(시간×조건 4구간)과 ActionQueuePanel("이번 주 먼저 할 일", 액션 5종 어휘)이 추가로 얹힘 (`DashboardView.tsx:61-71`)
- **평가**: "어떤 공고를 노릴까"라는 하나의 질문에 **4개의 표면**(Programs 카드 / OpportunityMap 버킷 / Roadmap 4구간 / ActionQueue)이 각기 다른 분류 어휘로 답한다. 버킷 어휘만 봐도 — OpportunityMap: `지금 받을 수 있어요/확인이 필요해요/준비하면 열려요/곧 받을 수 있어요`, Roadmap: `지금 적격/곧 적격/준비하면 적격/조건부 확인` — 유사하지만 불일치. 사용자는 같은 공고를 다른 이름의 서랍에서 반복해서 만난다.

### 2.5 지원서 가이드 (grants/[grantId] → workspace)

**(a) 공고 요약 페이지** — 🟢 **모범 사례**. 5섹션 상한, 아코디언 3개 기본 닫힘, 주 CTA 1개. 심지어 코드 주석에 "구조는 §4.2 그대로 5개, 입력 필드 금지"가 계약으로 박혀 있다 (`GrantOverviewView.tsx` 헤더 주석). **이 화면이 증명하는 것: 이 팀은 이미 단순한 화면을 만들 수 있고, 계약을 코드에 박는 방법도 안다. 나머지 화면에 같은 규율이 없을 뿐이다.**

**(b) 워크스페이스 필드 패널** — 복잡도의 진앙.
- 필드 카드 1장 = 정보 요소 최대 **9종**(라벨/섹션/필수 뱃지/상태 뱃지/위치 미확인 뱃지/중복 라벨 뱃지/값/액션군/작성 팁) + 액션 버튼 최대 **6개**(반영·제안 받기·수정·건너뛰기·되돌리기·이 항목이 뭐예요?) (`FieldCard.tsx:34-267`)
- 이것이 문서 필드 수만큼 반복. **필드 20개면 고정 결정 지점 약 130개**
- 상태 뱃지 어휘 **7종**: 미입력 / 자동 입력(사업자 정보) / 자동 입력(양식 기본값) — 확인 필요 / 제안 — 확인 필요 / 확정 / 확정(수정됨) / 건너뜀

**(c) 상태 신호의 분산** — 사용자가 "지금 무슨 상태인가"를 파악해야 하는 신호 체계가 5겹이다:
| 신호 체계 | 가짓수 | 소스 |
|---|---|---|
| 사다리 뱃지 | 3 | `WorkspaceView.tsx:35-39` |
| 정직 고지(honestNotice) 문구 | 7 | `workspaceData.ts:82-337` |
| 필드 상태 뱃지 | 7 | `FieldCard.tsx:34-53` |
| 프리뷰 오버레이 색 | 5 | `PreviewCanvas.tsx:23` |
| 초안/서류 상태 라벨 | 10 | `DraftFallbackEditor.tsx:823-843` |

**(d) DraftFallbackEditor** — 900줄 단일 컴포넌트, 한 화면에 요약 지표 4 + 문서 리스트 + 추가 입력 패널 + markdown textarea + 문항별 자동채움 편집 + 섹션 재생성 + 내보내기 5종 + 품질 피드백 폼 = **10종 이상의 컨트롤이 한 화면에**.

### 2.6 신청 관리 (`/applications`)

- 8단계(서류 준비/저장됨/추천/제출 완료/선정/탈락/신청 막힘/보류)를 **통계 카드 8개 + 토글 8개 + 레인 8개로 3중 반복** (`ApplicationPipelineView.tsx:127-255`)
- 헤더에 링크 버튼 5개, 카드당 데이터 필드 10+ 와 액션 7~9개
- **평가**: 초기 사용자의 신청 건수는 한 자릿수다. 한 자릿수 데이터를 위한 8레인 보드는 관리 도구의 형태를 사용자에게 전가한 것.

## 3. 횡단 문제 5가지 (근본 원인)

**P1. 상태 어휘 폭발 — 전사 사용자 노출 상태가 30종 이상.** 매칭 쪽 3~4종(적격/조건부/미해당 + 4버킷 + 4구간) + 워크스페이스 5겹 32종. 게다가 설계 문서 간에도 "4상태"의 정의가 3가지로 갈라져 있다(마스터 §9.1 vs 실행 WS-D.1 vs service-data §7 — 설계 문서 UX 방향 조사에서 확인). **단일 상태 어휘 SSOT가 없다.**

**P2. 병렬 표면 중복.** 매칭 결과 UI 2개(`/matches` Programs vs 대시보드 OpportunityMap — 별도 구현·별도 어휘), 공고 선택 관점 4개(+Roadmap, ActionQueue). 정보량이 두 배가 아니라, **같은 정보가 다른 모양으로 두 번** 나와서 학습 비용이 두 배가 된다.

**P3. 가치 앞의 노동.** 결과 페이지 진입 즉시 11개 프로필 입력 카드 + 충족도 게이지가 판정 결과와 화면을 경쟁한다. "사업자번호 하나로"의 약속과 정면 충돌.

**P4. push된 정직성.** 룰 트레이스·근거 등급·면책문·점수·출처는 신뢰의 근간이지만, 현재는 **기본 화면에 전시**된다. 결론(Q1~Q3)이 근거 밑에 깔린다. 정직성은 삭제가 아니라 **pull(요청 시 공개)로 강등**이 정답이다.

**P5. 다음 행동의 부재.** 어느 화면에도 "지금 이것 하나"가 없다. 매칭 결과에는 질문·필드 11개·카드 N장·피드백·저장이 병렬이고, 워크스페이스에는 필드 N개 × 6버튼이 병렬이다. ActionQueue가 "이번 주 먼저 할 일"로 이 문제를 풀려 했으나, 그 자체가 5번째 표면이 되어 문제를 가중했다.

## 4. 외부 대조

### 4.1 토스 — "One Thing per One Page"

토스의 제품 원칙 중 단 한 번도 변하지 않은 가치가 Simplicity이고, 그 실행 규칙이 "한 페이지에는 한 가지 일만"이다. 여러 입력을 한 화면에 몰지 않고 단계를 나눠 질문 하나씩 요청하는 것이 이탈률을 낮춘다는 것이 핵심 논거다. 금융(계좌·송금·대출)이라는 고복잡도 시스템 위에서 이 원칙을 지켰다는 점에서 cunote와 조건이 같다.
참고: [토스의 UX 철학: One page - One thing](https://disquiet.io/@rafa/makerlog/%ED%86%A0%EC%8A%A4%EC%9D%98-ux-%EC%B2%A0%ED%95%99-one-page-one-thing), [UX 원칙 - One thing per page](https://brunch.co.kr/@chulhochoiucj0/31), [거꾸로 입력하는 가입 화면](https://toss.tech/article/toss-signup-process)

### 4.2 TurboTax — 인터뷰 패턴

세금 신고라는 "공고문급" 복잡도를 **한 번에 질문 몇 개**의 인터뷰로 풀었다. 세 가지 규칙이 cunote에 그대로 이식 가능하다: ①조건부 로직 — 해당 없는 질문은 아예 보여주지 않는다 ②인지 부하 기준 그룹핑 — 쉬운 질문만 묶고, 생각이 필요한 질문은 단독 화면 ③화면에는 현재 질문과 경쟁하는 요소를 두지 않는다. cunote의 질문 플래너(`question-planner.ts`)는 이미 "가장 많은 공고를 확정시키는 다음 질문"을 계산한다 — **백엔드는 TurboTax인데 프런트가 국세청 홈택스인 상태**다.
참고: [UX/UI analysis of TurboTax](https://uxdesign.cc/https-medium-com-ux-ui-analysis-of-turbotax-171c86d80a41), [How TurboTax makes a dreadful UX delightful](https://www.appcues.com/blog/how-turbotax-makes-a-dreadful-user-experience-a-delightful-one)

### 4.3 웰로비즈 — 경쟁사의 반대 방향

직접 경쟁인 [웰로비즈](https://concierge.wellobiz.com/)는 "23,000개 사업 · 일정 관리 · 민원서류 자동 발급 · 조달 정보"의 **올인원 다기능** 포지션이다. 즉 경쟁사가 기능 나열로 갈수록, cunote가 "사업자번호 → 30초 → 결론 하나"의 극단적 단순함으로 가는 것은 기능 격차가 아니라 **포지셔닝 차별화**가 된다. 사용자의 명제("인터페이스 단순함이 비즈니스의 핵심")는 경쟁 구도상으로도 성립한다.

## 5. 설계 문서와의 정합

이번 평가의 방향은 기존 설계 문서와 충돌하지 않으며, 오히려 이미 계획된 항목의 확장이다.

- **WS-D.1(실행 문서)**: `needs_profile_input` 분리 렌더 + "답변 1개로 확정되는 공고 N건" 노출 — 본 평가의 P5(매직 루프 부활)와 동일 방향. 결함 대장에도 major/unresolved로 등재됨
- **D9 문구 게이트**: 결격 3축 자가신고 동안 "지원 가능성이 높음" 금지 — 단순화 제안에서도 이 게이트는 유지해야 할 제약
- **매칭대화에이전트 설계 §6**: "새 UI 체계 발명 금지, 기존 패턴 이식으로 한정" — 본 평가의 P2(표면 통합)와 같은 철학. 단, 대화형 표면 추가는 P2 관점에서 신중해야 함(표면이 하나 더 늘어나는 것)
- **주의(동결)**: 실행 문서 §4는 지원서 작성 기능 추가 투자를 매칭 베타 게이트까지 동결. 따라서 워크스페이스 단순화는 **기능 추가가 아닌 정보 재배치**로 한정하고 실행 순서도 매칭 표면 뒤로 배치해야 함
- **문서 위계 리스크**: "4상태"의 정의가 문서 3곳에서 미묘하게 다르다(§3 P1). 인터페이스 단순화의 선행 조건으로 상태 어휘 SSOT 확정이 필요

## 6. 종합 판정

> **인터페이스가 복잡한 이유는 정보가 많아서가 아니라, 제품이 화면에서 "목록"으로 행동하기 때문이다. 시스템의 실체는 "루프"(질문 하나 → 확정 증가 → 다음 행동)인데, 화면은 그 루프를 숨기고 재고 전량을 나열한다.**

- 이미 증명된 역량: 공고 요약 페이지(5섹션 계약)와 매칭 입력(1필드)은 목표 수준의 단순함을 달성했다
- 최우선 수술 부위: **매칭 결과 페이지** — 매직 루프를 화면의 주인공으로 올리고, 11필드 그리드와 지표 3개를 강등
- 구조 수술 부위: **표면 통폐합** — 매칭 결과 UI 2개→1개, 공고 선택 관점 4개→1개
- 규율 수술 부위: **상태 어휘 30+종 → 사용자 노출 4상태 + 필드 3상태**로 고정하는 SSOT
