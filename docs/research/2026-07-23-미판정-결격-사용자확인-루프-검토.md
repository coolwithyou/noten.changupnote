## 1. 발견 사실

최신 딥분석 런의 감사 화면에서 "미판정"으로 남은 항목들(= AI 블라인드 2차 감사가 1차 검수와 불일치했거나 unsure여서 사람 큐에 잔류한 항목)을 검수한 결과:

- **대부분의 1차 판정은 실제로 정확했다.** 그러나 미판정으로 남은 항목들은 공통적으로 "미묘한" 케이스였다.
- **대표 사례 — #11 기수혜(prior_award)**: 공고 원문은 *"3) 타 정부지원사업에서 **체계적합성시험비**를 기 지원받은 경우"*를 결격사유로 규정한다. 그런데 canonical 값은 `self_kind: same_year_other_support`(같은 해 타 지원 수혜)로 저장됐다. 원문의 결격 조건은 "무엇이든 수혜"가 아니라 **특정 항목(체계적합성시험비)의 기수혜**인데, canonical 어휘가 그 특정성을 담지 못해 조건이 실제보다 넓어졌다.
  - 이 canonical 그대로 매칭 평가에 쓰이면, 체계적합성시험비가 아닌 다른 지원을 받은 기업까지 결격 처리하는 **과잉 배제(false exclusion)**가 발생한다.
  - 이는 prior_award 분해기를 기본 OFF로 잠근 기존 결정(§6-C2: "현행 exact-match evaluator는 false pass 양산")과 정확히 같은 계열의 문제가 실측으로 재확인된 것이다.
- **#12·#13류(other/text_only)**: "과제수행계획서 허위 작성", "주관기업 참여만 인정" 같은 절차·자격 조건은 canonical 손실 문제가 아니라, **애초에 소싱된 기업 데이터로는 영원히 판정 불가**한 항목이다. sourceSpan 추출 자체는 정확하다.

### 1.1 발견의 분해 — 미판정에는 서로 다른 두 축이 섞여 있다

| 축 | 질문 | 해소 주체 | 사례 |
|---|---|---|---|
| **① 추출 정확도** | canonical이 공고 원문에 충실한가? | 내부 감사(사람/AI 블라인드) | #11 — canonical이 원문보다 넓음 → "수정 필요" |
| **② 판정 가능성** | 기업 데이터로 충족 여부를 알 수 있는가? | **사용자 자가신고** | #11(수정 후에도 "체계적합성시험비 기수혜 여부"는 자가신고), #12·#13(text_only 전부) |

핵심 통찰: **#11은 ①의 결함이 ②의 필요를 드러낸 사례**다. 감사가 canonical을 고쳐도(①), "그 특정 항목을 기수혜했는가"는 어떤 공공 데이터 소스에도 없으므로 결국 사용자에게 물어야 한다(②). 그리고 ②에 해당하는지는 감사 결과와 무관하게 **추출 시점에 이미 알 수 있다**(`scope=self`, `operator=text_only` 등이 신호).

또 하나의 관찰: 블라인드 감사가 정확히 이런 미묘한 항목들에서 불일치를 낸 것은 §10 설계가 의도대로 작동한다는 증거다(미묘함 탐지기로서의 감사).

## 2. 제안 요약 (사용자 원안)

1. 자가신고가 필요한 결격 항목(미판정류)은 **"만족한다"고 전제**하고 공고를 노출하되, **"확인 필요" 섹션**에 배치한다.
2. 사용자가 해당 공고의 "확인하기"를 클릭하면 **객관식 선택 모달**을 띄운다. 객관식 문항은 **딥분석 시점에 공고마다 LLM이 미리 생성해 캐시**한다(매번 재생성 금지).
3. 사용자의 답변에 결격 해당 사항이 없을 때만 **"지원 가능"으로 재분류**한다.
4. 사용자가 선택한 답은 **기업 메타데이터로 저장**해, 추후 다른 공고의 동일 확인 항목에서 **미리채움**한다.

## 4. 구현 가능성 검토 — 구성 요소별

### 4.1 질문 사전 생성 (갭 1) — 가능, 비용 미미

- `extractor.ts`의 tool 스키마(`emit_deep_grant_analysis`)에 자가신고형 항목(prior_award `scope=self`, other `text_only` 등)당 `confirmation` 필드를 추가한다:
  ```
  confirmation: {
    prompt: string,            // 객관식 질문 문장 (사용자 어휘)
    options: [{ value, label, disqualifies: boolean }],
    answer_type: "single" | "multi",
    reusable: "company_fact" | "per_notice",   // §4.4 참조
    condition_key: string | null               // reusable일 때만
  }
  ```
- **질문은 canonical이 아니라 `sourceSpan`(원문 인용)을 앵커로 생성하도록 프롬프트에 명시**한다. #11이 보여주듯 canonical은 손실될 수 있지만 sourceSpan은 정확했다(`spanVerified` 검증 활용). 질문이 canonical 기반이면 "올해 다른 지원을 받았나요?"라는 잘못된 질문이 나가고, sourceSpan 기반이면 "체계적합성시험비를 지원받은 적 있나요?"라는 올바른 질문이 나간다.
- 비용: 기존 런(공고당 ≈$0.395)에 출력 토큰 수백 개 추가 → **공고당 +$0.01~0.02 수준**. 확인 시점의 사용자별 LLM 비용은 0(사전 생성의 목적 그대로).
- 기존 30건 런은 질문 필드가 없으므로, 스키마 확장 후 배치 재실행 또는 질문만 생성하는 경량 보강 패스 중 택일(후자가 저렴).

### 4.2 프로덕션 착지 경로 (갭 2) — 가능하나 이 제안의 최대 의존성

- 현재 딥분석 결과는 spike-out 파일에만 있고, 메인 매칭이 읽는 것은 `grant_criteria`다. 확인 루프가 사용자에게 닿으려면 **감사 확정된 criteria + 질문의 승격 경로**가 필요하다.
- 착지점은 이미 구조가 마련돼 있다: `grant_criteria`(needsReview/confidence/sourceSpan 컬럼 보유) + 질문용 신규 테이블(아래). 승격 시 `promptVer`·감사 상태를 함께 기록한다.
- **순서 제약**: 기존 방침(§10, 메인 반영은 `lab:shadow` 섀도 측정 긍정 후)이 이 제안에도 그대로 적용된다. 확인 루프는 승격 경로 위에 얹히는 기능이므로, **섀도 검증 → 승격 파이프라인이 선행 관문**이다.
- 스키마 초안:
  ```
  grant_confirmation_questions:
    id, grant_id FK, criterion_ref(차원+kind+span 해시 또는 criteria FK),
    prompt, options jsonb, answer_type, reusable, condition_key(nullable),
    prompt_ver, created_at

  company_grant_confirmations:            -- 공고 스코프 답변 (재분류의 근거)
    company_id, grant_id, question_id, answer jsonb, answered_at, user_id
  ```

### 4.3 확인 모달 + 재분류 — 낮은 난도

- UI는 `DisqualificationSheet` 패턴 재사용(shadcn Sheet/Dialog + ToggleGroup). `check_source`/`one_answer` 버킷 카드에 "확인하기" CTA를 배선.
- 재분류: 답변 저장 → 해당 (company, grant)의 match_state 재계산. 평가기에 "확인 답변으로 해소된 criterion" 입력을 추가해야 한다 — 결격 아님 답변이면 해당 exclusion을 충족 처리(→ unknown 소거 → `eligible` 가능), 결격 해당 답변이면 hardFail(→ `closed`).
- **가드레일(기존 D9 준수)**: 결격 자가신고가 미완인 동안 "지원 가능성 높음" 문구 금지. "만족 전제"는 *open 버킷 편입*이 아니라 *확인 필요 버킷 노출*을 의미해야 하며, 이는 원안이 이미 그렇게 설계돼 있다. 재분류 후에도 "본인 확인 기반" 표기를 남긴다(자가신고와 소싱 데이터의 신뢰 등급 구분 — company_profiles의 source/confidence 체계 그대로).

### 4.4 답변의 기업 메타 저장·미리채움 (갭 3) — 가능하나 설계 주의 최다

원안에 대한 **수정 제안 2가지**:

1. **모든 답변을 기업 메타로 저장하면 안 된다.** 질문은 두 부류로 나뉜다:
   - `company_fact`(재사용 가능한 기업 사실): "체계적합성시험비 기수혜 여부", "특정 사업 참여 이력" → `company_profiles`(EAV)에 저장, 미리채움 대상.
   - `per_notice`(공고 국한 선언): "과제수행계획서를 허위 작성하지 않음"(#12), "주관기업으로 참여"(#13) → 공고 스코프 답변 테이블에만 저장. 이걸 기업 메타로 승격하면 무의미하거나(선언은 공고마다 새로 해야 함) 오염이 된다.
   - 이 분류 자체를 질문 생성 시 LLM이 산출(`reusable` 필드)하고, 감사 대상에 포함시킨다.
2. **미리채움의 전제는 "동일 확인 항목"의 식별이다.** ~~표준 condition_key 사전 거버넌스~~ → **[결정 2026-07-23] 사전 거버넌스는 무리로 판단, 채택하지 않는다. 질문 생성 시 LLM이 `reusable` 분류와 `condition_key`를 함께 판단·산출한다.** 표기 흔들림으로 초기 미리채움 적중률이 낮을 수 있다는 트레이드오프는 수용하고, 실측에서 문제가 되면 그때 보강(예: 기존 키 목록을 생성 프롬프트에 공급)한다.

### 4.5 품질·순환성

- 질문도 같은 Opus 추출물이므로 canonical과 같은 오류 위험을 공유한다. **질문 텍스트를 감사(AuditSheet) 범위에 포함**시켜, 감사 확정된 질문만 승격한다. §10의 3중 가드(추출≠검수≠감사 모델)가 질문에도 그대로 적용된다.
- 미판정(감사 불일치) 항목의 질문을 그대로 사용자에게 내보내면 #11처럼 **잘못된 질문**이 나갈 수 있다. 흐름은: 감사에서 canonical·질문 확정(①) → 확정본만 사용자 확인(②). ①을 건너뛰고 ②로 직행하지 않는다.

### 4.6 비용·과금 연결

- 사전 생성 방식이므로 확인 루프의 런타임 LLM 비용은 0. 딥분석 자체를 추후 사용자별 유료 기능으로 전환할 경우 `usage_events.featureCode` + `contextRef(grantId)` 확장으로 즉시 과금 연결 가능(크레딧 원장 완비, feature code만 미등록).
