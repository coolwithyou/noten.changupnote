## 1. 성패 기준 1 판정: 분류가 제대로 되었는가 — 조건부 예

### 1.1 체계(taxonomy) 설계는 타당하다 — 재설계 불요

- **22차원 enum이 단일 원천**(`packages/contracts/src/enums.ts:9-32`)으로 contracts·DB pgEnum·LLM tool schema를 관통하고, 차원별 value 스키마를 계약 수준에서 강제한다(`bizinfo/criteria-contract.ts`). criterion = dimension × operator(7종) × kind(required/preferred/exclusion) × jsonb value 구조는 표현력과 판정 가능성의 균형이 좋다.
- **결격 3축(납세·신용·제재)은 근거 기반 설계다** — 운영 DB 배제조건 27종 분해(`docs/research/2026-07-11-공고매칭-14차원-확장-검토.md` §3.2)에서 도출했고, canonical 플래그 20종+예외 3종+온보딩 문항 커버 매핑이 단일 사전이며 완전성이 테스트로 강제된다. **외부 대조 결과 국내 어떤 경쟁 서비스도 결격 축을 모델링하지 않는다**(§3.1) — 실제 차별화 지점이 맞다.
- **LLM 추출에 대한 다층 방어가 실전 검증됐다** — 근거 span 필수(M1), 예약축 강등(M4), 이중 계약 검증. 재추출 1,477건 실전에서 위반 0건, contract 차단 0.14%.
- enum 수준에서 **놓친 축은 없다**. 실제 갭(시군구, 본점/사업장 구분, 개인 대상 공고)은 전부 값 해상도·게이팅 문제이지 차원 누락이 아니다.

### 1.2 그러나 "제대로 되었다"의 실질은 미달 — 세 겹의 간극

**① 구조화율 99.3%는 존재 지표다.** 활성 공고 1,847건 중 1,834건이 구조화 criterion을 1개 이상 보유한다는 뜻일 뿐이다. 층을 벗기면:

| 층 | 수치 | 근거 |
|---|---|---|
| 활성 우주 criteria 중 text_only(기계판정 불가) | 26.7% | DB 실측 6,094건 중 |
| needs_review(검수 전 LLM/휴리스틱 추출) | 전체 36.1% — kstartup 51.1% vs bizinfo 1.7% (open 기준) | DB 실측 |
| 사업자번호만 입력 시 즉시판정률 | 39.09% | recovery plan §2 기준선 |
| 운영 match_state 확정판정률(eligible+ineligible) | 41.8% | 147,139행 실측 |
| 평균 조건 확인도 | 36~47% | baseline v0 · dry-run |

독립 지표 3개(즉시판정률 39%, 확인도 36~47%, 확정판정률 42%)가 **~40%로 수렴**한다. "매칭을 가르는 조건이 실제로 기계 판정되는 비율"의 정직한 추정치는 35~45%이고, 나머지는 conditional로 사용자에게 되돌아간다.

**② 차원별 분포는 실제 요건 분포가 아니라 추출기 능력의 왜곡상이다.** prior_award 38건(0.04%)·founder_trait 28건은 중복수혜 배제·여성/청년 우대가 한국 공고의 최빈 조건군이라는 현실과 정면 모순 — 낮은 카운트는 희귀 조건이 아니라 **미추출**이다(prior_award 분해기는 골든 검증 대기로 기본 비활성, kstartup C2 규칙이 구조화 금지). preferred kind 0.27%도 같은 왜곡(가점·우대 조건 추출 부재). **이 분포 수치로 체계 완성도를 자평하면 안 된다.**

**③ 추출 정확도의 실측값이 0이다.** 추출 recall 평가 하네스는 완성됐지만 labelStatus=reviewed 라벨을 요구하는데 reviewed 골든셋이 저장소에 없다(`reviewed-expanded/` 디렉토리 자체 부재, 지식루프 실측: golden_set 1건·eval_runs 0건). bizinfo LLM 추출은 needs_review=true인 채로 **사람 검수 없이 매칭에 그대로 소비되고 있다**. 결격 FLAG_RULES의 무문맥 정규식(`/부도/`, `/파산/`, `/압류/`) 오탐률도 골든 없이는 측정 불가.

### 1.3 노력 배분의 역전 — 최빈 축이 최하위 투자

tail 축(financial_health 97건·insured_workforce 68건·investment 31건, 전 코퍼스 0.1% 미만)에는 전용 evaluator·질문 정의·소싱 커넥터까지 완비된 반면, **criteria 빈도 2위(16.8%)이자 dry-run hard fail의 사실상 전부(15/20, 18/100)인 region 축은**:
- regions 값이 시도 코드라는 런타임 보증이 없고(실오염 실측 §4.2),
- 시도 17개 해상도에 고정되어 지자체 공고의 1차 게이트인 '관내(시군구)' 판정이 불가하며(활성 우주의 80%가 지자체 공고 비중 큰 bizinfo),
- 회사 측은 다사업장을 표현할 수 없다.

### 1.4 closed 아카이브 29,030건 — 방치 상태이나 지금 병목의 해법 소재

유일한 소비처는 인사이트 통계뿐. 그러나 closed 공고는 revision이 불변이므로, 활성 공고 라벨링의 고질 문제(검수 완료 전 sourceRevision 드리프트 → stale 거부 → 재검수)를 원천 회피하는 **추출 골든셋 라벨링 소재**다. 결격 정규식 오탐률·industry 룰 recall 측정에 즉시 전용 가능. 재공고 예측(연례 프로그램 계보)은 가치 있는 후순위.

### 1.5 결론

**(b) 부분 보강** — 재설계(c)는 근거 없음(골격이 좋다), 그대로 진행(a)도 불가(검증 0 상태에서 LLM 추출이 판정에 소비 중 + 최빈 축 무결성 미보증). 보강 우선순위: ① 사람 검수 골든셋(모든 후속 게이트의 단일 병목) ② region 값 무결성·해상도 ③ audience 게이트 적용 ④ industry 태그 canonical ⑤ 아카이브 활용 명문화. **이 단계의 병목은 엔지니어링이 아니라 사람 라벨 노동의 투입 결정이다.**

## 2. 성패 기준 2 판정: 필드를 채우면 정확히 매칭되는가 — 조건부 아니오

### 2.1 실사용자가 사업자번호를 넣으면 실제로 일어나는 일

프로덕션 경로(`serviceData.ts`) 실코드 기준:

1. NTS 무료 사전 게이트(미등록 404/폐업 409, 팝빌 과금 차단) → 팝빌 기업정보(30일 캐시) → 캐시 히트 시에도 NTS 일 1회 휴폐업 재확인 → SMPP 확인서 오버레이(positive-only). **이 체인 자체는 과금 방어·최신성 설계가 잘 되어 있다.**
2. 자동 충전 결과: 운영 19축 중 **완전/근사 6축**(region·biz_age·industry 부분·size 근사·target_type 부분·business_status) + 부분 2축(SMPP 여성/장애인 인증). 근로복지공단(employees)·KIPRIS(ip)·창업확인서·DART·FSC·NICE·CODEF·registry **9계열은 전부 dev 하네스 전용**(production 404).
3. 매칭 판정: **eligible 4.3% / conditional 58.2% / ineligible 37.5%** (match_state 147,139행).
4. 최종 화면: recommendable **0.61%** — 활성 100건 dry-run에서는 0건. 실사용자 대부분이 보는 화면은 "지원 가능 0건 + 확인 필요 목록 + 질문 카드 1장".

검증 과정에서 확인된 **중요한 교정 2건** (1차 조사 결과보다 실제는 낫다):
- "company_profiles 504행 100% self_declared = 자동채움 실적 0"은 **과장이다.** 티저(익명) 매칭은 company_profiles를 쓰지 않고 팝빌 캐시를 매 요청 즉석 변환해 프로필을 조립하며, 실동작 실적이 있다(팝빌 캐시 5개 사업자·라이브 미터링 4건). 팝빌→프로필 영속 경로(`/api/web/companies/enrich`)도 프로덕션에 배선돼 있고 실행 실적이 0일 뿐이다.
- 티저 캐시 병합 브리지(apick/kised/kipris)는 운영에서도 실제 동작한다(dev와 운영이 같은 DB 공유). 정확한 한계는 "**운영 트래픽이 스스로 캐시를 채울 수 없어**, 커버리지가 dev에서 수동 조회한 소수 사업자번호로 고정"이라는 것. ※ 부수 리스크: dev 하네스와 운영이 같은 DB를 공유한다는 사실 자체가 확인됨 — 의도적 선택인지 점검 필요.

### 2.2 '정확히 매칭된다'와의 거리 — 5개 갭

| 갭 | 현재 | 필요 |
|---|---|---|
| 충전 | 자동 6~8/19축 (실측), 문서 목표(개인 9~12, 법인 11~15)는 실측 아닌 추정 | STEP 3 실표본 30사 측정 미실행 |
| 결격 | 외부 소싱→known_flags 배선 **0건**, 100% 자가신고(conf 0.6) | §6′-E 계약 확정 + 소스 배선 |
| 판정 | conditional 58.2% 적체, 해소 루프 실사용 이벤트 1건 | conditional_resolution_rate ≥0.60 게이트 선언만 존재 |
| 추천 | recommendable 0.61% | 사용자가 체감할 최소 수 건 |
| 검증 | reviewed 0건 — 정확도 수치 전무, legacy 9쌍마저 88.9%로 드리프트 | eligible_precision ≥0.90 등 9종 게이트 측정 불능 |

**결격 갭의 위험 성격 (검증으로 정밀화)**: 결격 unknown이면 eligible이 아니라 conditional에 머물므로, 사용자가 결격을 "몰라서 무응답"인 경우 오판정은 발생하지 않는다. 잘못된 '지원 가능' 판정은 **허위/오인 자가신고 시에만** 발생한다. 현재 운영 DB에 결격 문항 응답이 0건이라 결격 걸린 공고는 전부 conditional — 즉 위험은 아직 잠재 상태이고, 그 대가로 conditional 적체가 크다.

### 2.3 병목 사슬의 실체 — 설계 결함이 아니라 실행 항목

**충전 사슬**: 커넥터 dev 격리 → §6′-E 계약 미합의 → CODEF/NICE 실측·계약 대기 → 실표본 측정 미실행.
- **§6′-E는 외부 협상이 아니라 내부 문서 합의다**(매칭팀=소싱팀=창업자+AI 세션). 이것이 승격 전체를 잠그는 것은 프로세스 잠금이지 실질 장애가 아니다 — 반나절이면 확정 가능.
- §6′-E가 엄밀히 필수인 것은 **결격(known_flags) 배선뿐**이다. positive-only 축(employees/ip/certification)은 evidence-priority 계약이 코드에 있고 라이브 실측(394-86-03207)까지 끝났으며, 프로덕션 병합 경로가 이미 존재한다. **"조회 시 캐시 온디맨드 충전"만 붙이면 승격 완성 — 며칠짜리 작업.**
- CODEF(개인사업자 매출의 유일 경로, D1 실측 대기)와 NICE(무계약)는 상한을 결정하지만 임계 경로가 아님 — 병렬 트랙 유지.

**검증 사슬**: reviewed 0 → 추출 recall 미측정 → prior_award(개방 조건: 독립 검수 10건)·audience(검수 패킷 81건 생성돼 있음) 게이트 폐쇄 → conditional 적체. 500쌍보다 훨씬 싼 단위들이 게이트를 연다(§5).

### 2.4 결론

현재는 "필드가 다 채워지지도 않고, 채워진 매칭이 정확한지 아무도 수치로 답할 수 없는" 상태 — **조건부 아니오**. 단, 판정을 담는 그릇(엔진 정책·evidence 계약·질문 루프·측정 하네스)의 품질은 충분하며, 유일한 실사용 신호는 긍정적이다(질문 1개로 conditional 19건 중 14건 확정, 해소율 0.74). 병목이 전부 실행 항목이므로 4주 안에 "측정된 정확도를 수치로 말하는 베타"까지는 도달 가능하다(§5).

## 3. 방향성 평가: 이 길이 맞는가 — 맞다, 유지하라

### 3.1 '22축 구조화 + 규칙 결정론 + 사람 검수 골든셋'은 2026년 기준으로도 옳다

**LLM 직접 판정 대비**:
- 비용 구조: 현재 설계의 LLM 비용은 O(공고)다(1,477건 재추출 ≈ $4.6). LLM 직접 판정은 O(공고×회사×갱신) — 현 match_state 규모(147k행)만으로 전량 재계산 1회당 수백 달러, 회사 5,000 규모면 패스당 수만 달러. `/api/web/matches`의 요청당 라이브 재계산(최대 5,000건) UX 자체가 성립 불가.
- 더 결정적인 것: ① **결정론**(같은 입력=같은 판정이 신뢰 상품의 전제 — 2026-07-09 matching-trust-gate 사건이 실패 모드를 이미 보여줌) ② **rule_trace가 곧 제품 기능**("이 질문에 답하면 14건 확정"의 정보가치 질문 플래너, 업력 해금, scoped refresh는 블랙박스 판정으로 구조적 불가능) ③ 한국 행정 뉘앙스('관내'·'중복수혜'·'예비창업자 제외')의 환각 위험과 오판 비용 비대칭.
- **외부 대조 재확인**: 복지·세제 자격 판정의 업계 정석(PolicyEngine/OpenFisca 계열 rules-as-code)이 정확히 이 구성 — 결정론 연산은 코드, 비정형 추출만 constrained LLM. 미국 Instrumentl은 AI 매칭을 표방하지만 실체는 관련성 랭킹이며, 이는 hard eligibility가 얕은 미국 재단 그랜트라서 가능한 정의다. 한국 공공지원사업은 hard gate가 깊어 **랭킹이 아니라 판정 문제**이므로 룰 엔진 선택이 도메인에 맞다.

**임베딩 검색+rerank**는 대안이 아니라 다른 문제(발견)의 해법 — "서울 소재 3년차 제조업"과 "서울 제외, 업력 3년 이상 제외"는 임베딩 공간에서 가깝다. 관련성 보조로는 후일 유용.

**LLM을 더 넣을 자리는 판정이 아니라 3곳**: ① 골든셋 사전 라벨러(§5.2) ② conditional 해소 보조(text_only 조건+원문을 읽고 근거 인용 딸린 draft 판정 — 판정을 바꾸지 않고 "AI 원문 확인(참고용)" 레이어/검수 우선순위로) ③ 섀도 평가 오라클(엔진 vs 독립 LLM 불일치 채굴로 골든셋 없이 버그 발견).

**구조적 경고 하나**: 22축은 복잡도 예산의 상한 근처다. text_only 잔존을 축 추가로 풀려는 유혹을 경계할 것(축 1개 = evaluator+사전+질문+검수 부담의 곱). 잔존 long-tail은 축이 아니라 LLM 해소 보조로.

### 3.2 conditional 58% / recommendable 0.61% — 정책의 대가가 아니라 파이프라인 미완의 결과

세 숫자는 원인이 다르므로 다르게 풀어야 한다:
- **recommendable 0.61%** = 공고 측 검수·변환 적체의 함수. 활성 100건 중 98건이 manifest partial(최대 원인: 첨부 미변환 2,787건 대기)이고 review gate 7단계가 곱해진 필연. **정책 상수가 아니다.**
- **eligible 4.3%** = 회사 프로필 결손의 함수. 소싱 승격 시 직접 상승.
- **conditional 58%** = 상당 부분 도메인 본질. 승부처는 이를 "확인 필요 목록"이 아니라 **"확정까지의 경로"**로 보여주는 것. 실측 유일 이벤트(질문 1개 → 19건 중 14건 확정)가 이 제품의 매직 루프를 이미 증명했는데, 현재 UI는 needs_profile_input(답하면 확정)과 needs_core_review(대기)를 한 섹션에 뭉뚱그려(`logic.ts:543-545`) 루프를 묻어버렸다. first-mission plan §1.3의 4상태 문구 계약이 정답을 정의해놓고 미구현.

**"틀린 추천 없음" 원칙은 완화하지 말 것** — 완화하는 순간 보조금 스팸 사이트와의 유일한 차이가 사라진다.

### 3.3 외부 환경 — 차별화는 실재하나 '설계상 차별화'이지 '실증된 차별화'가 아직 아니다

국내 실측 비교(웹 조사, 2026-07-13): 보조금24(필터 ~4축)·기업마당 맞춤서비스(자가입력 프로필 필터)·웰로비즈(OCR+알림+신청 오퍼레이션 DX, 결격 구조화 없음)·소상공인24(자격 확인은 정책자금 심사 단계만) — **어디에도 결격 구조화 판정이 없다.** 내부 외부대조 문서(§4.1)의 결론을 독립 재확인.

단 세 가지 경계 신호:
1. **스키마는 복제 가능하다** — cunote 스스로 1,477건을 $4.6에 재추출한 것이 증거. 자금력 있는 경쟁자(웰로비즈: 63만 사업 데이터, 혜택알리미 공식 민간 채널 선점)가 결격 구조화를 붙이는 데 몇 주의 엔지니어링이면 된다. **복제 불가능한 해자는 ①검수된 정답 데이터 ②회사 측 소싱 배선 ③선정 결과 피드백 데이터 — 셋 다 현재 0이다.**
2. **공공마이데이터 기업 확대(2026년 4월 249종 예정)** — CODEF 스크래핑의 대체 경로이자 규제 리스크(금융 마이데이터의 2022년 스크래핑 전면 금지 전례). STEP 4A(이용기관 승인 검토)가 문서에 인지돼 있으나 **사전 문의조차 미실행**. 먼저 움직이는 쪽이 가져가는 순풍.
3. **정부의 발견(discovery) 층 잠식** — 혜택알리미(3,600종 선제 알림)·소상공인24가 기업으로 확장되면 "찾아주는 서비스" 포지션은 무료 정부 채널과 정면 경쟁. 방어선은 발견이 아니라 **판정 정밀도 + 지원서 작성** 층이다. 결격 기능은 "자동 검증"이 아니라 **"구조화된 pre-flight 점검(한 번 답하면 전 공고 전파)"**으로 포지셔닝해야 과장 없이 경쟁사가 못 주는 가치가 된다.

### 3.4 지원서 가이드 트랙과의 자원 배분

**변환 서버(Gate 0 검증 완료, LibreOffice+H2Orestart)는 지원서 트랙의 매몰비용이 아니라 매칭 트랙의 최대 지렛대다.** 활성 공고 98%의 manifest partial(=recommendable 강등 최대 원인)이 첨부 미변환에서 오고, "매칭을 가르는 자격요건은 첨부 HWP에 있다"가 공고API 분석의 결론이었다. Cloud Run 재배포 → 활성 공고 첨부 일괄 변환 → LLM 재추출 → manifest 승격이 recommendable을 올리는 가장 싼 경로. 반면 HWPX 채움·Gate 2/3 추가 투자는 매칭 게이트 통과까지 명시 동결이 맞다(매칭이 추천을 못 내는데 지원서 쓸 사용자가 없다).

### 3.5 문서 위계 충돌

마스터 설계 문서는 2026-07-04 이후 미갱신인 채 "충돌 시 마스터 우선"을 선언 중이고(매칭 언급 8회, 정량 기준 전무), 실질 SSOT는 first-mission plan("완료 판정은 본 문서를 따른다")이다. AI 에이전트 위임 중심 조직에서 문서 위계 충돌은 곧 **에이전트 오배치**다. 마스터 상단에 "매칭 우선, 완료 판정은 first-mission plan" 위계 단락 하나로 해소 가능(반나절).

## 4. 검증된 구체 결함 목록

8건을 반박 검증했다(회의적 검증자가 코드 실행·git 이력·DB 실측으로 반박 시도). confirmed 4건, 부분 교정 4건. 아래는 교정 반영 후의 확정 서술이다.

### 4.1 [confirmed·긴급] 업력 해금(unlock) 회귀 — pnpm test가 현재 깨져 있다

미커밋 scoring-verification-v3 리팩토링에서 `evaluateBizAge`의 min 미달 fail 분기가 삭제되며 `unlock_at_months` 방출이 소실됐다(커밋 3210a97에서 도입 → HEAD까지 존속 → 작업트리에서 소멸). 생산처 0곳 vs 소비처 4곳으로, **soon 버킷·"N개월 후 충족" 해금 칩·로드맵 eligibleFrom 전이가 전부 죽은 기능**이 됐다. `pnpm verify:service-usecases`가 AssertionError(expected 'soon', actual 'preparable')로 실패하며 **이 스크립트는 pnpm test 체인에 포함되어 있어 전체 테스트가 통과 불가능한 상태**다. 2차 영향: match_state 재계산 시 eligible_from이 null로 지속. `packages/core/dist`도 이미 리팩토링 코드로 재빌드되어 로컬 실행 경로에서 즉시 발현 중.

### 4.2 [confirmed·긴급] region 값 무결성 — 이론이 아니라 실오염

- 검증 로직이 어디에도 없다: canonicalize는 String·trim만, contract는 "문자열 배열"만, kstartup parseRegion은 미등록 라벨을 **원문 그대로** regions에 넣는 폴백(`REGION_CODES[label] ?? label`).
- **실측**: regions 원소 13,924개 중 비 2자리 코드 9개('국내'×3, 'domestic'×2, 'KR', 'all', '전국', '전남광주') — 전부 required·needs_review=false라 **모든 회사가 소재지 무관 확정 탈락**, 이 중 open 공고 5건. 추가로 **2자리 형식이지만 잘못된 코드체계**('37'×12 — 경북 시군구 라벨을 시도 코드 자리에 저장, 경북 회사 코드는 47이라 전원 fail, open 상태) — regex 검증만으로는 못 잡는 표면이 더 넓다.
- 구제 경로 부재는 예상보다 강하다: region fail은 질문 플래너 대상이 아니고(unknown만 후보), 팝빌 온보딩 회사는 region evidence가 authoritative·complete라 자가 교체가 400으로 거부되며, `allowAuthoritativeOverride`를 전달하는 호출자가 코드베이스에 전무하고, 설정 UI에도 region 편집이 없다.
- 회사 다사업장(지사·공장) 미표현 확인 — 본사 경기+공장 인천인 회사는 인천 한정 공고에서 구제 불가.

### 4.3 [confirmed] 평가 게이트 모순 + CI 부재 + 골든셋 미커밋

- `verify:matching-eval-v2` **실행 실패**(178229: expected ineligible, got conditional). 미커밋 신규 게이트 v3는 **바로 그 드리프트가 정확히 1건 존재해야 통과**하도록 작성됨 — 두 게이트는 어떤 엔진 상태에서도 동시 통과 불가능한 모순. legacy accuracy 실측 8/9=88.9%(baseline 문서의 100%는 stale).
- v2/v3/matching-unit은 pnpm test 체인에 미포함(포함된 v1은 6케이스뿐이라 드리프트를 못 잡음), `.github` 디렉토리 자체가 없어 CI 부재.
- 골든셋·홀드아웃 manifest·평가 모듈 대부분 git 미추적 — "사전 고정" 장치의 시점 증명 불가, 유실 위험. **골든 라벨을 갱신하지 않고 검증 스크립트를 엔진에 맞춰 고친 것은 '정답이 엔진을 따라가는' 역방향 관행의 씨앗**이다(178229의 정답을 라벨 갱신 vs 게이트 수정 중 한 방향으로 명시 확정 필요).

### 4.4 [confirmed] audience 게이트 미적용 — 단, 의도된 게이트이고 영향 규모는 작다

분류기는 인제스트마다 실행되지만 grants에 audience 컬럼이 없어 미영속, 매칭 필터 미적용. 활성 1,802건 중 individual 6건(0.33%, 전부 교육생 모집류·safeToExclude)·mixed 28·unknown 6 — 최대 2.2%. "검수 precision ≥0.95 전 write 금지"라는 **의도된 게이트**이며, 검수 패킷 81건이 이미 생성돼 있어 약 1일 검수로 개방 가능. (참고: 0045 마이그레이션은 audience와 무관한 profile_question_events이며 적용 완료 상태 — 1차 조사의 부가 전제 오류를 교정함.)

### 4.5 [교정됨] 결격 3축 — "외부 소싱발 known_flags 쓰기 0건"이 정확한 서술

known_flags 쓰기 자체는 자가신고 문항 경로에 존재한다. NICE 커넥터·조달청 어댑터·registry_index 스키마·admin 업로드까지 3단계가 준비돼 있고 **배선만 남은 상태**. 위험은 "몰라서 무응답 → conditional 유지(오판정 없음)"이며, 오판정은 허위/오인 신고 시에만. 현재 결격 문항 응답 0건 → 결격 걸린 공고 전부 conditional. sanction의 유일 exact 소스(조달청 부정당 CSV)는 사업자번호 컬럼 부재 파일 문제로 차단 중 — 전량본 재확보가 즉효약.

### 4.6 [교정됨] deferUnreviewedHardFail — 골든셋에는 구조적, 운영에는 미미(현재)

draft 골든셋 500쌍의 엔진 예측 ineligible 0은 이 정책 + draft 전량 needs_review 하드코딩의 산물이 맞다(층화 퇴화·검수 anchoring 위험). 단 **이 함수는 HEAD에 없는 미커밋 워킹트리(v5) 코드**이고, 운영 match_state의 ineligible 55,234건은 98.8%가 v3 엔진 산출물이며 v5 전면 적용 시 뒤집히는 것은 ~0.9%(507건)뿐이다. 운영 hard criteria의 대부분이 needs_review=false(bizinfo 1.7%)라 실판정 영향은 작다. ※ match_state에 v5 행 78건 존재 — 미커밋 엔진으로 부분 refresh가 이미 수행된 흔적. 미커밋 코드가 운영 데이터에 스며들고 있다.

### 4.7 [교정됨] 관련성 업종 신호 — "전면 누락"이 아니라 "정확 매칭 경로가 ~98%에서 닫힘"

f_industries 충전률 0.003%(31,388건 중 1건)는 맞지만, relevance는 런타임 criteria projection 병합 + 제목 키워드 겹침 fallback을 갖고 있고, 무신호 시 0점이 아니라 컴포넌트 스킵(점수 30 캡)이다. relevanceScore는 정렬 4순위 키라 영향 제한적. 백필해도 projection 조건 충족 공고가 594건(1.9%)뿐 — **근본 원인은 industry criteria의 87%가 required/text_only**라는 추출 문제로 되돌아간다.

### 4.8 기타 확인된 결함 (엔진·소싱, 요약)

- scalar evidence gate가 dimension 판정보다 먼저 단락해 **nationwide 공고까지 unknown 강등** → 불필요 conditional·불필요 지역 질문 (`match.ts:159-166`).
- founder_trait/ip는 canonical 사전 없이 완전 문자열 일치 — '여성'≠'여성기업' no-hit으로 정당한 pass가 conditional 잔존.
- industry 라벨 3자 이상 양방향 substring 매칭('바이오' ⊂ '바이오디젤')의 오긍정 여지 — 검수 전은 review gate가 완충하나 needs_review=false 구조화분은 eligible 직행 가능.
- region 질문 답변의 라벨→코드 변환 폴백이 미등재 라벨을 code로 저장해 프로필 오염 가능.
- provider 식별자 불일치('kised' vs 'startup_confirmation') — 단일 사전 부재로 우선순위 무성 역전 위험.
- dev 하네스에서 FSC/NICE가 같은 필드 키에 경합(Promise.all 완료 순서 의존) — 승격 시 evidence-priority로 정리 필요.
- 사업자 유형 추론이 중간 2자리 휴리스틱(81~88=법인) — 팝빌 personCorpCode를 안 씀.
- bizinfo stale open 469건(마감 경과인데 status open), dedup_links 0건(교차 소스 중복 이중 노출 가능), `.env.example`에 신규 소싱 키 전부 누락.

## 5. 권고 — 완주 순서의 재설계

### 5.0 핵심 재설계: 검수 병목을 "AI 사전라벨 + 창업자 단독 검수"로 푼다

현행 설계(독립 사람 2인 × pair 500쌍, 담당·일정 미정)는 1인 조직과 불일치해 reviewed=0에서 무기한 정지 중이다. 그런데 **`v3-annotations.ts`의 AI 차단 regex는 reviewerId에만 적용된다** — annotator는 AI여도 스키마가 허용한다. Gate 1에서 이미 확립한 "opus 사전라벨 → 사람 검수 확정" 패턴의 이식이며, 라벨러가 매칭 엔진이 아닌 독립 LLM이면 순환성 차단 철학과 충돌하지 않는다.

- 독립 LLM이 공고 100건·pair를 사전 라벨 → 엔진 예측과 **불일치하는 쌍만 전수 검수** + 일치 쌍 20% 표본 감사 → 창업자가 유일한 사람 reviewer.
- 초기 게이트를 500 → **150~200쌍 층화 표본 "베타 게이트"**로 하향(신뢰구간 넓게 명시), 500쌍은 정식 오픈 게이트로 유지.
- 예상 노동: 공고 100건 × 10~15분 + 불일치 pair ~150건 × 4분 ≈ **창업자 풀타임 1.5~2주**.
- 주의: reviewer가 확인만 하는 구조의 anchoring 위험은 남는다 — holdout은 blind 유지(이미 구현됨), 일치 쌍 표본 감사에서 불일치 발견 시 감사 비율 상향.

**순서 역전 — pair보다 공고 검수 먼저.** 공고 criterion 검수는 이중 배당이다: (a) pair 라벨의 선행 조건이고 (b) reviewed-publication으로 프로덕션 needs_review가 풀리며 **recommendable이 즉시 해금**된다. 검수 노동 한 줄기가 골든셋과 제품 개선 두 출력을 낸다. 대상 선정은 `planExtractionImprovements`("이 공고를 고치면 몇 개 회사의 eligible이 풀리는가 × 마감 임박도")가 이미 해준다. 추출 정확도 검증분은 **closed 아카이브에서 표집**(revision 불변 → stale 재검수 낭비 없음).

- **industry 긍정 태그 canonical 사전**: 운영 태그 전수 덤프 → 클러스터링 → 사전 enum 전환, substring 매칭 폐기. (빈도 3위 축의 정밀도)

### 5.4 4주 후 도달 가능한 주장 수준

"우리 매칭은 정확하다"(무조건)는 4주 안에 불가하다. 도달 가능한 것은:

> "활성 노출 상위 공고 100건은 사람이 검수했고 추출 recall은 X%다. 홀드아웃 27~90쌍에서 eligible/ineligible precision Y%를 확인했다. 실표본 30사에서 사업자번호만으로 축 Z/19가 자동 충전됐다."

즉 **측정된 정확도를 수치로 말하는 베타** — 현재의 "수치 0개" 상태와 질적으로 다르고, 경쟁 방어 자산(§3.3)의 첫 축적이다. 단서 두 가지: (1) 임계 자원은 코드가 아니라 검수 노동 — W1에 검수 체계 결정이 미끄러지면 전체가 민다. (2) 결격 3축이 자가신고인 동안 "지원 가능성이 높음" 문구 배포 게이트를 닫아둔 현 방침은 4주 후에도 유지가 옳다.

## 6. 잘된 것들 (보존해야 할 설계 자산)

평가의 균형을 위해 명시한다. 아래는 반박 검증·실측을 거친 확인 사항이다.

1. **오판 비용 비대칭의 일관 구현** — "false ineligible이 가장 비싸다"가 주석·정책·지표(eligible_recall이 conditional 보존 인정)까지 관통. dry-run에서 발견된 false-negative 4건(중소↔중소기업 문자열 불일치 등)이 전부 코드+회귀 테스트로 반영 확인.
2. **점수 과신 청산** — 기존 60~95점 fitScore가 확인 완료 수준을 과장했음을 인정하고 '조건 확인도'로 재정의, UI 라벨·숨김 처리까지 배선. 자기 판정을 스스로 의심하는 review gate 4-tier.
3. **라벨 순환성 방지의 다층 강제** — annotator≠reviewer, AI reviewer regex 차단, 엔진 예측 sentinel 강제 교체, ruleset/scoring 버전+입력 SHA-256 fingerprint 드리프트 검출, blind holdout+확인 문구 게이트. 스키마 수준 강제라는 점이 중요.
4. **증거 원천 5분류와 결정론적 우선순위 리졸버** — partial이 complete를 못 덮고, 자가신고가 권위값을 못 덮는 이중 차단, 패자 supplemental 보존(provenance).
5. **정보가치 기반 질문 플래너** — 영향 공고 수×판정 확정 수×마감 긴급도−입력 부담, 구간→정밀 2단계, 모름 TTL, 답변 후 scoped refresh 폐루프. 실측 해소율 0.74가 유효성 시사.
6. **과금 방어 소싱 체인** — NTS 무료 사전 게이트, in-flight dedup, 캐시 히트 시 휴폐업 일 1회 재확인.
7. **재현 가능한 평가 인프라** — 해시 기반 결정론 샘플링, 오프라인 CSP 차단 워크벤치, 배치 무결성 게이트, 층화 before/after 백필 측정 방법론(21.1%→13.9% 실측).
8. **원천별 추출 프로파일 분리** — K-Startup deterministic(LLM은 검수 전제 draft만) vs bizinfo LLM 인라인(전량 needs_review+span 필수) — 난이도 차이를 문서 분석으로 확인 후 다르게 설계.

1차 조사(6 에이전트, 도구 호출 359회) → 2차 반박 검증+비판(12 에이전트, 도구 호출 433회). 검증 8건 판정: V1 unlock 회귀 **confirmed** · V2 region 취약성 **confirmed**(규모 보정: 오염 0.065%+코드계 오류 13건) · V3 커넥터 격리 **partially**(티저 즉석 조립은 동작 — "자동채움 0" 과장 교정) · V4 게이트 모순 **confirmed** · V5 관련성 신호 **partially**("전면 누락"→"정확 매칭 경로 ~98% 폐쇄"로 교정) · V6 audience **confirmed**(의도된 게이트, 영향 0.33~2.2%) · V7 결격 자가신고 **partially**(위험은 허위/오인 신고 시로 한정 교정) · V8 defer 영향 **partially**(운영 영향 ~0.9%로 정량화, 골든셋 층화 퇴화는 확인).
