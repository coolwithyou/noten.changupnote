# 인터페이스 단순화 제안 (2026-07-14)

> **전제**: 짝 문서 `2026-07-14-인터페이스-복잡도-현황평가.md`의 판정을 입력으로 한다.
> **목표**: 시스템 복잡도(22축 매칭·룰 트레이스·문서 변환 사다리)는 그대로 두고, **표면을 "루프"로 재구성**한다 — 매 화면이 사용자의 3가지 질문(받을 수 있어? / 뭘 하면 돼? / 언제까지?)에만 즉답하고 나머지는 요청 시 공개.
> **범위**: 제안만. 구현 없음. 실행 순서는 마스터 실행 문서(WS-D)와 정합하도록 설계.

## 0. 설계 철학 — "복잡한 엔진, 단순한 조종석"

현재 UI는 엔진룸의 계기판을 전부 조종석에 옮겨 놓은 상태다. 제안의 방향은 하나다:

> **화면은 항상 "결론 + 다음 행동 하나"만 말한다. 근거·점수·트레이스·상태는 물으면 보여준다.**

이것은 정보 삭제가 아니다. 정직성(룰 트레이스, 근거 등급, D9 문구 게이트, 면책)은 이 제품의 방어선이므로 전부 보존하되, **위계를 push(기본 노출)에서 pull(요청 시 공개)로 옮긴다.**

## 1. 인터페이스 헌법 (복잡도 예산)

공고 요약 페이지가 코드 주석에 "5섹션 상한·입력 필드 금지"를 계약으로 박아 성공한 방식을 전 화면으로 확장한다. 아래 8조를 각 화면 컴포넌트 헤더 주석 + UX 계약 문서로 명문화하고, 위반은 리뷰에서 반려한다.

| 조항 | 규칙 | 현재 최악 사례 |
|---|---|---|
| **제1조** | 한 화면 = 사용자 질문 1개. 그 화면이 어떤 질문에 답하는지 한 문장으로 못 쓰면 화면을 쪼갠다 | 매칭 결과(판정+프로필 입력+탐색+피드백 4역할 혼재) |
| **제2조** | 주 CTA는 화면당 1개. 나머지 행동은 보조 스타일 또는 오버플로(⋯) | FieldCard 버튼 6개 |
| **제3조** | 접힘 카드의 정보 요소 ≤ 4 (제목 · 상태 · 마감 · 금액) | ProgramCard 접힘 상태 8요소 |
| **제4조** | 사용자 노출 상태 어휘는 전 제품 공통 **판정 4상태 + 필드 3상태**로 고정 (§2) | 전사 30+종 |
| **제5조** | 결론·다음 행동은 push, 근거·점수·트레이스·출처는 pull | 룰 트레이스 기본 전시, 첫 카드 defaultOpen |
| **제6조** | 같은 사용자 질문에 답하는 표면은 1개. 새 표면 추가 금지, 기존 표면 재사용·통합 우선 | 공고 선택 표면 4개 병존 |
| **제7조** | 입력 요구는 한 번에 1개, 반드시 보상과 함께 ("이걸 답하면 N건 확정") | 11필드 그리드 일괄 노출 |
| **제8조** | 목록의 기본 노출 ≤ 5건 + 더보기. 전량 나열 금지 | Programs 전량 세로 나열 |

## 2. 상태 어휘 SSOT (선행 과제)

설계 문서 3곳에서 다르게 정의된 "4상태"를 하나로 확정하고, 모든 표면·문서가 이를 참조한다. 제안 어휘 (D9 문구 게이트 준수 — "지원 가능성이 높음" 사용 금지):

**판정 4상태 (공고 단위)**

| 코드 | 사용자 문구 | 의미 | 현재 대응 |
|---|---|---|---|
| `open` | **지금 신청 가능** | 확인된 조건 전부 충족 | eligible |
| `one_answer` | **답하면 확정** | 질문에 답하면 판정 확정 (needs_profile_input) | conditional 일부 |
| `check_source` | **원문 확인 필요** | 공고 text-only 등 사람 확인 필요 (needs_core_review) | conditional 일부 |
| `closed` | **이번엔 어려움** | 배제 조건 해당 | ineligible |

핵심은 `one_answer`와 `check_source`의 분리 — 이것이 WS-D.1의 요구이자 매직 루프의 전제다. "답하면 확정"은 사용자가 풀 수 있고, "원문 확인 필요"는 기다리거나 직접 읽어야 한다. 서로 다른 다음 행동인데 현재는 한 서랍에 있다.

**보조 분류 (버킷 `OpportunityBucket = now/soon/preparable/conditional` 흡수 규칙)** — 확장된 매칭 로직의 시간축·준비축 분류는 판정 뱃지를 늘리지 않고 다음처럼 흡수한다:

| 버킷 | 처리 | 사용자 표기 |
|---|---|---|
| `soon` (접수 예정) | 뱃지가 아니라 **마감 슬롯의 표기**로 | 카드 4요소 중 D-day 자리에 `7/21 접수 시작` |
| `preparable` (조건 취득 시 가능) | 뱃지가 아니라 **접힌 섹션**으로 | 결과 화면에 `준비하면 열려요 N건 ▸` 섹션 추가 (기본 접힘) |

판정 뱃지 어휘는 4개로 고정 유지하되, 정보는 삭제하지 않는다(제5조·제4조 동시 충족). `soon`/`preparable` 공고의 카드 펼침에는 "무엇을 하면/언제부터 열리는지" 한 줄(unlock 조건)을 표시한다.

**작성 지원 3모드 (`WriteSupportLevel = template_fill/ai_draft/web_form_guide`)** — 접힘 카드에서는 강등하되(제3조), 다음 위치에서 노출한다: ①공고 요약(P5)의 주 CTA 문구 분기 ②작성 화면(§3.4)의 모드 분기. 매칭 결과 카드에서는 펼침 시 pull 영역에만.

**필드 3상태 (지원서 항목 단위)** — 현행 7종을 흡수:

| 사용자 문구 | 흡수하는 현행 상태 |
|---|---|
| **채워짐** | 확정 / 확정(수정됨) / 건너뜀(취소선 표기) |
| **확인만 하면 됨** | 자동 입력(사업자 정보) / 자동 입력(양식 기본값) / 제안 |
| **입력 필요** | 미입력 |

출처 구분("사업자 정보에서" vs "AI 제안")은 pull — 카드 상세에서만. 오버레이 색상 5종도 이 3상태 + 경고 1색으로 축소.

## 3. 화면별 재설계안

### 3.1 매칭 결과 — "목록"에서 "루프"로 (최우선)

**제거·강등**: StatTile 3개(헤드라인 문장에 흡수, "총 지원 가능액" 합산은 오해 위험으로 폐기), ProfileSection 11필드 그리드(→ "내 정보" 시트로 이동), 피드백 4버튼(→ ⋯ 메뉴), 첫 카드 defaultOpen(폐지), 전량 나열(→ 5건 + 더보기).

**승격**: 다음 질문 카드를 화면 2순위로. 실측(질문 1개 → 19건 중 14건 확정)이 증명한 제품의 심장을 주인공으로.

**정밀도 루프 확정 (2026-07-14 사용자 피드백 반영)** — 결과 화면의 동기 설계 4원칙:
1. **자동 vs 직접의 구분이 정보 구조의 1축** — 사업자번호로 자동 확인된 정보(민트 체크)와 직접 채울 정보(블루 빈 슬롯)를 전 표면에서 같은 문법으로 구분
2. **매칭 정밀도 게이지 상시 노출** — 답변·입력마다 게이지가 눈에 띄게 상승. "정보 충족도 7/11" 같은 관리자형 표기 대신 동기형 표기(`정밀도 64% · 다 채우면 91%`)
3. **모든 빈 슬롯에 보상 표기** — 빈 항목은 결핍이 아니라 보상으로: `채우면 정밀도 +5%p · 공고 2건 판정에 영향`. 채우고 싶게 만드는 것이 디자인 포인트
4. **좁혀짐의 시각화** — 답할 때마다 확정 카드는 민트로 올라오고 제외 카드는 흐려지며 빠진다. 결과가 안 바뀌는 답변도 정밀도 상승으로 보상(`바뀐 공고는 없어요 — 정밀도는 올랐어요`)

```
┌──────────────────────────────────────────┐
│  지금 신청 가능 3건 · 답하면 확정 11건       │  ← Q1 즉답 (한 문장)
│                                          │
│  ┌────────────────────────────────────┐  │
│  │ ❓ 작년 매출이 10억 원 미만인가요?      │  │  ← 화면의 유일한 입력
│  │    [예] [아니요] [모름]               │  │     보상 명시가 헤드라인
│  │    답하면 11건 중 최대 9건이 확정돼요   │  │
│  └────────────────────────────────────┘  │
│                                          │
│  지금 신청 가능                            │
│  ├ 초기창업패키지        D-12  최대 1억     │  ← 접힘 = 4요소 (제3조)
│  ├ 청년창업사관학교      D-30  최대 1억     │
│  └ 소상공인 성장지원     상시   2천만       │
│                                          │
│  답하면 확정 11건 · 원문 확인 필요 4건 ▸    │  ← 접힌 섹션
│  내 정보 7/11 확정 · 수정 ▸               │  ← 시트로 강등된 프로필
└──────────────────────────────────────────┘
```

- 질문에 답할 때마다 카드가 `답하면 확정 → 지금 신청 가능`으로 **이동하는 것이 보이게** — 루프의 보상을 시각화. 이것이 WS-D.1 "답변 1개로 확정되는 공고 N건"의 구현 형태 제안이기도 하다
- 카드 펼침 시: 충족 조건은 "충족 12건 ✓" 한 줄 요약으로 접고, **확인 필요한 조건만** 행으로 표시. 전체 룰 트레이스는 "근거 전체 보기" pull
- 관련성 뱃지·추천 근거 3줄·면책문 → 펼침 하단 pull 영역으로

### 3.4 지원서 워크스페이스 — 인터뷰 모드 (매칭 베타 게이트 이후)

> **범위 결정(2026-07-14, 같은 날 번복)**: 처음 "리뷰어 성격 페이지"로 보고 디자인 라운드에서 제외했으나 이는 내부 리뷰어 워크스페이스(internal/review)와의 혼동 — 지원서 작성 도우미(HWP 렌더링·칸별 설명·값 추천/삽입)는 사용자용 핵심 기능으로 **디자인 범위 복귀**(전달 프롬프트 화면 8). 단, 구현 시점은 여전히 Phase C(매칭 베타 게이트 이후)로 유지.

실행 문서 §4 동결을 존중해 **Phase C**로 배치. 기능 추가가 아닌 정보 재배치.

**작성 지원 3모드를 모두 커버한다** (이미 개발된 기능 — HWPX 채움 트랙 완료):
- **`template_fill` (원본 양식 채움, 사다리 a)**: 아래 인터뷰 모드 본체. HWP 미리보기 + 필드 하이라이트 + HWPX 다운로드
- **`ai_draft` (원본 양식 없음 → 초안 작성, 현행 DraftFallbackEditor)**: 같은 인터뷰 문법으로 문항 확인 → 초안 미리보기 → 내보내기(DOCX/Markdown)의 3단계. 고지 1문장 템플릿("이 공고는 별도 원본 양식이 없어 초안으로 도와드려요")
- **`web_form_guide` (웹폼 접수 안내)**: 신청 항목 체크리스트 + 항목별 준비 값 + 접수처 원문 이동
- **변환 중/실패 (사다리 b·c 진입 전)**: 공고 요약(P5)의 CTA가 분기 — "서류 준비 중 — 채팅으로 먼저 물어보기" (현행 로직 유지)

- **기본 뷰 = 인터뷰 모드**: "확인이 필요한 항목 k개 중 1번째"를 한 번에 하나씩. 좌측 프리뷰는 해당 필드 위치로 자동 스크롤·하이라이트. TurboTax 규칙 적용 — 쉬운 확인(자동값 승인)은 묶고, 생각이 필요한 입력은 단독 표시
- 필드 카드 액션 6 → **주 행동 1 + ⋯**: 자동값 있으면 `[반영]`, 없으면 `[입력]`. 건너뛰기·되돌리기·다시 제안은 오버플로, "이 항목이 뭐예요?"는 채팅 프리필로 일원화
- 전체 필드 목록은 보조 탭으로 존치 (전문 사용자용)
- honestNotice 7종 → 템플릿 1개("이 서류는 {사유}. 대신 {대안}을 도와드려요")로 통합, 사유만 치환
- DraftFallbackEditor 900줄 단일 화면 → 초안 생성 / 문항 확인(인터뷰 모드 재사용) / 내보내기 3단계 분리
- 진행 표시는 "k/N 확인 완료" 하나만

### 3.5 신청 관리 — 8레인 보드 → 리스트 하나

- 통계 카드 8 + 토글 8 + 레인 8의 3중 반복 → **단일 리스트 + 상태 칩 필터 1중**
- 사용자 노출 그룹은 3개(진행 중 / 결과 대기 / 종료)로, 내부 8단계는 데이터 모델에 보존하고 카드 상세에서만 노출
- 카드당 액션 7~9 → 주 행동 1(단계에 맞는 다음 행동) + ⋯
- 제출 후 관리 입력(담당자/리마인더/메모)은 카드 상세 시트로

### 3.6 공고별 AI 조언 — 분류 외 조건의 처리 (2026-07-14 사용자 결정: 채팅 폐기 → 조언)

질문에는 두 종류가 있고, 처리 인터페이스도 둘로 나뉜다:

| 결측의 종류 | 예 | 처리 인터페이스 |
|---|---|---|
| **분류 내(22축) 결측** — 구조화된 표준 질문이 이미 존재 | 연 매출, 상시근로자, 대표자 연령 | **폼 질문 카드** (§3.1의 다음 질문 카드, WS-D.1) — 답하면 재판정·좁혀짐 |
| **분류 외 결측** — 공고가 요구하는 고유·세부 조건(text_only 등) | "대표자 주민등록상 주소지", "특정 품목·세부 분야 영위" | **공고별 AI 조언** — 묻지 않고 카드 안에서 안내 |

주의: `수출 실적`(export_performance)·`사업장 보유`(premises)는 분류 **내** 축(승격 대기, `CRITERION_DIMENSIONS` 22축에 포함)이므로 분류 외 예시로 쓰지 않는다 (2026-07-14 실코드 대조).

**결정 경위**: 처음 검토한 대화형(채팅으로 사실 수집 → 재필터링)은 폐기. 이유 — ①채팅 패널은 결국 표면 하나 추가(제6조와 긴장) ②실시간 LLM 파싱·확인 카드·Phase 5 개방이라는 무거운 선행조건 ③사용자를 심문하는 흐름은 "결과를 보러 온" 맥락과 충돌.

**형태**: 카드 펼침 안의 정적 조언 블록 1개 —
> 💡 이 공고는 대표자의 주민등록상 주소지가 부산인 기업만 지원할 수 있어요. 사업장이 부산이어도 대표자 주소지가 다른 지역이면 대상이 아니에요.

- **정적 생성 + 검수 가능**: 공고 수집·정규화 시점에 배치로 생성해 캐시. 공고 검수 게이트에 태워 품질 통제 가능. 실시간 LLM 리스크 없음
- **판정이 아니라 통역**: 회사를 판정하지 않고 공고의 요구를 사람 말로 풀어줄 뿐 — D9 "구조화된 사전 점검" 포지셔닝과 일치. 확률·합격 예측 표현 금지는 동일 적용
- **매칭대화에이전트 설계와의 관계**: 그 문서의 직무 ②(통역-요건)를 대화가 아닌 정적 조언으로 출력하는 것. 직무 ①(수사관 대화)은 도입하지 않음. 대화에이전트 설계 문서는 후순위 설계로 존치
- **잃는 것과 열린 결정**: 조언만으로는 `원문 확인 필요` 더미가 저절로 줄지 않는다(사용자의 자기 판단이 시스템에 남지 않음). 보완책으로 조언 아래 자가응답 원탭(`우리 회사가 해당돼요 / 해당되지 않아요`, 자가신고 라벨 + 되돌리기)을 붙이는 변형이 있다 — §6 Q8에서 결정

## 6. 열린 결정 (함께 검토할 것)

- **Q1. fitScore % 노출 유지?** 제안은 접힘 카드에서 제거하고 펼침에서만 표시. 숫자 하나가 "조건 확인도"라는 개념 학습을 요구하는 비용 vs 신뢰 기여를 저울질 필요. D9 게이트와도 연동되는 문제
- **Q2. 랜딩 SocialProof 실공고 카드 3장** — 유지(실데이터 신뢰) vs 기관 로고 밴드로 대체(밀도 최소). 제안은 간소화 유지 쪽
- **Q3. 질문 UI의 형태** — ✅ **해소(2026-07-14 사용자 결정, 같은 날 2차 개정)**: 분류 내(22축) 결측은 폼 질문 카드로 묻고, 분류 외 세부 조건은 **묻지 않고 공고별 AI 조언으로 안내**(§3.6). 채팅은 폐기
- **Q4. Roadmap 전용 페이지 존폐** — 시간축 관점(곧 열림/준비하면 열림)은 결과 화면의 탭/필터로 흡수 가능해 보이나, 별도 페이지의 고유 가치가 있는지 사용 데이터 확인 필요
- **Q5. applications 8단계** — 운영·CS 관점에서 8단계 구분이 필요한가? 사용자 표면은 3그룹으로 줄이더라도 내부 모델 유지가 전제
- **Q6. Phase A의 착수 시점** — WS-D.1과 한 몸으로 갈지(권장), 매칭 정확도 트랙과 분리된 UI 트랙으로 병렬화할지
- **Q7. `preparable`(준비하면 열려요)의 지위** — 제안은 접힌 섹션으로 강등(§2 보조 분류). 그러나 "인증 하나 따면 5천만 원짜리가 열린다"는 준비축이 제품 가치의 축이라면 판정 뱃지 5번째로 승격할 수도 있음. 접힌 섹션(단순) vs 5번째 뱃지(가치 노출)의 저울질
- **Q8. AI 조언의 자가응답 원탭 채택 여부** — §3.6의 두 변형: ①순수 조언(문장만 — 결정 지점 0개 추가, 대신 `원문 확인 필요` 더미가 안 줄어듦) vs ②조언+자가응답(`해당되지 않아요` 원탭으로 카드 정리 — 자가신고 라벨·되돌리기 필수). 목업 브리프에 두 변형 나란히 포함 — Claude Design 결과를 보고 결정
