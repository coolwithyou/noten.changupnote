# CODEF 트랙 착수 전 외부 대조 (관문 공통 의례)

> 2026-07-11. CALIBRATION-TEMPLATE 절차. 대상 관문: CODEF 간편인증(L1) 스파이크 착수 전.
> 직전 대조(같은 트랙 근거): `docs/research/2026-07-11-codef-field-sourcing-scenario.md`(같은 날 실측 조사) — 중복 조사 금지, **계약 확정에 필요한 최신 교차검증만** 수행.

## 범위 결정 (감독자)

- **기술 축만 수행.** 제품·규제 축(AI 작성물 정책·경쟁 지형)은 CODEF 데이터 소싱 커넥터 스파이크와 무관 → 필드테스트 전 대조 소관. 07-04·07-10 선례(짧은 델타 시 외부 축 생략)와 동일 판단.
- 근거 문서가 **같은 날(2026-07-11) developer.codef.io 실측**이므로, 재조사 대신 **Phase A가 하드코딩하는 계약(엔드포인트·파라미터·응답 인코딩·2-way 프로토콜)을 독립 소스로 교차검증**하는 표적 대조로 한정.
- developer.codef.io 개발가이드는 Vue SPA라 WebFetch 렌더 불가 → **CODEF 공식 오픈소스 SDK(codef-node/easycodef-*)** 를 독립 원천으로 사용(계약이 코드에 박혀 있음).

## 대조할 설계 전제 (Phase A가 딛는 결정)

1. 토큰: `https://oauth.codef.io/oauth/token`, Basic 인증, `grant_type=client_credentials`, accessToken 7일 → DB 캐시
2. 요청/응답 전문은 **URL-encoded JSON**(양방향 decode/encode 필요)
3. 2-way: 1차 → `CF-03002`+`continue2Way` → `jobIndex`/`threadIndex`/`jti`/`twoWayTimestamp` 보관 → `is2Way=true`+이전 body 재요청
4. `loginType="5"`(회원 간편인증), `id` 파라미터로 **1회 인증 세션 다건 SSO**
5. 엔드포인트 base: demo `https://development.codef.io` / prod `https://api.codef.io`

## 발견 (출처)

- **codef-node SDK 실측**(https://github.com/codef-io/codef-node README):
  - 토큰: `tokenUrl='https://oauth.codef.io/oauth/token'`, `Authorization: Basic base64(id:secret)`, body `grant_type=client_credentials&scope=read`, 응답 `access_token`·`token_type:"bearer"`·**`expires_in: 604799`(≈7일)** ✔ 전제 1 확정
  - 요청 body: **`request.write(urlencode.encode(JSON.stringify(body)))`** — JSON을 URL-encode해 전송
  - 응답 body: **`urlencode.decode(body)`** 후 처리 — decode 선행 ✔ 전제 2 확정(양방향)
  - 상품 요청: `POST`, 헤더 `Content-Type: application/json` + `Authorization: Bearer <token>`
- **easycodef-* SDK + 교차검색**(github.com/codef-io/easycodef-java, easycodef-node, itnote.site 사례):
  - `CF-03002` = 추가인증 필요 → 이전 request body + `jti`·`twoWayTimestamp`(및 `jobIndex`/`threadIndex`/`continue2Way`) 재요청 ✔ 전제 3 확정
  - `loginType="5"` = 간편인증 ✔ 전제 4(부분: loginType) 확정
- 엔드포인트 base(dev/prod) 및 상품 경로: 시나리오 문서 §0/§2 실측 그대로, SDK 검색 결과와 모순 없음 ✔ 전제 5 유지

## 전제별 대조표

| # | 전제 | 판정 | 근거 |
|---|---|---|---|
| 1 | 토큰 7일·oauth.codef.io·Basic·client_credentials | **유지** | codef-node `expires_in:604799` 실측 |
| 2 | 요청/응답 URL-encoded JSON(양방향) | **유지·보강** | codef-node가 요청 `urlencode.encode`, 응답 `urlencode.decode` — **Phase A `client.ts`에서 반드시 양방향 처리**(이 한 줄이 없으면 전 응답 파싱 실패) |
| 3 | 2-way: CF-03002→jti/twoWayTimestamp+이전body+is2Way | **유지** | easycodef SDK·교차검색 일치 |
| 4 | loginType="5" 간편인증, id 세션 SSO | **유지(부분)** | loginType="5" 확정. **id 세션 SSO는 벤더 주장 — D1 실측 대상**(스파이크 가정1 자체) |
| 5 | base dev/prod 분기 | **유지** | 모순 없음 |

## 판정 요약

- **재고(재설계) 필요 지점 없음.** Phase A 계약을 그대로 구현하되, 전제 2(양방향 URL 인코딩)를 client.ts 구현·A7 fixture 테스트의 **명시 검증 항목**으로 승격.
- 외부 대조로 **해소 불가**한 3대 가정(①id 세션 SSO 실동작 ②정식 단가 ③간이/면세 개인 매출 커버리지)은 설계대로 **D1 사용자 동반 실측이 유일 경로** — 스파이크의 존재 이유. 대조가 이를 앞당기지 못함을 확인.
- 벤더 주장 vs 실측 구분: "id 1회 인증 다건 처리"·"부가세과세표준 개인 커버리지"는 developer.codef.io **문서 주장**이며 미검증. 데모 키 라이브 런에서만 확정.
