# ops 공고 관제 대시보드 기획 — 집계 캔버스 + 트리아지 큐 하이브리드 (v2)

> 2026-07-23. 상태: 기획(v2) — 구현 미착수.
> **개정 이력**: v1은 공고 1건=노드 1개인 전면 노드캔버스였다. 검토 결과
> ① 그래프가 읽기 전용이고 모든 공고의 토폴로지가 동일해 배선에 정보가 없다
> ② 트리아지(정렬·키보드 이동·일괄 처리)는 표가 캔버스보다 빠르고 밀도가 3~4배 높다
> ③ v1의 최난도 엔지니어링(200노드 성능·페이지 스폰·자동 접기)은 전부 캔버스 선택이
> 만든 자기유발 문제다 — 라는 판정으로 **하이브리드(Airflow의 DAG 뷰+그리드 뷰 병존
> 패턴)** 로 개정했다. 요청 원문(원천 소스→공고 노드, 검수 상태·HWP 수·데모·22축
> 기대값의 시각적 확인)의 과업은 전부 유지하되, 형태를 과업에 맞춘다.

## 1. 목적과 위치

- **무엇**: apps/admin(changupnote-ops)에 공고 운영의 메인 도구를 신설한다.
  구성은 2층 — **상단 집계 캔버스**(소스→파이프라인 버킷의 고정 노드 그래프, 관제탑)와
  **하단 트리아지 큐**(밀도 높은 테이블 + 상세 Sheet, 작업대).
- **왜 지금**: admin에는 공고 화면이 전무하다(크레딧·라이브매칭·명단 운영만,
  `apps/admin/src/components/app-sidebar.tsx`). 공고 상태는 DB 5개 계층에 흩어져 있어
  "이 공고 지금 어떤 상태인가"에 SQL 없이 답할 수 없다. 딥분석 실험(analysis-lab)이
  GO되면 계층이 하나 더 늘어난다 — 그 전에 운영 단일 화면이 필요하다.
- **아닌 것**: 매칭 결과 대시보드(`/internal/live-match`)·사용자향 화면·analysis-lab
  (dev 실험 도구)의 대체가 아니다. 다만 §9의 확장 슬롯으로 딥분석 상태를 흡수한다.

### 1.1 형태 원칙 — 80% 표, 20% 캔버스

- **캔버스가 잘하는 것**: 파이프라인 전체의 건강 상태를 한눈에(어느 단계에 얼마나
  쌓였고 어디가 빨갛나). 노드 ≤15개의 고정 그래프라 성능·레이아웃 문제가 없다.
- **표가 잘하는 것**: 오늘 처리할 N건의 정렬·필터·키보드 순회·일괄 액션. 관리자는
  하루의 대부분을 여기서 보낸다.
- **금지**: 공고 1건을 캔버스 노드로 스폰하지 않는다. 첨부·서식을 와이어 달린
  서브노드로 전개하지 않는다(같은 정보를 Sheet 탭이 더 빨리 보여준다 — v1 §5.1의
  ④⑤ 폐기).

## 2. 사용자와 핵심 과업 (Jobs to be Done)

사용자는 관리자(운영자) 1~2인. 각 과업의 담당 층:

| # | 과업 | 지금의 방법 | 어디서 해결 |
|---|---|---|---|
| J1 | 오늘 **관리자 확인이 필요한 공고**가 몇 건이고 무엇인가 | SQL/없음 | 캔버스 주황·빨강 배지 → 클릭 시 큐에 로드 |
| J2 | 특정 공고가 파이프라인 **어느 단계에 멈춰 있나** | grant_raw.status 조회 | 캔버스 버킷 위치 + 큐 "단계" 열 |
| J3 | 이 공고의 **첨부 HWP 몇 개, 변환은 됐나** | attachment 테이블 조인 | 큐 `HWP 3(1⚠)` 열 + Sheet 첨부 탭 |
| J4 | 이 공고가 **22축에서 무엇을 기대하나** | grant_criteria 조회 | 큐 22축 미니그리드 셀 + Sheet 22축 탭 |
| J5 | 실제 사용자에게 **어떻게 보이나** (데모) | 웹에서 검색 | Sheet 데모 탭 |
| J6 | 확인 후 **처리**(검수 승인·재변환·재추출) | 스크립트 수동 실행 | 큐 행/일괄 액션 + Sheet 액션 바 |

## 3. 데이터 현황 — 상태는 5계층의 합성이다

노드·큐 상태 모델의 근거. 단일 원천은 `apps/web/src/lib/server/db/schema.ts`.

| 계층 | 테이블·컬럼 | 값 |
|---|---|---|
| L1 수집 파이프라인 | `grant_raw.status` | fetched → converted → extracted → normalized → published / **failed** |
| L2 공고 정본 | `grants.status` | upcoming / open / closed / unknown (+ applyEnd로 D-day) |
| L3 첨부 변환 | `grant_attachment_archives.conversionStatus` (+품질 `apps/conversion/src/types.ts`) | queued / converted / failed, 품질 usable / usable_with_review / **manual_required** / failed |
| L4 서식 추출 | `grant_application_surfaces.extractionStatus` | pending / preview_ready / fields_ready / **failed** |
| L5 criteria 검수 | `grant_criteria.needsReview`(축별 행), `extraction_log.status`(auto/review/labeled) | 검수 필요 여부·건수 |

- 소스 3종: `grantSourceEnum` = kstartup / bizinfo(기업마당) / bizinfo_event.
- 공고↔첨부는 FK가 아니라 `(source, sourceId)` 자연키 조인.
- 22축 단일 원천: `packages/contracts/src/enums.ts` `CRITERION_DIMENSIONS`,
  한국어 라벨: `apps/web/src/lib/server/analysis-lab/diff.ts` `DIMENSION_LABELS`.
- admin은 Drizzle 미공유, raw SQL(`apps/admin/src/lib/server/db/client.ts`)
  → 집계 쿼리는 admin 측 신규 작성.

### 3.1 파생 "관리 상태" (managementState) — 화면 전체의 1급 어휘

관리자가 보는 색은 L1~L5 원시 상태가 아니라 그 **합성 판정** 하나다.
우선순위 상위부터 (먼저 걸리는 것이 승자):

| 상태 | 색(토큰 의미) | 판정 조건 (합성 규칙) |
|---|---|---|
| `failed` 실패 | 빨강 | L1=failed **또는** L3 품질=failed **또는** L4=failed |
| `needs_admin` 관리자 확인 필요 | 주황 | L3 품질=manual_required **또는** L5 needsReview>0 **또는** extraction_log=review |
| `auto_reviewable` 자동 검수 가능 | 파랑 | L1=published이고 needsReview=0이지만 검수 이력 없음 (딥분석 GO 후: AI 검수 실행 대상) |
| `in_pipeline` 진행 중 | 회색-파랑 | L1이 fetched~normalized (자동 진행 대기) |
| `ok` 정상 | 초록 | L1=published + 검수 완료(labeled/approved) + 첨부 전부 usable |
| `closed` 마감 | 회색(감쇠) | L2=closed — 기본 필터에서 숨김 |

합성 규칙은 **서버 한 곳**(신규 `apps/admin/src/lib/server/admin/pipelineGraph.ts`)에서만
계산하고, 캔버스·큐·Sheet는 결과 enum만 소비한다. 규칙이 바뀌어도 UI 불변.

### 3.2 착수 전 실측 (P0의 첫 작업 — 설계 가정 검증)

v1의 "needs_admin은 통상 수십 건" 가정은 실측 없이 깔린 것이다. 구현 착수 전에
아래를 1회 실측해 기획 수치를 갱신한다. 결과가 가정과 크게 다르면(예: needs_admin
수백 건) 큐 기본 필터·정렬 우선순위를 재조정한다:

- 관리 상태 6종별 공고 수 (open+upcoming 기준)
- `grant_criteria.needsReview=true` 보유 공고 수·행 수 분포
- 첨부 변환 실패·manual_required 공고 수
- open 공고 실카운트 (문서 실측치는 ≈948건,
  `docs/grant-document-draft-autofill-implementation-plan.md` L7)

## 5. 집계 캔버스 (관제탑) — 노드 ≤15개 고정

### 5.1 노드

**① 소스 노드** (3개: K-Startup / 기업마당 / 기업마당 이벤트)
- 본문: 최근 수집 시각, 오늘 신규 n, 누적 open n.
- 수집이 오래 멈췄으면(임계 초과) 헤더에 경고 점 — 수집 파이프라인 자체의 헬스체크.

**② 버킷 노드** (렌즈에 따라 4~6개, §5.3)
- 헤더: 버킷 이름 + 상태 색 점. 본문: 총 카운트(큰 숫자) + 소스별 분해(작은 줄).
- ⚠/✖ 카운트가 0이 아니면 헤더에 주황/빨강 배지 — J1의 시각 신호.
- **클릭 = 하단 큐에 해당 버킷 로드** (노드 스폰 아님 — v1과의 결정적 차이).
- 선택된 버킷은 하이라이트 링 — 캔버스와 큐의 연동 상태를 항상 표시.

### 5.2 와이어

- 소스→버킷, 버킷→버킷(파이프라인 순서). 소속·흐름만 표현하는 베지어 1홉.
- 와이어 두께/불투명도로 유량(공고 수) 감각 전달은 선택 사항(P4) — 1차는 균일선.

### 5.3 렌즈 (버킷 축 전환)

| 렌즈 | 버킷 축 | 기본 사용 |
|---|---|---|
| **검수 렌즈** (기본) | 관리 상태 6종(§3.1) | 아침 트리아지 — J1 |
| 파이프라인 렌즈 | grant_raw.status 6단계 | 수집·변환 배치 모니터링 — J2 |
| 마감 렌즈 | D-day 버킷(오늘/3일/7일/30일/이후) | 마감 임박 우선 처리 |

렌즈 전환은 버킷 노드 재배치만 — 큐의 필터·선택 상태는 유지.

### 5.4 구현 — React Flow를 기본 도입하지 않는다

- 노드 ≤15개·고정 좌표·팬/줌 불요(한 화면에 다 들어옴) → **자체 구현이 기본안**:
  절대 배치된 shadcn Card 노드 + SVG `<path>` 와이어. 신규 의존성 0, 토큰·컴포넌트
  통제 완전.
- `@xyflow/react`는 **공고 레벨 그래프가 실제로 필요해지는 시점에만** 도입(현재로선
  근거 없음). v1에서 전제였던 도입을 명시적으로 철회.

## 6. 트리아지 큐 (작업대) — 이 도구의 무게중심

### 6.1 열 구성

| 열 | 내용 | 근거 데이터 |
|---|---|---|
| ● | 관리 상태 색 점 (툴팁: 합성 사유 — "criteria 검수 2건") | §3.1 |
| 제목 | 1줄 말줄임, 클릭=Sheet | grants.title |
| 소스 | 배지 | grants.source |
| D-day | D-n, 3일 이하 강조 | grants.applyEnd |
| HWP | `3 (1⚠)` — 첨부 수(문제 수) | attachment 집계 |
| 22축 | **미니그리드 셀**: 22칸 도트 스트립(11×2). 초록=채움·주황=needsReview·빈칸=공란. 도트 호버 시 `지역 ∈ [서울, 경기]` 식 툴팁 | grant_criteria 집계 |
| 검수 | needsReview 행 수 | grant_criteria |
| 단계 | L1 파이프라인 단계 | grant_raw.status |

### 6.2 동작

- **정렬**: 기본 D-day 오름차순. 열 헤더 정렬(D-day/검수 건수/첨부 문제 수).
- **필터**: 상단 바와 공유(소스·상태 칩·검색어) + 버킷 선택. 필터 상태는
  URL 쿼리에 반영(공유·복원 가능 — v1 D5의 localStorage 대체).
- **키보드 트리아지**: `j/k` 행 이동, `Enter` Sheet 열기/닫기, `x` 선택 토글,
  `a` 주 액션(상태별 — 주황이면 검수 승인), `r` 재변환. 캔버스에선 불가능했던,
  하이브리드 전환의 실질 이득.
- **일괄 액션**: 다중 선택 시 하단 고정 바(선택 n건 · 검수 승인 · 재변환).
- **페이지**: 50행/페이지. 가상화는 실측(§3.2) 결과 상시 수백 행이 확인될 때만 도입.
- 구현은 shadcn data-table 패턴(`@tanstack/react-table` — 정렬·선택·페이지네이션
  표준 제공). admin에 `table` 프리미티브 설치돼 있음.

## 7. 상세 패널 (행 클릭 → 우측 Sheet)

큐는 요약까지, 판단 근거는 Sheet에서. shadcn `sheet`(설치돼 있음) 사용.

- **헤더**: 제목 · 소스 배지 · 관리 상태 배지 · 접수기간(D-day) · 기관 ·
  sourceId(복사 버튼).
- **탭 1 — 22축 기대값** (J4): 22행 테이블.
  `축(한국어 라벨) | 종류(필수/우대/배제) | 연산자 | 기대값 | 확신도 | 근거 원문(rawText) | 검수 플래그`.
  needsReview 행은 주황 배경 + `[검수 완료로 표시]`. 공란 축은 감쇠 표시
  ("요구 없음"과 "추출 누락"의 구분은 딥분석 편입 후 — §9).
- **탭 2 — 첨부·변환** (J3): 첨부 전건 목록(파일명·형식·bytes·변환 상태·품질),
  markdown 미리보기 링크, 실패 오류 메시지, `[재변환 요청]`(개별/전체).
- **탭 3 — 데모** (J5): 사용자향 노출 미리보기. 1차는 **새 탭 링크**(매칭 카드·캘린더
  노출 지점), iframe 임베드는 보류(열린 결정 D2).
- **탭 4 — 이력**: grant_raw 타임스탬프, 변환 시각, parserVersion/modelVer,
  extraction_log·golden_set 승격 여부.
- **하단 고정 액션 바**: 상태별 주 액션 1개 강조. Sheet를 연 채 `j/k`로 다음 행 이동
  가능(연속 검수 흐름).

## 8. 기술 설계

### 8.1 데이터 API (admin 신규, raw SQL)

서버 로직: `apps/admin/src/lib/server/admin/pipelineGraph.ts`
(관리 상태 합성 규칙 §3.1 포함 — 단일 함수).

| 엔드포인트 | 반환 | 용도 |
|---|---|---|
| `GET /api/admin/pipeline/summary` | 렌즈별·소스별 버킷 카운트 + 소스 헬스(최근 수집 시각) | 캔버스 1콜, 60초 폴링 |
| `GET /api/admin/pipeline/notices?lens=&bucket=&source=&q=&sort=&cursor=` | 큐 행 레코드 50건(제목·D-day·관리상태·HWP n(⚠n)·22축 도트 배열·검수 n·단계) | 큐 로드 |
| `GET /api/admin/pipeline/notices/[source]/[sourceId]` | 상세(criteria 22축 전행·첨부 전건·surface·이력) | Sheet 지연 로드 |
| `POST /api/admin/pipeline/actions` | `{action: mark_reviewed\|reconvert\|reextract, targets[]}` | 액션 (단건·일괄 공용) |

집계 쿼리 골격: `grants` ⋈ `grant_raw`(source,sourceId) ⋈ `grant_attachment_archives`
집계 ⋈ `grant_criteria` 집계(축 채움 수·needsReview 수). 기본 스캔은 open+upcoming,
closed는 필터 옵션. 인덱스는 기존(`grants_source_status_idx`,
`grant_criteria_review_idx`)으로 1차 충분 — §3.2 실측과 함께 확인.

### 8.2 신규 의존성

- `@tanstack/react-table` (shadcn data-table 표준 패턴) — 큐.
- 그래프 라이브러리 **없음** (§5.4). v1 대비 의존성 부담이 React Flow → TanStack
  Table로 교체된 셈이며, 후자가 더 가볍고 이 도구의 무게중심(표)에 직결된다.

### 8.3 액션 실행 경로

- `mark_reviewed`: `grant_criteria.needsReview=false` + extraction_log 기록.
  비파괴적 — confirm 1단으로 충분.
- `reconvert`: 변환 서버 큐 API 호출 — **변환 서버 Cloud Run 재배포가 선행 잔여 작업**
  (hwpx 트랙 핸드오프 참조). 재배포 전에는 버튼 비활성 + 사유 툴팁.
- `reextract`: 추출 파이프라인 재큐잉의 API화 — 허용 범위는 열린 결정 D3.

## 9. 딥분석(analysis-lab) 편입 슬롯 — 지금 만들지 않지만 자리를 판다

analysis-lab 실험이 GO(`docs/plans/HANDOFF-2026-07-23.md`) 되면 공고 측 criteria

  AI 검수 통과 / 감사 대기 / 확정. → `auto_reviewable`이 "AI 검수 실행 가능"으로
  실체화된다(현재는 자리만).
- Sheet 탭 1에 **A/B diff 열**(현행 criteria vs 딥분석 결과) — analysis-lab의
  `DimensionDiffTable` 패턴 이식.
- 액션에 `run_deep_analysis`(비용 발생 — 배치 확인 다이얼로그 필수, 공고당
  $0.11~0.45 실측).
- 합성 규칙(§3.1)이 서버 단일 함수이므로 L6(분석 계층) 추가는 프론트 무변경으로

## 10. 구현 로드맵 — 데이터 → 큐 → 액션 → 캔버스

v1의 "캔버스 먼저"를 뒤집는다. 운영 가치가 가장 빨리 나오는 순서:

| Phase | 내용 | 완료 기준 |
|---|---|---|
| **P0 데이터 레이어** | §3.2 실측(가정 검증·기획 수치 갱신) → `pipelineGraph.ts` 합성 규칙 + summary/notices API | 실DB로 관리 상태 6종 카운트가 API로 나옴 |
| **P1 트리아지 큐 + Sheet** | `/pipeline` 라우트, 큐 테이블(열 8종·정렬·필터·URL 쿼리)·Sheet 4탭(읽기 전용) | J1~J5가 화면에서 완결(읽기) — 이 시점부터 운영 투입 가능 |
| **P2 액션** | mark_reviewed(단건→일괄) → 키보드 트리아지 → reconvert(변환서버 재배포 후) | 주황 1건을 화면에서 처리해 초록 전환 확인 — J6 |
| **P3 집계 캔버스** | 자체 SVG 관제탑(소스+버킷 노드·와이어·경고 배지·버킷 클릭→큐 연동), 검수 렌즈 | 캔버스에서 버킷 클릭 → 큐 로드 왕복 동작 |
| **P4 렌즈·부가** | 파이프라인/마감 렌즈, 캔버스 접기, 검색 점프, 와이어 유량 표현 | 렌즈 전환 시 큐 상태 유지 |

- P0 0.5세션, P1 1~1.5세션, P2 0.5~1세션, P3 1세션 규모 추정. **P1 완료 시점에
  이미 쓸 수 있는 도구**가 나온다는 것이 v1 대비 핵심 개선.
- 구현 시 shadcn 스킬 로드(프로젝트 UI 규칙) + admin components.json(base-nova) 준수.
- 드리프트 스캔 규칙 적용: 캔버스 노드·큐 셀에도 raw `<button>` 금지 → shadcn Button.
