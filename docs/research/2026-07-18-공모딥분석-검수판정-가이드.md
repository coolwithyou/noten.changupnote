# 공모 딥분석 검수 판정 가이드 (실전)

> 2026-07-18. 대상: `/dev/analysis-lab` 검수 탭에서 코호트 3건을 판정하는 검수자(창업자).
> 화면의 "검수 안내"가 기준 **정의**라면, 이 문서는 **무엇을 위주로, 어떤 순서로, 애매하면 어느 쪽으로** 판정하는지의 실행 규칙이다.
> 판정 원칙은 Gate 1 기준서(`docs/gate1-field-map-labeling-guide.md`)의 원칙을 딥분석 검수에 전용한 것.
> 실런 데이터 기반: 통합공고 run-1217, 식품 해외인증 run-0804, 원스톱 run-1227(최신).

## 0. 단 하나의 리트머스 질문

모든 criterion 판정은 결국 이 질문 하나다:

> **"이 criterion 을 이대로 DB에 넣고 매칭 판정에 썼을 때, 원문과 다른 결론이 나오는 기업이 존재하는가?"**

- 존재하지 않는다 → **정확** (표현이 다소 축약돼도 판정 결과가 같으면 정확이다)
- 존재한다, 그러나 요건 자체는 원문에 실재 → **수정 필요** (무엇을 어떻게 고칠지 사유에)
- 요건 자체가 원문에 없다(창조·오독) → **오류**
- 원문·첨부만으로 확정 불가 → **판단 불가** (남발 금지 — 정밀도 분모에 포함되는 측정 대상)

검수는 "B(딥분석)가 원문에 맞는가"이지 "A(현행 DB)와 같은가"가 아니다.
**A는 참고일 뿐** — A와 달라도 원문 기준으로 B가 맞으면 정확이고, A가 틀렸을 수도 있다.

## 1. 판정별 무게 — 무엇이 실험 결론을 좌우하나

게이트 5종 중 검수 판정이 직접 움직이는 것은 3개다. 무게 순으로:

1. **정확 vs 수정 필요의 경계** (정확 비율 ≥80% 게이트의 전부).
   여기서 관대하면 실험이 허위 통과하고, 엄격하면 허위 미달한다. §0 리트머스로만 판정하고
   흔들리는 항목은 사유에 근거를 남겨라 — 나중에 경계를 재조정할 수 있는 건 사유뿐이다.
2. **오류(wrong)** (≤10% 게이트, 치명 신호). 원문에 없는 요건을 만들었거나 다른 조건을
   잘못 읽은 경우만. "실재하지만 부적절"은 오류가 아니라 수정 필요다(§2-A).
3. **누락 있음** (공고당 ≤1건 게이트). 빈 축 확인에서 "원문에 그 축의 요건이 실재하는데
   B가 못 잡은" 경우만. 발견하면 반드시 원문 문구 기준으로 서술(필수 입력).

나머지 2개(커버리지·비용)는 자동 계산이라 판정 부담이 없다. 단 커버리지 분자가
"정확 확정" 건수이므로, 결국 1번 경계가 커버리지도 결정한다.

## 2. 경계 규칙 — 애매하면 이쪽으로

Gate 1 기준서의 원칙("근거 명시 우선 — 추정 승격 금지", "안내문 배제", "판단불가는 명시적 판정")을 전용한 결정 규칙:

- **(A) 절차·제재·위임 조항의 요건화** — "서류 미비 시 배제", "허위 기재 시 제재",
  "개별 공고문 참고" 같은 문구를 criterion 으로 제안한 경우:
  문구는 원문에 실재하므로 **오류가 아니라 수정 필요**. 사유에 "자격요건 아님(절차 조항)"
  이라 쓰고, 축·kind 재배정이 필요하면 함께 적는다. 이 유형은 3개 런 모두에 있다(other 축).
- **(B) 인용에 없는 구조값** — 근거 인용에는 없는 수치·기간·목록이 value 에 들어간 경우
  (예: 인용은 "39세 이하"인데 value 는 "19~39세"): 붙임·본문 다른 곳에 근거가 실재하면
  **정확**, 찾을 수 없으면 **수정 필요**(추정 승격 금지). 원문 검색을 먼저 하라.
- **(C) 값 축약·정보 손실** — 원문 예외가 여럿인데 플래그 1개로 축약된 경우 등:
  §0 리트머스 적용. 판정 결과가 달라지는 기업이 있으면 수정 필요, 없으면 정확.
- **(D) 구조화 다운그레이드** — 원문에 수치가 있는데 text_only 로 제안한 경우:
  **수정 필요** (매칭 기계판정 가능률이 이 실험의 존재 이유다).
- **(E) 같은 축 중복 제안** — kind 가 다르면(required+preferred 등) 정상. 같은 취지의
  중복이면 하나만 정확, 나머지는 수정 필요("중복" 사유).
- **(F) 빈 축의 "없음 확인"** — "그 축의 자격요건이 원문 전체에 없다"를 확인하는 것.
  가점(preferred)성 조건이 실재해도 B 가 안 잡았으면 **누락 있음**이다.
- **(G) 판단 불가** — 붙임 파일이 입력에 없거나(변환 실패) 원문이 실제로 모호할 때만.
  "귀찮음"의 도피처가 아니다 — unsure 비율 자체가 품질 지표다.

애매해서 여기 없는 유형을 만나면: 즉석 재량으로 판정하지 말고 **전체 메모에 기록**하고
판단 불가로 두라. 집계 후 이 가이드 §5 사례집에 규칙을 추가하고 재판정하는 것이 순서다
(기준서 단일 원천 원칙).

## 3. 항목당 30초 루틴

1. **근거 인용 읽기** → 원문("공고 원문" 링크·"분석 문서" 탭)에서 실재 확인.
   **"근거 미확인" 배지가 붙었으면 이 단계를 건너뛰지 말 것** — 서버가 인용을 원문에서
   못 찾은 항목이다(오독·창조의 최우선 용의자).
2. **값·연산자 대조** — value 의 모든 구조값(수치·목록·플래그)이 근거 또는 원문 다른
   곳에 실재하는가 (§2-B).
3. **kind 적정성** — 필수/우대/결격이 원문 취지와 맞는가. 절차 조항의 결격화(§2-A),
   부담비율 차등 참여를 필수로 굳힌 경우 등.
4. **축 적정성** — other 남용, 기존 축(biz_age 등)에 갈 것이 other 로 갔는지.
5. 판정 클릭 → 정확이 아니면 **사유에 "무엇이 틀렸고 올바른 값은 무엇"**을 쓴다.
   이 사유가 골든셋의 원재료다.

빈 축(섹션 B)은 축 이름을 보고 원문에서 해당 주제(연령·세금·수출실적…)를 검색해 보는
것으로 충분하다. 딥분석의 축별 검사 코멘트(배지 옆)가 힌트를 준다.

## 4. 공고별 최우선 확인 항목 (실런 데이터 기반)

권장 순서: **원스톱(가볍다) → 식품(본격) → 통합공고(개념 판정)**. 부분 저장을 활용하라.

### 4-1. 원스톱 지원센터 (7 criteria + 빈 축, 최신 런 run-1227…5e4aab)

| 우선 | 항목 | 쟁점 |
|---|---|---|
| ★★★ | #2 industry(필수, 기술창업) | **유일한 "근거 미확인"** — 인용("중소기업창업 지원법 제2조제9호…")을 원문에서 직접 찾아 확인 |
| ★★★ | 빈 축 biz_age | **이 런의 핵심 쟁점.** A(DB)에는 업력 lte 120개월(conf 0.98)이 있는데 B 는 빈 축 + #7 other/text_only(conf 0.5)로 강등. 원문이 수치 상한을 명시하는지 확인 — 명시하면 "누락 있음"(biz_age), 아니면 A 쪽이 구조화 필드 기계 복사인지 확인 |
| ★★ | #7 other(업력 초과 요건검토) | 위와 동전의 양면 — biz_age 로 갔어야 할 요건이면 수정 필요(축 재배정) |
| ★ | #5·#6 other(사적상담 불가·허위 제재) | 절차 조항의 요건화(§2-A) — 자격요건인지 판단 |

### 4-2. 식품 해외인증 (16 criteria + 빈 축, run-0804…)

| 우선 | 항목 | 쟁점 |
|---|---|---|
| ★★★ | #3 size(필수, 중소기업) | conf 0.6 + **본문 부담비율 표는 대·중견기업(40~60%) 참여 허용** — required 로 두면 오배제. 이 런에서 가장 결과가 갈리는 항목 |
| ★★★ | 빈 축 premises·export_performance | 검사 상태가 "조건 발견"인데 제안이 없음 — 클러스터 입주 우선, 수출실적 가점이 실재하면 **누락 있음** 후보 |
| ★★ | #13 founder_age(우대, 19~39세) | 인용은 "39세 이하"만 — 하한 19세가 붙임 가점기준에 실재하는지(§2-B) |
| ★★ | #16 prior_award(우대) | programs·2년 내·완료 등 구조값이 인용에 없음 — 붙임 대조 |
| ★★ | #12 other(BASA 진단 선행) | 절차적 선행조건의 exclusion 화 — kind 적정성(§2-A) |
| ★ | #15 other(국산 농산물 가점 3점) | "3점" 수치 근거 확인 |
| ★ | #7 business_status / #9 credit_status | "미가동"이 closed 로 커버되는지 / 예외 다수의 1플래그 축약(§2-C) |

결격 5건(#5~#9)은 인용이 명확해 빠르게 처리 가능. 정확 후보가 많은 런이다.

### 4-3. 통합공고 (5 criteria + 빈 축 18, run-1217…)

이 런은 개별 항목보다 **"정보성 카탈로그의 요약값을 매칭 요건으로 써도 되는가"**라는
개념 판정이 본질이다. 모델 스스로 cautionNotes 에 "508개 사업 포괄 요약이라 특정 사업
요건과 다를 수 있음"이라 썼다.

| 우선 | 항목 | 쟁점 |
|---|---|---|
| ★★★ | #1 region(필수, 전국)·#2 target_type | 통합공고 요약값을 required 로 굳히면 개별 사업 매칭에 과대적용 — §0 리트머스로 판정하고, 판정 근거를 사유·전체 메모에 상세히 남겨라(통합공고 분해 트랙의 설계 입력이 된다) |
| ★★ | #2 target_type(conf 0.5) | value 9종 vs 인용 7종 — "예비창업자·창업기업" 추가분의 근거(§2-B) |
| ★★ | #3 biz_age(text_only, conf 0.4) | A 는 구조화(lte 120, conf 0.98) — 508개 요약이라 구조화 불가라는 B 판단이 타당한가(§2-C·D의 교차점) |
| ★ | #4·#5 other(개별 공고문 위임 2건) | 절차적 위임의 요건화(§2-A) — 신규 축 제안 delegated_to_subprogram 과 역할 중복 |

**빈 축 18건 주의**: 개별 사업(청년 39세 이하, 관내 소재 등)에만 있는 요건을 "누락"으로
잡지 말 것 — 이 공고 자체의 공통 요건 기준으로만 판정한다. 모델의 축별 코멘트("개별
사업에 존재하나 통합 요건 아님")가 대체로 타당하므로 "없음 확인"이 기본값, 단
founder_age(모호 판정)는 코멘트를 읽고 판단.

## 6. 저장·집계

- 부분 저장 가능 — 공고 하나 끝날 때마다 저장하라. 저장 전 다른 런 선택·분석 실행 시
  초안이 사라질 수 있다(분석 완료 시에는 미저장 보호가 걸려 있다).
- 3건 모두 "검수됨"이 되면: `pnpm lab:aggregate` → 게이트 5종 판정 + **보조 지표
  "기계판정 가능"**(정확 확정 중 구조화 비율, 공고당 기계판정 가능 criteria A→B 배수).
  보조 지표는 게이트가 아니지만 피벗의 근거 진단(기계판정 가능률 ~40% 병목)을 직접
  측정하는 수치다 — §0 리트머스와 §2-D 판정이 이 수치의 품질을 결정한다.
- 소표본(3건) 주의: 판정은 확대 실험(층화 30~100건) 착수 근거이지 최종 결론이 아니다.
