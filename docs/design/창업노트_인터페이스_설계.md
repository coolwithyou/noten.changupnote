# 창업노트 — 인터페이스 & 클라이언트 설계

작성일: 2026-06-26 · 역할: 두 클라이언트(웹/앱)의 화면·인터랙션·API 표면 계약.
기준: `사용자플로우_로드맵` + `매칭스코어_모델` + `레포전략_배포토폴로지` + `정규화스키마_설계` + `DB스키마_통합`.

> 이 문서는 "무엇을 보여주나(화면)"와 "무엇을 주고받나(API)"를 한 곳으로 묶는다. 데이터 진실의 원천은 `grant_criteria`·`match_state`·`rule_trace`이고, 인터페이스는 그 위의 **표현 계약**이다.

---

## 0. 두 설계 축 (제품 차별화의 전부)

| 축 | 한 줄 | 거부하는 것 |
|---|---|---|
| **① 토스식 첫 화면 훅** | 큰 숫자 하나 + 단일 입력(사업자번호) + 즉각 개인화 카운트업 | 회원가입 먼저, 장황한 설명, 폼 더미 |
| **② 목록 패러다임 탈피** | "기회 맵"으로 직관적 확인 + "신청 준비 시트"로 *왜·얼마·언제·어떻게·뭐가 필요*를 한 화면에 | 리스트 전수검토 후 선택하는 올드 UX |
| **③ 게임화 월드 모델** | 지원사업=퀘스트(조건 충족→활성화), 기업=성장 RPG(레벨·업적·투자단계). 디아블로식 "요구 충족 시 해금" | 무미건조한 공공기관 톤, 공허한 포인트·랭킹 |

이 세 축이 클라이언트 차별화의 거의 전부다. 나머지 화면은 이 셋을 떠받친다.

## 0.5 게임화 월드 모델 (퀘스트 × 성장 RPG × 살아있는 월드) ★

> 축 ②(목록 탈피)의 *세계관*. 게임처럼 보이되 모든 요소가 실데이터에 1:1로 묶여 정직하다(원칙 #2·#7). 메타포가 기존 아키텍처(파이프라인·`match_state`·알림)에 공짜로 올라탄다.

### 두 레이어
| 레이어 | 게임 언어 | 데이터 |
|---|---|---|
| **퀘스트 = 지원사업** | 조건 충족 → 활성화, 미충족 → 잠금 | `grants`·`grant_criteria` / `rule_trace` |
| **성장 RPG = 기업** | 캐릭터 스탯·레벨·업적·투자단계(예비→Seed→A→B→C) | `company`·`company_profiles` |

### 카드(퀘스트) 상태 = `rule_trace`로 계산
- **적격(활성)**: 필수 전부 pass → "바로 신청". *확정 광택은 진짜 적격만*.
- **조건부(확인 필요)**: unknown 존재 → progressive로 활성화 유도.
- **잠금(준비하면 적격)**: 미충족 필수 → "벤처확인 취득 시 / Series A 도달 시 해금".

### 살아있는 월드 (양방향·시간성)
- **스폰/디스폰** = 수집 파이프라인(신규 `open`, 마감 `closed`). 월드가 매일 갱신.
- **열림 ↔ 닫힘 양방향**: 업력↑로 *열리는* 카드 / 업력 초과·마감으로 *닫히는* 카드. **"곧 사라지는 전설 퀘스트" = 가장 강력하고 정직한 긴급 장치**.
- **상태 전환 트리거 3종**: ①시간(업력) ②자격 획득(인증·투자단계·선정이력) ③progressive 입력.

### 재방문 루프 = 상태 전환 diff 알림
자리 비운 사이 변한 월드를 diff로: "새 퀘스트 N개 활성화 / 1개 곧 닫힘(D-3) / 곧 해금". 알림 = 리텐션 심장(사용자플로우 §6에 "곧 닫힘·새로 해금" 추가).

### 게이미피케이션 가드레일
- 보상은 *진짜*(자금·해금·성장)에 묶기. **가짜 XP·전체 랭킹(사업자 민감) 금지**.
- 톤 = 성장 RPG(진중·성취) ≠ 캐주얼게임(유치).
- 업적·투자단계 = **노튼 "기업 성장 데이터 = AI 자산"의 entry point** + 플라이휠 outcome. 재미·데이터·비전이 한 몸.

### 시간 기반 재매칭 (엔진) — 매칭 §9.6
업력은 매일 변하지만 *매일 전수 재평가는 비효율*. → **전이 날짜 사전계산**: 매칭 시 `eligible_from`/`eligible_until`을 산출해 `match_state`에 저장, 가벼운 일일 잡이 *그날 전이되는 건만* flip + 알림. 조회 시점엔 현재 날짜로 표시 보정. 프로필 변경(인증 등)은 이벤트 기반 재매칭.

## 1. 신뢰 계약 ↔ 후킹의 균형 (불가침)

토스식 후킹은 설계 7원칙 #2(**거짓확신 금지**)와 충돌할 수 있다. 숫자는 매력적이되 정직해야 한다.

- **확정/추정 분리**: 헤드라인 금액은 항상 **"최대 ₩X (적격 N건 기준)"**. 조건부 금액은 별도 줄("확인 시 +₩Y").
- **카운트업 연출은 허용, 숫자 과장은 금지**. 연출은 표현, 숫자는 사실.
- **부적격 숨김 금지(recall)**: 첫 화면은 긍정 우선이되, 부적격은 *접되 사유와 함께 펼침 가능*.
- **unknown = "확인 필요"**, 선정가능성·가치 = "추정" 라벨. (매칭 §5 UX 계약 준수)

> 후킹 ≠ 과장. 공공 임무라 신뢰가 자산이다(원칙 #7).

## 2. 웹/앱 API 표면 분리 (별개 설계)

도메인 로직(`packages/core`)은 공유, **API 표면은 별개**. 분기하는 것은 인증·응답 포장·버전 정책뿐.

| 축 | **웹 API (BFF)** | **앱 API** |
|---|---|---|
| 인증 | NextAuth 세션(httpOnly 쿠키, SameSite) | access+refresh **JWT**(Bearer) |
| 표면 | `/api/web/*` Route Handler + **Server Actions** | 버전드 REST `/api/app/v1/*` |
| 응답 | **화면 단위 aggregation**(over-fetch OK, SSR) | **리소스 단위 슬림** + cursor 페이지네이션 |
| 계약 | 내부용, TS 타입 직접 공유 | **OpenAPI → Dart codegen**(외부 계약) |
| 진화 | 화면과 함께 자유롭게 | 버전 호환 보장(앱스토어 지연 → 구버전 장기 공존) |
| 엔벨로프 | `ActionResult<T>` / RSC shape | `{ data, meta, error }` 표준 |

```
   [웹]                              [Flutter 앱]
    │ 세션 쿠키                          │ Bearer JWT
 /api/web/* + Actions            /api/app/v1/*
 (BFF·aggregation)               (버전드·리소스·codegen)
    └───────────┬────────────────────────┘
        requireCompanyAccess(가드, core 단일)
                │
          packages/core use-case
                │
          repository → Supabase PG (RLS 이중방어)
```

### 2-A. 웹 표면 (화면 지향)
| 화면 | 패턴 | 경로 / 액션 | 인증 |
|---|---|---|---|
| 첫 화면(집계훅) | RSC loader | `GET /api/web/stats` | 익명·캐시 |
| 번호 입력→티저 | Server Action | `startMatch(bizNo)` → teaser | 익명·비저장 |
| 소유권 검증 | Server Action | `verifyOwnership(bizNo, ceoName, openDate)` | 익명→검증 |
| 로그인 | NextAuth | `/api/auth/*` | — |
| 보강 | Server Action | `enrichCompany(companyId)` | 세션 |
| 기회 맵(대시보드) | RSC loader | page-level **aggregation** | 세션 |
| 매칭 필터/정렬 | Route Handler | `GET /api/web/matches?companyId&sort&filter&cursor` | 세션 |
| progressive | RSC + Action | `GET /api/web/next-question` · `submitProfileField(...)` | 세션 |
| 신청 준비 시트 | RSC loader | `/grants/[id]` (apply-sheet 포함) | 세션 |
| 2D 로드맵 | RSC loader | `/roadmap` page | 세션 |
| 피드백 | Server Action | `submitFeedback(...)` | 세션 |
| 회사 전환 | Server Action | `switchCompany(companyId)` | 세션 |
| 알림 설정 | Route Handler | `GET/PUT /api/web/notifications` | 세션 |

웹 응답은 BFF라 화면 단위로 묶는다(예: 대시보드 = `matches[] + companySummary + nextQuestion + counts` 한 번에).

### 2-B. 앱 표면 (`/api/app/v1/*`, 리소스 지향)
**인증**: `POST /auth/login` · `/auth/{provider}`(PKCE code 교환) · `/auth/refresh`(회전) · `/auth/logout`.
**집계/티저**: `GET /stats`(익명) · `POST /teaser`(번호→비저장 1차).
**회사**: `GET /companies` · `POST /companies` · `POST /companies/{id}/verify` · `POST /companies/{id}/enrich` · `GET /companies/{id}/profile`.
**매칭/로드맵**: `GET /companies/{id}/matches?status&sort&cursor` · `GET /companies/{id}/next-question` · `POST /companies/{id}/profile/fields` · `GET /grants/{id}`(apply-sheet) · `GET /companies/{id}/roadmap`.
**피드백/이벤트**: `POST /matches/{companyId}/{grantId}/feedback` · `POST /matches/{companyId}/{grantId}/events`(surfaced/clicked/saved/apply_click).
**알림/기기**: `GET/PUT /notifications/settings` · `POST /devices` · `DELETE /devices/{id}`.

앱 응답은 슬림 리소스 + 표준 엔벨로프 `{ data, meta:{cursor,hasMore,rulesetVer}, error:{code,message,field} }`.

## 3. 화면 1 — 첫 페이지 (집계훅 → 입력 → 개인화)

### 3-A. 입력 전: 익명 집계 훅 (추정 아닌 사실 = 정직)
```
   오늘 바로 신청할 수 있는 지원사업
            1 2 0 개          ← 카운트업
   지금 열려있는 지원금 총 ₩ 4,820,000,000
   ┌─────────────────────────────┐
   │  사업자번호 10자리            │   ← 단일·큰 입력
   └─────────────────────────────┘
        [ 30초 만에 내 지원금 확인 ]
   · 예비창업자세요? 번호 없이 시작 →
```
데이터: `GET /stats` → `{ openCount, totalAmount, deadlineSoonCount }`. `grants WHERE status=open AND apply_end>=today` 집계, Upstash 5분 캐시.

### 3-B. 입력 직후: 개인화 카운트업 (= 1차 티저 재해석)
```
   서울 · 1인 사업자 · 업력 2년     ← 보강 칩(왜 이 숫자인지)
   김대표님이 받을 수 있는 지원금
   최대  ₩ 12,340,000             ← 확정(적격) 합, 카운트업
   ✅ 지금 적격 8건
   ⏳ 조건부 12건 (확인 시 +₩6,200,000)  ← 추정, 분리 표기
   🔥 마감임박 3건
        [ 무료로 전체 보기 ]  ← 로그인 게이트
```
- 비저장·비로그인. "전체 보기/저장/알림"이 로그인 게이트(플로우 §10).
- 카피 공식: `{지역} {규모} 사업자가 받을 수 있는 지원금 **최대 ₩{확정합}**` + `적격 N · 조건부 M`.

## 4. 화면 2 — 기회 맵 (목록 탈피, 2D 로드맵을 메인으로)

리스트 전수검토를 버리고, 2D 로드맵(시간×조건)을 **메인 결과 화면**으로 승격. "지금 적격"을 가장 크게.
```
🟢 지금 받을 수 있어요 (8건)        ← 가로 캐러셀·큰 카드, 가치/마감순
  [청년창업 ₩50,000,000 적합98] [SW바우처 D-5🔥 적합91] [...]
🟡 곧 받을 수 있어요
  ○ 3개월 뒤 업력 3년 → 청창사 (+1)
🔵 준비하면 받을 수 있어요
  ◇ 벤처확인 취득 시 → +5건
  ◇ 매출 입력 시 → 12건 확정/제외 판단   ← progressive 진입점
▸ 아쉽지만 해당 안 돼요 (사유 보기)        ← 부적격 접힘·사유 노출
```
- 버킷 = `now | soon | preparable | conditional`. unknown·미충족 criteria가 그대로 미래 노드(정보이득 동일 소스).
- "곧/준비하면" 노드 탭 → 해당 조건 해소 액션(progressive 입력 or 인증 취득 안내).
- 부적격은 접되 **양방향 프레이밍**: progressive 입력은 "확정 또는 제외에 도움"(매칭 §6).

## 5. 화면 3 — 신청 준비 시트 (한 화면에 모든 것)

공고 하나를 열면 다른 곳 안 가도 신청 준비 끝. *왜·얼마·언제·어떻게·뭐가 필요*가 한 화면.
```
청년창업사관학교 14기
적합도 98 · 최대 ₩50,000,000 · ⏰ 마감 D-5 (7/1 18:00)
───────────────────────────────
✅ 내가 이미 충족한 것          ← rule_trace result=pass
  ✅ 업력 7년 이내 — 2년 2개월
  ✅ 서울 소재
⚠ 확인이 필요한 것             ← result=unknown
  ⚠ 매출 10억 이하 → [지금 입력]  ──▶ progressive
📄 준비해야 할 서류             ← required_documents(신규)
  □ 사업계획서  □ 사업자등록증(보유✓)
  □ 운영사 추천서 ⚠ 원문 확인     ← text_only
───────────────────────────────
[ K-Startup에서 신청하기 ↗ ]  ← 딥링크
[ 🔔 마감 알림 받기 ]
```
- 신청은 **딥링크 + 서류 준비 도움 + 일정 알림**까지(대리신청 ❌, 플로우 §7).
- 카드의 적합/부적합 피드백 → 플라이휠(`feedback`), apply_click → `match_events`.

## 5.5 화면 4 — NPC 액션 큐 ("지금 당장 뭘" = 다음 할 일) ★

> 사용자 3대 니즈의 ③(①뭘 지원 가능=기회 맵 / ②언제까지 뭘=2D 로드맵 / **③지금 당장 뭘=여기**). 디아블로식 **퀘스트 기버**: 모든 퀘스트에 흩어진 행동을 모아 *지금 이거 하나부터*를 제시. **챗봇이 아니라 엔진 산출의 narration**.

- **무엇을 보여주나**: 레버리지·긴급순으로 정렬된 행동 리스트. 각 항목 = 무엇 · **어디서**(딥링크/발급처) · 왜(unlock 건수·금액) · 언제(마감) · 쉬움(소요) · CTA.
- **행동 4종 (결정론적 출처)**:
  - **입력**(unknown 해소) ← conditional 카드의 unknown → "업종 입력 → 3건 활성화"
  - **취득/준비** ← locked 카드의 미충족 자격(인증·투자단계·서류) → "벤처확인 취득 → 5건 해금". 발급처 = `required_documents.source`(cert→발급기관)
  - **신청** ← eligible + 마감 임박 → "D-3 신청", 접수 딥링크
  - **프로필 보강** ← 등록증 업로드 등(정확도↑·서류 재사용)
- **정렬** = 레버리지 × 긴급 ÷ 쉬움 (매칭 §6.1 액션 랭킹). 한 행동이 여러 퀘스트를 열면 합산.
- **정직 가드레일**: 모든 안내는 `rule_trace`·`required_documents`·전이날짜에서 *계산*. 자동 판정 불가(text_only)는 "원문 확인" 행동으로, **절대 날조 금지**. LLM은 말투만(원칙 #2·#3).
- **게임 연결**: 행동 완료 → 퀘스트 활성화/해금 연출(§0.5 살아있는 월드). 액션 큐 = NPC 대사, 기회 맵 = 퀘스트 보드.
- **위치**: 첫 매칭 직후 + 대시보드 상단 고정(재방문 시 "이번 주 할 일" = 리텐션 심장).

## 6. rule_trace → 준비 체크리스트 재프레이밍 (핵심 엔진)

`rule_trace`를 "판정 설명"이 아니라 **"준비 체크리스트"**로 재해석한다. 같은 데이터, 다른 가치.

| `rule_trace.result` | 시트 섹션 | 사용자 메시지 | 액션 |
|---|---|---|---|
| `pass` | ✅ 이미 충족 | "업력 7년 이내 — 2년 2개월" | (안심, 없음) |
| `unknown` | ⚠ 확인 필요 | "매출 10억 이하 확인 필요" | → progressive 입력 |
| `text_only` | 📄 준비 서류 / 원문 확인 | "운영사 추천서 필요" | → 원문 딥링크 |
| `fail`(preferred) | ➖ 미해당(가점) | "비수도권 우대 미해당" | (정보, 없음) |

이 매핑이 축2-B의 데이터 엔진이다. rule_trace 칩에 `checklistSection`·`action`을 부착해 클라이언트가 그대로 렌더.

## 7. 데이터·계약 갭 (✅ 2026-06-26 반영 완료)

두 축을 구현하려면 현재 스키마/계약에 3가지가 빠져 있었고, 각 소유 문서에 반영했다:

| 갭 | 보완 | 반영 위치 |
|---|---|---|
| **집계 훅 숫자** | `GET /stats`(open 수·총 지원금·마감임박, 익명·Upstash 캐시) — 웹/앱 양쪽 | `구현착수_가이드` §2 |
| **개인화 티저 총액** | `/teaser` 응답에 `estimatedMaxAmount`(확정)·`conditionalUpside`(추정)·`counts` 분리 | `구현착수_가이드` §2, 본 문서 §8 `TeaserResult` |
| **필요서류** | `grants.required_documents` 신규 필드 + LLM 추출(criteria 아님, 신청 메타) | `정규화스키마` §2 · `DB스키마` §3 · `수집파이프라인` S3·§5 |

> `required_documents`는 데이터 계약 변경이라 grants 헤더(`정규화스키마`)·`grants` 테이블(`DB스키마`)·추출 단계(`수집파이프라인` S3/§5) 3곳에 일관 반영. **criteria가 아니라 신청 메타**라 매칭 판정에 쓰지 않고, 클라이언트가 `rule_trace`/`company_profiles`와 조인해 "보유 ✓ vs 준비"를 표시한다.

## 8. 핵심 DTO 계약 (`packages/contracts`)

도메인 DTO는 공유(TS 타입 + OpenAPI 동시 생성), 엔벨로프만 표면별 분리.

```ts
interface StatsResult { openCount: number; totalAmount: number; deadlineSoonCount: number }

interface TeaserResult {                       // 비저장, companyId 없음
  attributes: { region: string; size: string; bizAgeMonths: number; industry: string[] }
  estimatedMaxAmount: number                   // 적격(확정) 합
  conditionalUpside: number                    // 조건부 추정 추가분
  counts: { eligible: number; conditional: number; deadlineSoon: number }
}

interface MatchCard {
  grantId: string; title: string; agency: string
  eligibility: 'eligible' | 'conditional' | 'ineligible'
  bucket: 'now' | 'soon' | 'preparable' | 'conditional'   // 기회 맵 버킷
  fitScore: number                              // 적합도 헤드라인(확정)
  competitiveness?: { value: number; estimated: true }     // 선정가능성 v1+
  value?: number
  supportAmount: { min: number; max: number; unit: 'KRW'; per: '기업' | '건' }
  applyEnd: string; dDay: number
  ruleTrace: RuleTraceChip[]
  matchConfidence: number
  rulesetVer: string; scoringVer: string
}

interface RuleTraceChip {
  dimension: string; kind: 'required' | 'preferred' | 'exclusion'
  result: 'pass' | 'fail' | 'unknown' | 'text_only'
  label: string; companyValue?: string; sourceSpan?: string
  checklistSection: 'satisfied' | 'needs_check' | 'document' | 'preferred_miss'
  action?: { type: 'progressive' | 'external_link'; target: string }
}

interface RoadmapNode {
  bucket: 'now' | 'soon' | 'preparable' | 'conditional'
  grantId: string
  unlock?: { dimension: string; kind: 'time' | 'attribute'; detail: string; etaDate?: string }
  deltaCount?: number                           // "+5건"
}

interface ApplySheet {                          // 신청 준비 시트
  grant: GrantDetail
  satisfied: RuleTraceChip[]                    // result=pass
  needsCheck: RuleTraceChip[]                   // result=unknown → progressive
  documents: RequiredDocument[]
  applyMethod: string; deepLink: string
  schedule: { applyStart: string; applyEnd: string; dDay: number }
}

interface RequiredDocument {                     // 저장 스키마(grants.required_documents)와 동일 enum
  name: string; required: boolean
  source: 'self' | 'portal' | 'cert'            // self=자가작성·portal=포털발급·cert=인증기관
  alreadyHave?: boolean                         // company_profiles join 판정(런타임)
  fromTextOnly?: boolean; sourceSpan?: string; note?: string
}

interface NextQuestion {                        // 정보이득 1위
  dimension: string; prompt: string; inputType: 'number' | 'select' | 'boolean'
  options?: string[]
  framing: string                               // "매출 입력 시 12건 확정/제외 판단" (양방향)
  affectedGrantCount: number
}
```
