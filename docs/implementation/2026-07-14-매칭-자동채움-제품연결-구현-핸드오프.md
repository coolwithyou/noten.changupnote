## 1. 이번 재개의 단일 목표

목표는 새 matcher나 새 데이터 프레임워크를 만드는 것이 아니다.

> 이미 만든 service-data typed profile·evidence·matching 기반을 실제 `/matches`, dashboard, web/app API, answer/enrich, background 경로에 연결해, 이후 로직 변경이 즉시 사용자 결과에 반영되는 **하나의 제품 경로**를 완성한다.

`100% 연결`은 값이나 정확도가 100%라는 뜻이 아니라 다음 경로가 빠짐없이 하나의 계약을 쓴다는 뜻이다.

1. 승인된 source/cache/Q&A가 typed observation을 만든다.
2. 접근 context와 consent가 허용한 observation만 한 번 materialize한다.
3. 중앙 resolver가 final `CompanyProfile`을 만든다.
4. 모든 matcher·질문·저장·background consumer가 그 resolver 결과를 사용한다.
5. 사용자 화면에 운영 19축의 값/unknown/partial, 안전한 출처, 기준일, 다음 행동이 나타난다.
6. 답변 직후 같은 응답에서 profile, 매칭 수, 카드, 다음 질문이 함께 갱신된다.
7. 우회 경로가 생기면 CI가 실패한다.

시각 리디자인과 matcher 품질 개선은 이 연결이 완료된 뒤 별도 작업으로 한다.

## 4. 이미 확인된 P1 결함 후보

새 세션은 P1 전체를 다시 설계하지 말고 아래 두 재현을 failing test로 고정해 필요한 만큼만 보정한다.

1. `business_status` 같은 compound 값이 의미상 같아도 객체 key 삽입 순서가 다르면 serialized profile이 달라지는 사례가 있었다.
2. 같은 값/source/asOf의 legacy observation과 versioned observation replay가 동일 결과를 만들지 않는 사례가 있었다.

확인 대상은 `packages/core/src/company/assemble-company-profile.ts`와 기존 P1 tests다.

수용 조건:

- update 순서, DB row 순서, compound key 순서가 final profile/decisions/serialized DTO에 영향을 주지 않는다.
- legacy envelope와 versioned metadata의 존재 여부가 semantic value/evidence 선택을 바꾸지 않는다.
- observation metadata를 일부러 삭제하거나 모든 legacy 값을 새 값으로 재작성하지 않는다.
- 새 generic canonicalization framework를 만들지 않는다.

이 두 결함이 기존 테스트로 이미 안전하다고 증명되면 코드 변경 없이 재현 test와 판단만 남긴다.

## 5. 현재 사용자 경로의 확인된 단절

새 세션은 이 목록을 다시 찾느라 시간을 쓰지 말고 구현 전후 tripwire로 사용한다.

| 사용자 경로 | 현재 우회/단절 |
|---|---|
| `/matches?biz=...` | `MatchResultsExperience`가 `/api/web/teaser`와 client manual profile/localStorage를 사용한다. 새 product resolver와 safe 19축 DTO가 없다. |
| web/app teaser | `resolveTeaserCompanyProfileWithEvidence()`와 route-local legacy merge를 사용한다. |
| anonymous teaser | 사업자번호만으로 저장 profile을 조회하며 owner/user scope를 보장하지 않는다. empty body는 scope 없는 `resolveCompanyProfile({})`를 시도한다. |
| web/app matches | `loadServiceDashboard()`로 repository profile을 직접 matcher에 보낸다. |
| web/app profile answer | `updateCompanyProfileField()` → save → `buildInitialCompanyMatch()`를 route에서 직접 실행한다. 공통 command가 없다. |
| enrich | Popbill 결과를 기존 merge wrapper로 저장·재매칭한다. source policy와 consent context가 중앙화되지 않았다. |
| apply/detail/dashboard/roadmap/applications | 각자 `loadServiceDashboard()` 또는 `loadServiceApplySheet()`를 소비한다. 동일 materialization receipt가 없다. |
| background/state/feedback | repository profile 또는 direct `matchNormalizedGrant()` 경로가 남아 있다. |

명확한 privacy blocker:

- anonymous가 owner row, user answer, consent-derived observation을 읽으면 안 된다.
- empty-body/fallback이 임의 저장 profile로 대체되면 안 된다.
- cache miss가 live/유료 provider 호출의 암묵적 trigger가 되면 안 된다.

이 privacy 경계를 닫기 전에는 branch를 main이나 배포 환경에 올리지 않는다.

## 6. 이번 세션에서 얻은 운영 교훈

### 반드시 지킬 것

1. **사용자 수직 경로를 먼저 만든다.** 내부 계약을 더 확장하기 전에 `/matches`에서 입력 → resolve → match → answer → 갱신이 보여야 한다.
2. **단일 writer가 직접 구현한다.** 기본값은 한 clean worktree의 한 Codex 세션이다.
3. **Orca orchestration DAG를 만들지 않는다.** Orca는 worktree/terminal 관리에만 사용한다. 사용자가 별도로 병렬 감독을 요청할 때만 orchestration을 쓴다.
4. **리뷰는 기능 checkpoint 뒤 한 번만 한다.** 작은 단계마다 새 reviewer를 만들지 않는다. 구현자가 focused test를 통과시킨 뒤 fresh `gpt-5.6-sol` ultra read-only review를 한 번 수행한다.
5. **Fable을 사용하지 않는다.** 이전 모델 라우팅 문구보다 이 결정이 우선한다.
6. **과거 G1~G7 검증을 반복하지 않는다.** 관련 코드가 바뀐 경우에만 해당 회귀를 실행한다.
7. **주요 사용자 기능마다 commit한다.** 지나치게 큰 미커밋 diff와 장시간 세션을 만들지 않는다.
8. **검증을 구현보다 앞세우지 않는다.** failing privacy/behavior test → 구현 → focused test 순서로 진행하고, 전체 검증은 checkpoint에서만 실행한다.
9. **실제 product path 증거와 fixture/dev 증거를 구분한다.** dev monitor green을 `/matches` 완료로 해석하지 않는다.
10. **새 provider, migration, generic engine, UI 리디자인을 금지한다.** 목표에 직접 필요한 최소 코드만 만든다.

### 반복하지 말아야 할 패턴

- 장문 계획을 계속 보강하며 실제 route/UI 구현을 미루는 것
- 동일 diff를 여러 fresh session이 broad audit하는 것
- reviewer transport/모델 문제를 해결하느라 기능 구현보다 오래 쓰는 것
- P2~P4가 비어 있는데 P5/P6 shadow/canary 설계를 먼저 정교화하는 것
- 테스트 수와 문서 분량을 완료율로 착각하는 것
- branch만 존재하는 상태를 실제 사용자 반영으로 보고하는 것

## 7. 최소 구현 순서와 checkpoint

아래 순서는 과거 P0~P6 요구사항의 안전 의미를 유지하면서, 사용자에게 보이는 결과를 앞당기도록 재정렬한 실행 순서다.

### R1 — P1 결정성 결함만 닫기

작업:

- §4의 두 사례를 focused test로 재현한다.
- `assembleCompanyProfile()` 안에서만 필요한 최소 보정을 한다.
- 기존 production merge parity, persistence rollback, profile tripwire를 확인한다.

금지: resolver, route, UI, provider 작업을 함께 시작하지 않는다.

checkpoint 예시:

```text
fix: make product profile assembly replay deterministic
```

### R2 — 최소 product resolver와 privacy 경계

작업:

- `anonymous_teaser`, `owned_read`, `owned_refresh`, `system_recompute` 네 context를 받는 `resolveProductCompanyProfile()`을 만든다.
- 실제 존재하는 producer만 작은 고정 mapping으로 분류한다. 추측한 source를 추가하지 않는다.
- anonymous/other-user/consent-revoked/empty-body negative test를 먼저 만든다.
- live acquisition은 명시적 owner refresh에서만 허용한다.
- owner/consent scope를 기존 schema로 안전하게 표현하지 못하는 source는 `disabled`로 닫는다. migration으로 우회하지 않는다.
- raw payload와 민감 식별자가 없는 `MatchingProfileView` allowlist를 만든다.

checkpoint 예시:

```text
web: add product profile resolver and privacy boundary
```

### R3 — `/matches` 사용자 수직 경로 완성

이 checkpoint가 이번 재개의 첫 사용자 기능 완료점이다.

작업:

- web/app teaser를 같은 resolver에 연결한다.
- web/app matches read를 같은 resolver와 explicit `asOf`에 연결한다.
- `/matches`가 raw `CompanyProfile` 대신 safe view를 소비한다.
- 운영 19축을 `known | partial | unknown`으로 모두 표시한다.
- anonymous answer는 request/session 범위에서 server-side normalize한다. client localStorage를 profile 정본으로 사용하지 않는다.
- 답변 응답 안에서 profile view, 매칭 수/카드, 다음 질문을 함께 반환한다.
- 기존 디자인 구조를 유지하고 기능적 표시/입력만 추가한다.

checkpoint 예시:

```text
web: connect matching and autofill to the matches experience
```

R3 직후 사용자가 이미 실행 중인 서버가 있을 때만 `/matches?biz=7465400870`을 browser smoke한다. 서버가 없으면 실행을 요청하고 Codex가 `next dev`를 시작하지 않는다.

### R4 — 모든 제품 read/write consumer 통합

작업:

- dashboard, roadmap, applications, reports, notifications
- app company preview/create, web/app enrich
- profile field/answer
- grant detail, apply sheet, document generation

모든 경로를 같은 materialization/resolver/answer command 뒤로 옮긴다. response authority를 즉시 typed로 강제할 필요는 없지만, legacy가 필요하면 중앙 adapter 하나에만 둔다. route-local merge와 raw repository profile matcher 입력을 제거한다.

checkpoint 예시:

```text
web: route all product profile reads and commands through the resolver
```

### R5 — background/state와 경로 완전성 CI

작업:

- initial match, ruleset/grant revision refresh, feedback provenance, match-state recompute가 resolved profile을 사용하게 한다.
- company scope와 user overlay를 공용 state에 섞지 않는다.
- 기존 tripwire를 실제 central resolver/command 경로 기준으로 갱신한다.
- 새 matcher dimension 또는 consumer가 contract/UI/question/persistence/product test 중 하나를 빠뜨리면 CI가 실패하게 한다.

checkpoint 예시:

```text
test: enforce matching profile product path completeness
```

### R6 — 외부 proof와 점진 승격

R1~R5가 모두 local green인 뒤에만 시작한다.

- 사용자가 실행 중인 dev/preview 서버를 사용한다.
- 승인된 표본과 권한만 사용한다. 승인된 실표본이 없으면 synthetic 결과로 대체하지 않고 external pending으로 남긴다.
- anonymous/owner/consent revoke/provider failure/empty result를 browser/API에서 검증한다.
- web/app/주요 화면이 같은 company/profile/asOf에서 같은 판정을 보이는지 확인한다.
- production traffic canary와 legacy 제거는 별도 명시 권한과 rollback 증거가 있을 때만 수행한다.

## 8. 제품 연결 완료 체크리스트

다음 항목이 모두 참이기 전에는 “100% 연결 완료”라고 보고하지 않는다.

- [ ] P1 독립 review의 BLOCKER/MAJOR가 0이다.
- [ ] 네 access context가 중앙 resolver에서 강제된다.
- [ ] anonymous/other-user/consent-revoked/empty-body 누출이 0건이다.
- [ ] 승인 source는 consumed/disabled/unavailable 중 명시적 상태이며 `pending`으로 방치되지 않는다.
- [ ] `/matches` web/app teaser와 authenticated matches가 중앙 resolver를 사용한다.
- [ ] answer/enrich가 공통 command와 같은 resolved snapshot을 사용한다.
- [ ] dashboard/apply/detail/background/state 경로의 direct matcher 우회가 없다.
- [ ] 운영 19축이 사용자 화면에 값 상태, 안전한 출처, asOf, completeness, action과 함께 나타난다.
- [ ] 모든 질문 형식이 server-side normalize → resolve → match 경로를 사용한다.
- [ ] 답변 응답에서 profile, 매칭 수/카드, 다음 질문이 함께 갱신된다.
- [ ] owner persistence가 19축/evidence/question state를 손실 없이 round-trip한다.
- [ ] anonymous 답변은 owner/shared persistence를 쓰지 않는다.
- [ ] typed candidate가 허용 전 persistence/state authority가 되지 않는다.
- [ ] raw payload, 대표자명, 연락처, token, 민감 식별자가 DTO/log/snapshot에 없다.
- [ ] product path completeness CI가 우회 경로를 실패시킨다.
- [ ] 사용자 실행 서버에서 주요 web/app/browser smoke가 통과한다.
- [ ] 최종 구현 commit이 `main`에 의도적으로 통합되고 실제 실행/deployment commit이 확인된다.

## 10. commit과 review 규칙

- R1~R5의 주요 기능 단위마다 commit한다.
- staging 전에 `git diff --name-only`로 사용자 research 문서가 포함되지 않았는지 확인한다.
- 구현 중에는 같은 worktree에 reviewer를 병렬로 두지 않는다.
- R3와 R5 완료 뒤 각각 fresh `gpt-5.6-sol` ultra read-only review를 최대 한 번 수행한다.
- review는 substantial BLOCKER/MAJOR와 목표 이탈만 찾는다. 스타일·추상화 취향 MINOR를 이유로 기능 작업을 장시간 중단하지 않는다.
- 수정이 필요하면 같은 writer가 좁게 고치고 focused re-review만 한다.
- reviewer terminal은 완료 즉시 종료한다.

## 11. 명시적 중단 조건

다음 중 하나면 범위를 넓히지 말고 사용자에게 정확한 증거를 보고한다.

- consent/user scope를 기존 schema로 안전하게 표현할 수 없어 migration이 필요함
- 새 유료 provider/계약/env가 필요함
- 현재 `main`의 제품 코드가 대상 branch와 충돌하고 사용자 변경 소유권을 구분할 수 없음
- anonymous privacy negative test를 통과시키지 못함
- required grant universe/base identity가 unavailable인데 sample fallback으로만 진행 가능함
- 사용자 실행 서버나 승인 실표본이 없어 external proof를 만들 수 없음
- production canary/배포/traffic 변경 권한이 없음

이 경우 local 구현 완료와 external blocker를 분리한다. blocker 때문에 이미 완료한 코드를 되돌리거나 새 아키텍처를 만들지 않는다.
