# 창업노트 — 수집(Ingestion) 파이프라인 상세 설계

작성일: 2026-06-19 · 위치: GCP/Python (Cloud Run + Scheduler + Jobs) · 산출: `grant_raw` → `grants`/`grant_criteria`

## 0. 원칙
1. **단계별 멱등(idempotent)** — 어느 단계든 재실행해도 중복/오염 없음. 상태머신으로 진행.
2. **증분 우선, 전수 정합은 주기적.** 매일 변경분만, 주기적으로 전체 대조(마감·삭제 반영).
3. **추출은 비싸다 → 변경분만.** 변환·추출 결과 캐시. backfill은 의도적 트리거.
4. **실패 격리.** 한 공고 실패가 전체를 막지 않음. dead-letter + 부분 발행(가진 만큼 + needs_review).
5. **recall 우선.** 애매하면 숨기지 말고 노출 + 플래그(공공 디스커버리 목적).

## 1. 소스 어댑터 (3종 → 공통 raw)

| 소스 | 호출 | 키 | 특징 |
|---|---|---|---|
| K-Startup 공고 | `https://nidapi.k-startup.go.kr/api/kisedKstartupService/v1/getAnnouncementInformation` page·perPage=100·returnType=json | KSTARTUP_SERVICE_KEY | S0-A 완료: 지역·업력·연령·분야·대상·일정 구조화. LLM은 scoped 텍스트 보강 |
| 기업마당 지원사업 | `bizinfoApi.do` dataType=json | BIZINFO_SERVICE_KEY | 자격요건 `bsnsSumryCn`(HTML)+첨부 HWP |
| 기업마당 행사 | `bizinfoEventApi.do` | 동일 | 게이팅 약함, `areaNm` 구조화 |

각 어댑터가 소스별 필드명을 **공통 raw 스키마**(source, source_id, title, body_html, structured_fields{}, attachments[], updated_at)로 정규화. 소스 스키마 드리프트 대비 **스키마 검증 + 알림**.

---

## 2. 파이프라인 스테이지 (상태머신)

```
grant_raw.status:  fetched → converted → extracted → normalized → published
                          ↘ failed(스테이지별, dead-letter)
```

**S1. Fetch (Cloud Run Job, 스케줄)**
- 소스 페이징 순회 → 목록. `source_cursor`에 페이징/마지막 수집시각 저장.
- **변경감지**: 항목 `updated_at`/`raw_hash` 비교 → 신규·변경만 통과(미변경 skip).
- 첨부(HWP/PDF) → **R2 저장**(파일 콘텐츠 해시를 키로, 멱등). `attachments[]` ref 기록.
- `grant_raw` upsert(raw payload+해시+collected_at). status=fetched. 큐에 enqueue.

**S2. Convert (워커)**
- 첨부 HWP→MD: **pyhwp `hwp5html` → 마크다운**(S0-B 검증 — `hwp5txt`는 표 손실, `hwp5html`는 표 보존). PDF→MD(pdfplumber). **표 추출 우선**(자격요건이 표인 경우 많음). 스캔형 PDF/HWP는 OCR 또는 `needs_review`.
- 변환 결과는 **첨부 해시로 캐시**(재변환 금지). 스캔 PDF 등 변환 불가 → `needs_review` 플래그.
- status=converted.

**S3. Extract (LLM 워커)**
- K-Startup: 구조화 필드 파서가 지역·업력·연령·분야·대상·일정 criteria를 먼저 생성한다. `aply_trgt_ctnt`/`aply_excl_trgt_ctnt`의 업종·규모·인증·지역배제(예: 수도권 제외)만 scoped LLM/규칙 보강 대상으로 보낸다.
- 기업마당: 입력 조립 = 소스 구조화 필드 + 본문(`bsnsSumryCn` 정제) + 변환 첨부 MD.
- Claude **구조화 출력(tool use)** → `grant_criteria[]` + 정규화 후보값 + **필드별 confidence + source_span**. `model_ver/prompt_ver` 태깅.
- **신청 메타도 동시 추출**: `required_documents[]`(제출서류 목록 — criteria 아님, grants 헤더로). `{name, required:bool, source:self|portal|cert, source_span}`. "신청 준비 시트"용(인터페이스 §5). `source_span` 필수, 없으면 빈 배열.
- 골든셋에서 few-shot 주입. status=extracted. → `extraction_log` 기록.

**S4. Normalize / Resolve (워커)**
- 추출값 → canonical 매핑(`industry_taxonomy`·`region_hierarchy`·`size_thresholds`). `f_*` 투영 산출. **embedding(pgvector)** 생성.
- status=normalized.

**S5. Upsert / Publish (트랜잭션)**
- `grants`·`grant_criteria` upsert(키: source+source_id). 저신뢰/`needs_review` → **라벨링 큐**.
- status=published → 서빙에 노출.

---

## 3. 증분 · 변경감지 · 상태 수명주기

- **변경감지 단위**: `raw_hash` diff. 자격요건 영향 필드 변경 → 재추출, 조회수만 변경 → skip.
- **수명주기**: `apply_start/end` + 소스 존재 여부로 `upcoming|open|closed` 도출.
- **삭제/마감**: 주기적 전수 대조에서 소스에서 사라진 공고 → `closed/archived`(죽은 사업 추천 방지 = 신선도).

---

## 4. 중복 제거 (Entity Resolution)

K-Startup ↔ 기업마당이 같은 사업을 각각 공고 → 중복 추천 위험.
- **dedup 키(보수적)**: 정규화 제목 + 수행기관 + 신청기간 근접 → 후보 클러스터. 고신뢰만 `canonical_grant_id`로 묶고, 애매하면 **둘 다 노출**(recall 우선, 숨기지 않음).
- 원본은 둘 다 보존, 표시만 대표 1건. 불확실 → 라벨링 큐.

---

## 5. 추출 계약 (LLM)

- **출력 스키마 = `grant_criteria` 미러링 + `required_documents[]`(신청 메타)**, **tool-use 강제**(S0-C1 검증: raw 텍스트 JSON은 ~1/3 파싱 실패, tool-use는 0). `operator`/`dimension`은 **enum 고정**, value는 dimension별 형태 고정(일관성). `required_documents`는 criteria 판정에 쓰지 않으며 표시 전용.
- 규칙: **controlled vocab만**(업종·지역·인증), `source_span` 필수, **없으면 null(날조 금지: 업력 미명시→생성 안 함)**, 모호하면 `needs_review:true`, 필수/우대/배제 `kind` 판정.
- 골든셋 예시를 few-shot으로 → 플라이휠로 품질 상승.
- 표 중심 첨부는 표-인지 2차 패스 옵션.

---

## 6. Backfill (재추출)

- 트리거: `prompt_ver`/`model_ver`/taxonomy 갱신 시.
- **재크롤 없이** R2의 원본·캐시 MD로 S3~S5만 재실행(Cloud Run Jobs 일괄).
- 신·구 버전 `eval_runs` 비교로 회귀 확인 후 활성화. 비싼 작업이라 게이트.

---

## 7. 신뢰성

- **큐(Upstash Redis 단일)**: 스테이지 핸드오프, 작업단위=공고. 멱등키 = source+source_id+raw_hash. Pub/Sub 병용은 MVP 범위에서 제외.
- **재시도/백오프**: API(기존 4회·1.5s 백오프 계승), LLM 타임아웃·레이트리밋 → 지수백오프, N회 후 dead-letter.
- **부분 발행**: 추출 일부 실패해도 가진 필드로 발행 + `needs_review`.

---

## 8. 관측 · 비용 · 레이트리밋

- **메트릭**: 수집 fetched/new/changed, 변환 실패율, 추출 성공·평균 confidence·저신뢰율, dead-letter 수, LLM 비용·지연.
- **LLM 추적**: Langfuse(프롬프트·토큰·실패).
- **비용통제**: 변경분만 추출 + 변환 캐시 + 배치. backfill만 의도적 대량.
- **레이트리밋**: K-Startup/기업마당 API 쿼터(perPage=100, 일일), LLM 동시성 캡.

## 10. 리스크
- **dedup 정확도** — 보수적 시작(고신뢰만 병합), 과병합이 더 위험(사업 숨김).
- **소스 스키마 드리프트** — 어댑터 + 검증 + 알림.
- **첨부 다양성**(hwp/hwpx/pdf/스캔) — 스캔은 OCR 필요, 불가 시 needs_review.
- **LLM 처리량·비용** — backfill이 비용 피크, 게이트 필수.
- **신선도** — 마감/삭제 미반영 시 죽은 사업 추천.

## 11. 미결정
- 표-인지 2차 추출 패스 도입 여부(첨부 표 비중 보고 결정).
- dedup 임계값·알고리즘(제목 유사도 메트릭).

## 12. 확정 반영
- K-Startup 신엔드포인트·실응답 필드는 S0-A로 확정. 구 `apis.data.go.kr/...getAnnouncementInformation01`는 사용하지 않는다.
- K-Startup은 파서 우선, 기업마당은 첨부/본문 변환+LLM 우선의 서로 다른 추출 프로파일로 운영한다.
