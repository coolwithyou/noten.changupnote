## 3. 기준서로 판정이 안 되는 애매 케이스 (판정 사례집 추가 후보)

### 3-A. "제출 서류 목록" 안내표를 필드로 셀 것인가 (doc31 p1, doc10 첨부서류, doc29 첨부서류)
기준서: "순수 안내문/유의사항은 필드 아님. 단 '~를 첨부하시오' 지시는 file 필드로 셈."
- doc10/doc29는 첨부 항목이 8~10개인 체크 목록. 각 항목을 **개별 file 필드**로 셀지, **1개 file 필드로 통합**할지 기준서에 없음. 본 파일럿은 **통합 1개**로 처리(notes에 명시).
- doc31 p1은 별도 문서(표지 성격)에 서류 목록만 있고 뒤 양식(양식1~3)이 실제 기입란 → p1 목록은 필드화하지 않음. 그러나 ⑤~⑧(재무제표/완납증명서/4대보험/수출실적증명서)은 뒤 양식에 대응칸이 없어 **누락 위험**.
- **제안 규칙**: "첨부 목록은 항목당 file 필드 1개로 센다. 단 동일 안내 블록 내 세부 나열(예: 증빙서류 3종)은 상위 1개로 묶는다." — 검수자 합의 필요.

### 3-B. 대분류가 있는 복수선택 체크박스를 몇 개로 셀 것인가 (doc02 해당분야, doc23 현재/개발 주력분야, doc29 담보구분)
기준서: "체크박스 그룹 = 필드 1개."
- doc02 "해당분야선택"은 유통/생산자/개인/제조업 **4개 대분류**, 각 대분류마다 하위 옵션 4~5개. 이를 **1개 checkbox**로 볼지 **대분류별 4개**로 볼지 애매. 본 파일럿은 기준서 문구대로 **1개**로 통합.
- doc23 "현재 주력분야"/"개발 추진분야"도 커넥티드카/자율주행차/제어기및OS/보안솔루션 4대분류 구조. 동일하게 각 1개로 처리.
- **제안 규칙**: "옵션이 계층 구조(대분류→하위)면 대분류 단위로 필드를 나눈다"를 추가할지 검토. 자동채움 매핑 관점에선 대분류별 분리가 유리할 수 있음.

### 3-C. 주민등록번호 기입란의 manual 판정 (doc23 총괄책임자, doc29 법인(주민)번호)
기준서 manual 무조건 조건: "신분증/통장 사본 등 개인정보 원본 요구." — 주민번호를 **직접 타이핑**하는 칸은 '원본 요구'와 다름.
- doc23 총괄책임자 인적사항 표에 "주민등록번호 -*******" 칸 존재. 서명·첨부는 아니나 민감정보라 보수적으로 **manual=true** 처리(notes 명시).
- doc29 "법인(주민)번호"는 법인번호 또는 주민번호 겸용란 → 개인사업자면 주민번호. 역시 보수적 manual=true.
- **제안 규칙**: "주민등록번호/고유식별정보 기입란은 manual=true (개인정보 원본 성격)"을 manual 조건에 명문화할지 결정 필요. 현재 문구로는 false로 읽힐 소지가 커 라벨러 간 불일치 예상.

### 3-D. 신청기업/컨설팅기업 배타 작성 서식의 중복 필드 (doc23)
doc23은 참여유형(신청기업/컨설팅기업) 택1에 따라 서식1(신청기업, p3~8) **또는** 서식2(컨설팅기업, p9~16) 중 하나만 작성. 그러나 두 서식이 물리적으로 다 존재.
- 본 파일럿은 **양쪽 다 라벨**(company_name 등 같은 key가 p1/p3/p9/p14에 4회 반복).
- **제안 규칙**: (a) coverage 계산 시 배타 서식을 어떻게 셀지, (b) 동일 key가 한 문서에 여러 번 나올 때 recall/agreement 매칭을 인스턴스 단위로 할지 key 단위로 할지 — 기준서에 매칭 단위 정의 필요.

### 3-E. 한 셀에 여러 값이 들어가는 복합 칸 (doc10 대표자(생년월일), doc23 총괄책임자 묶음, doc31 회사명 한글/영문)
기준서: "작성내용 셀 1개 = 필드 1개" vs "한 문장에 빈칸 2개면 필드 2개."
- doc10 "대표자(생년월일)" = 대표자명+생년월일 한 셀. doc31 "회사명 (한글)(영문)" = 2줄 1항목. doc23 총괄책임자 = 성명/부서/직위/전화/메일 여러 칸이 한 논리 블록.
- 본 파일럿은 **셀/항목 단위 1개**로 통합. 하지만 "빈칸 2개=2필드" 규칙과 충돌 소지.
- **제안 규칙**: "한 셀 안에 명확히 구분된 라벨(예: 성명/생년월일)이 병기되면 각 1필드. 단 물리적으로 한 입력칸이면 1필드." 경계 명확화 필요.

### 3-F. 표 안에 서술 입력이 섞인 복합 셀 (doc23 컨설팅전문분야 p11, 컨설팅실적상세 p13; doc31 동의표 p4)
- doc23 p11 "컨설팅 전문 분야" 표는 체크박스 + 투입인력 성명/자격증 서술이 한 표에 혼재 → checkbox·text 분리가 애매. 본 파일럿은 checkbox 그룹 1개 + personnel/cert 별도로 쪼갬(불확실).
- doc31 p4 동의표는 반복행 table이면서 각 행에 동의 체크 + 서명 → table 1개(manual)로 처리했으나 checkbox/signature 분해 여지.
- **제안 규칙**: "표 셀 내부에 체크·서명·서술이 혼재하면 우세 유형으로 표 전체를 1 필드 처리하고 notes에 혼재 명시" 정도로 단순화 검토.

## 4. 기준서 개선 제안 (요약)

1. **첨부 목록 카운팅 규칙** 명문화 (3-A). 항목당 file인지 통합인지.
2. **계층형 체크박스** 카운팅 규칙 (3-B). 대분류 분리 여부.
3. **주민번호/고유식별정보 기입란 → manual=true** 명시 (3-C). 현재 문구 모호.
4. **동일 key 중복·배타 서식**의 매칭 단위 정의 (3-D). recall/agreement 지표 안정성 직결.
5. **복합 셀(한 셀 다값)** 분리 기준 (3-E).
6. **표+체크+서명 혼재 셀** 처리 규칙 (3-F).
7. **`applicantFills: false` 사례 부재**: 이번 5개 문서엔 기관 기입란(공고번호/접수번호)이 없었다. 다른 문서에서 등장 시 대비해 예시를 사례집에 남길 것.
8. **`required` 판정 근거 로그**: 별표/필수 명시가 거의 없어 대부분 '통념상 미기재 시 접수 불가'로 판단. 라벨러별 편차 큼 → 문서 유형별 required 디폴트(예: 기업명/대표자/연락처/사업자번호는 항상 required) 가이드 추가 제안.

## 5. 소요 감각과 55개 전체 예상

| doc | 페이지 | 필드 | 체감 난이도 |
|-----|-------|------|-----------|
| doc02 | 5 | 29 | 중 (1페이지 밀집 표 + 서술) |
| doc10 | 6 | 39 | 중 (표 위주, 명확) |
| doc23 | 16 | 75 | 상 (배타 서식·중복 key·복합표 최다) |
| doc29 | 3 | 26 | 중상 (동의 항목 밀도 극도로 높음) |
| doc31 | 6 | 34 | 중 (전형적 신청서+전시 양식) |

- 이미지 판독 기반 사전 라벨 체감: 단순 신청서(doc10/doc31)는 문서당 판독+구조화 **20~25분**, 밀집·복합 문서(doc23)는 **50~60분** 수준. 기준서 예상(20~40분)과 대체로 부합하나 **doc23류 대형 사업계획서 결합 문서는 상한 초과**.
- **55개 전체 예상**: 평균 30~35분/문서 가정 시 순수 라벨링 **28~32시간**. 여기에 교차검증 20%(11개 재라벨) + 애매 케이스 합의 시간 포함하면 기준서의 "1인 4~5일"은 **타이트**하며, doc23급 대형 문서 비중에 따라 **5~6일**로 볼 여지. bbox를 정밀 측정(픽셀 실측)까지 하면 추가 소요.
- **bbox 정확도 주의**: 본 파일럿 bbox는 육안 추정으로 오차가 있다(특히 doc23 p1처럼 행이 촘촘한 표). 정밀도가 필요하면 라벨 도구/좌표 실측 단계가 별도로 필요.
