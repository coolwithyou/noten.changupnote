작성일: 2026-06-25 · 기준 문서: `창업노트_검수가이드.md` · 목적: 구현 착수 전 문서 간 모순 제거와 누락 보완

## 1. 검수 범위

읽은 자료:
- 설계 문서 전부: 인덱스, 공고 API 분석, S0 PoC 결과, 정규화 스키마, 데이터 표준화, DB 스키마, 매칭, 수집 파이프라인, 플라이휠, 계정/권한/개인정보, 사용자 플로우, 스택, 레포 전략, 외부연동, 상세구현계획, 구현착수 가이드, 검수가이드.
- PoC 코드: `poc/kstartup_normalizer_poc.py`, `poc/kstartup_match_demo.py`, `poc/llm_extract_poc.py`.
- 샘플: `samples/kstartup_announcement_sample.json`, `samples/bizinfo_hwp_converted.md`.
- 과거 인수인계 DOCX: `창업노트_보완버전_연동API_목록.docx`.

참고 제한:
- 원본 제안 PDF는 이미지 기반 PDF라 현재 환경의 텍스트 추출로 내용 확인이 거의 불가능했다. 이번 검수의 근거는 텍스트화된 설계 문서와 PoC 산출물이다.

## 2. 검수 결론

전체 방향은 검수가이드 Part 0의 원칙과 대체로 맞다. 특히 정규화 `grant_criteria` 계약, `unknown/text_only`, 규칙 판정과 LLM 추출 분리, 플라이휠 버전 귀속은 구현 가능한 형태로 설계돼 있다.

다만 구현 착수 문서 일부에 과거 결정이 남아 있어, 그대로 개발하면 팀이 잘못된 엔드포인트·큐 구성·토큰 저장 방식으로 시작할 수 있었다. 이번 개선은 구현 계획의 범위를 키우기보다, 이미 확정된 결정을 실제 착수 문서까지 일관되게 전파하는 데 집중했다.

| High | Flutter 앱 refresh token 회전/폐기를 위한 서버 상태 저장소가 DB 스키마에 없었다. | `app_refresh_tokens` 테이블을 추가하고 계정/권한 문서와 구현착수 API 표면에 반영. |
| High | 팝빌 재과금 방지 캐시를 `company_profiles`에 직접 저장한다고 되어 있어 원 조회 캐시와 매칭용 canonical 투영이 섞였다. | `company_enrichment_cache`를 추가하고 `company_profiles`는 canonical 투영으로 명확화. |

| Medium | `match_state`와 `match_events` 분리가 일부 문서에서 다시 섞여 있었다. | DB·스택·매칭 문서에서 현재 상태와 행동 이력 역할을 분리 표기. |

| Low | `packages/shared`와 `packages/contracts` 명칭이 섞여 있었다. | 구현착수 가이드에서 `packages/contracts`로 통일. |

## 4. 문서별 판정

| 문서군 | 판정 | 비고 |
|---|---|---|
| 계약·데이터 기반 | 조건부 통과 | 스키마·표준화는 구현 가능. 골든셋 50~100건 손매핑 전에는 정확도 완료 주장 금지. |
| 핵심 엔진 | 조건부 통과 | PoC로 K-Startup 척추 매칭은 실증. 기업마당 LLM 추출은 S0-C1 정성 검증 상태라 P/R 필요. |
| 교차·고위험 | 조건부 통과 | RLS·동의·토큰 원장은 보완됨. 법무/보안의 신용정보법·위탁·인가 검증은 별도 필요. |
| 아키텍처·실행 | 통과 | 레포·배포·큐·도메인·마이그레이션 결정은 구현 착수 가능 수준으로 정리됨. |

## 5. 남은 게이트

- **S0-C2 골든셋 P/R**: 도메인 전문가 50~100건 라벨링으로 추출/매칭 precision·recall 측정.
- **S0-D 팝빌**: 테스트계정으로 실제 필드 커버리지, 건당 비용, 조회결과 저장/캐시 약관 확인.
- **팝빌 런타임**: Node SDK `checkBizInfo`가 Vercel route에서 IP 제한 설정과 함께 안정 동작하는지 확인. 실패 시 GCP 고정 egress 경유.
- **S0-E dedup**: K-Startup↔기업마당 중복률과 보수적 병합 임계값 실측.
- **법무/보안 검토**: PASS, 팝빌, CODEF 위탁 고지·동의·파기·접근로그와 BFF/RLS 우회 경로 확인.

## 6. 구현 착수 권고

1. T0에서 DB 마이그레이션·RLS·`app_refresh_tokens`·`company_enrichment_cache`·계약 스키마를 먼저 고정한다.
2. T1/T4는 K-Startup 파서와 PoC 매칭 엔진을 기반으로 빠르게 세운다. 이 구간은 LLM 없이도 핵심 가치 검증이 가능하다.
3. T2 기업마당은 `hwp5html → md → Claude tool-use` 경로로 붙이되, S0-C2 평가 하니스 없이 품질 개선 배포를 자동화하지 않는다.
4. T7 플라이휠 로깅은 첫 매칭부터 켠다. 후행으로 소급하기 어렵다.
