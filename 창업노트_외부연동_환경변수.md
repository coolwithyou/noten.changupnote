## 2. 데이터 소스 결정 (확정)
- **기업 기본정보 = 팝빌 기업정보조회로 고정.** moneypin 등 **제3자 크롤링 배제** — 봇 차단(Vercel checkpoint 403)·ToS 위반·추정치 부정확·신용정보법/저작권 리스크.
- 매출·고용 = 추정 크롤링 금지, **v1에서 홈택스·4대보험 동의연동(팝빌/CODEF)**으로 정확히.
- 팝빌 가입: **기업정보조회 + 사업자등록상태조회** 선택(홈택스수집은 Phase 2). 회사정보는 운영주체(노튼/바톤) 명의, 테스트 계정부터.
- **팝빌 기업정보 캐시 정책**(재과금 차단): 원 조회 결과는 `company_enrichment_cache(provider,biz_no,scope)`에 저장하고, `company_profiles`에는 매칭용 canonical 필드만 투영한다. 동일 사업자번호·동일 scope 조회는 캐시 사용. TTL 권고 — *기업정보(업종·주소·설립일)는 변동이 느려 30~90일*, *상태조회(휴폐업)는 7일급 + 신청 직전 재확인*. 캐시 만료 시에만 재호출. ⚠ 팝빌 약관의 조회결과 저장·캐싱 허용범위 확인(기업정보는 공개정보성이라 대체로 가능).

## 2.1 팝빌 기업정보조회 구현 계약 (문서 확인 반영)
Node SDK 기준 호출:
```
bizInfoCheckService.checkBizInfo(CorpNum, CheckCorpNum, UserID, success, error)
```
- `CorpNum`: 팝빌회원 사업자번호 10자리(`-` 제외). 운영주체의 사업자번호이며 조회 대상이 아니다.
- `CheckCorpNum`: 조회할 사업자번호 10자리.
- `UserID`: 팝빌회원 아이디. 선택값이지만 운영 추적을 위해 env로 명시한다.
- 성공 응답은 `BizCheckInfo`, 실패는 `PopbillException(code,message)`로 분리된다. 성공 응답 내부에도 `result/resultMessage`, `closeDownState`, `closeDownTaxType` 등 상태 필드가 있으므로 단순 HTTP 성공만으로 보강 성공 처리하지 않는다.

### BizCheckInfo → canonical 투영
| 팝빌 필드 | 용도 | 창업노트 반영 |
|---|---|---|
| `corpNum` | 사업자번호 | `companies.biz_no` 대조 |
| `corpName` | 상호 | `companies.name`, `company_profiles.dimension=name` 후보 |
| `CEOName` | 대표자명 | 검증/표시 최소 사용. PII로 취급, 장기 저장은 필요성 검토 |
| `personCorpCode` | 법인/개인 코드 | `companies.legal_type` |
| `corpScaleCode` | 기업규모 코드 | `company_profiles.dimension=size` 1차값. 매출/고용 기준 확정 전 confidence 중간 |
| `industryCode` | KSIC 10차 산업코드 | `company_profiles.dimension=industry`의 KSIC backbone |
| `bizClass`, `bizType` | 업종/업태 텍스트 | KSIC 보조·정책태그 확장 입력 |
| `establishDate`, `establishCode` | 설립일/설립구분 | `biz_age` 계산 입력. 개인사업자 개업일 대체 가능성은 S0-D에서 확인 |
| `headOfficeCode`, `workPlaceCode` | 본점/지점, 사업장 구분 | 지역 criteria의 본사/지사/공장 인정 판단 보조 |
| `addr`, `addrDetail`, `zipCode`, `addrCode` | 주소 | 행안부 도로명주소 API로 시도+시군구 canonical 변환 |
| `closeDownState`, `closeDownStateDate` | 휴폐업 상태/일자 | `company_profiles.dimension=business_status` 및 신청 직전 재확인 |
| `closeDownTaxType`, `closeDownTaxTypeDate` | 과세유형/전환일 | 표시·검증 보조. 매칭 hard criterion으로 쓰기 전 법무/도메인 검토 |
| `checkDT`, `result`, `resultMessage` | 조회시각/상태 | `company_enrichment_cache` 메타, TTL·오류 분석 |

### 구현 주의
- SDK 설정은 `LinkID`, `SecretKey`, `IsTest`, `IPRestrictOnOff`, `UseStaticIP`, `UseLocalTimeYN`를 가진다. Vercel 서버리스는 고정 egress IP가 보장되지 않으므로 `IPRestrictOnOff`/`UseStaticIP` 조합을 테스트에서 확인하고, 문제가 있으면 팝빌 호출을 고정 egress가 가능한 런타임(GCP Cloud Run 등)으로 옮긴다.
- SDK 예시는 callback 기반이므로 Next.js route에서는 Promise wrapper로 감싸고 timeout·retry·idempotency를 명시한다.
- `result=100` 성공, 오류는 음수 코드+메시지 형태로 기록한다. 실패도 `company_enrichment_cache`에 `last_error` 메타로 저장해 반복 과금/반복 실패를 막는다.

## 6. S0-D 로컬 검증 명령
`POPBILL_API_KEY`만으로는 호출이 불가능하다. `POPBILL_LINK_ID`, `POPBILL_CORP_NUM`, `POPBILL_CHECK_CORP_NUM`을 함께 설정한 뒤 아래를 실행한다.
```
npm exec --package=popbill@1.64.2 -- node poc/popbill_checkbizinfo_probe.mjs
```
출력에는 원문 기업정보를 찍지 않고, 매칭에 필요한 주요 필드의 존재 여부와 상태 코드만 표시한다.
