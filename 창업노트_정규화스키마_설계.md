# 창업노트 — 정규화 grant 스키마 설계 (두 레이어의 계약)

작성일: 2026-06-19 · 역할: 수집(채운다)·매칭(읽는다)·플라이휠(점수·버전 단다)이 공유하는 단일 계약

---

## 1. 설계 원칙

1. **하이브리드 구조** — ㉠ 빠른 후보 필터용 *denormalized 투영 컬럼* + ㉡ 정밀 판정·설명용 *구조화 자격요건 리스트(`grant_criteria`)*. ㉡가 진실의 원천, ㉠은 거기서 파생된 인덱스.
2. **필수 / 우대 / 배제를 1급 구분** — 못 맞추면 탈락(required), 가점(preferred), 해당하면 탈락(exclusion).
3. **결측은 fail이 아니라 unknown** — 기업 데이터·공고 둘 중 하나라도 비면 "조건부 가능"으로. 절대 거짓탈락 금지.
4. **모든 값에 출처·신뢰도·버전** — `source_span`(원문 인용), `confidence`, `model_ver/prompt_ver`. 설명가능성과 라벨링의 토대.
5. **혜택(지원금)과 자격(조건)을 분리** — 지원금액은 필터 조건이 아니라 표시·랭킹 정보.

---

## 2. `grants` — 공고 본체 (헤더 + 투영)

```
grants
─ 식별/메타
  id (pk)            source (kstartup|bizinfo|…)   source_id
  title              url                            attachments[] (R2 ref)
  agency_관할        agency_수행
  category_대        category_중
─ 일정/혜택/신청 (표시·랭킹용, 조건 아님)
  apply_start  apply_end  apply_method  status(open|upcoming|closed)
  support_amount {min, max, unit:'KRW', per:'기업|건'}
  required_documents[] {name, required:bool, source:self|portal|cert, source_span}
  benefits[] {family, type, amount?, unit?, quantity?, certainty(확정|최대|선정시), detail, source_span}
  obligations[] {type(자부담|정산|사업기간|중복제한), value, source_span}   -- 혜택의 짝(부담 균형)
─ 후보 필터 투영 (grant_criteria에서 파생, 인덱스/pgvector용)
  f_regions[]            -- 시도코드. [] = 전국
  f_industries[]         -- 표준 업종태그/KSIC
  f_biz_age_min_months   f_biz_age_max_months
  f_sizes[]              -- 예비|소상공인|중소|중견
  f_founder_traits[]     -- 청년|여성|장애인…
  f_required_certs[]     -- 벤처|이노비즈…
  embedding              -- pgvector (의미 후보축소)
─ 수집/추출 메타
  raw_hash  collected_at  overall_confidence  model_ver  prompt_ver
```

> `f_*`는 **읽기 최적화 캐시**다. 매칭 1단계(후보 축소)에서만 쓰고, 최종 적격 판정은 항상 `grant_criteria`로.

> `required_documents`는 **자격요건(criteria)이 아니라 신청 메타**다. 매칭 판정에 쓰지 않고, "신청 준비 시트"(`창업노트_인터페이스_설계.md` §5)에서 *뭐가 필요한지*를 한 화면에 보여주기 위한 표시용. 클라이언트가 `company_profiles`/`rule_trace`와 조인해 "이미 보유 ✓ vs 준비 필요"를 판정한다. 추출 시 `source_span` 필수, 없으면 빈 배열(날조 금지). `source`=self(자가 작성)·portal(포털 발급)·cert(인증기관). text_only criterion에서 파생된 서류는 그쪽 원문 링크와 함께.

## 3. `grant_criteria` — 자격요건 (진실의 원천)

자격요건 한 줄(절)을 한 레코드로. 매칭 엔진이 이걸 순회하며 판정한다.

```
grant_criteria
  id (pk)   grant_id (fk)
  dimension   -- region | biz_age | industry | size | revenue | employees
              -- | founder_trait | certification | prior_award | ip | other
  operator    -- in | not_in | lte | gte | between | exists | text_only
  value       -- jsonb, dimension별 형태 (아래)
  kind        -- required | preferred | exclusion
  weight      -- preferred일 때 점수 기여 (기본 0)
  confidence  -- 0~1 (추출 신뢰도)
  source_span -- 원문 인용("…전남도 소재 중소기업…")
  raw_text    -- 해당 절 원문
  needs_review -- 신뢰도 낮음/모호 → 라벨링 큐
```

### dimension별 `value` 형태
| dimension | operator 예 | value 예 |
|---|---|---|
| region | in / not_in | `{"regions":["전남"]}` |
| biz_age | lte / between | `{"max_months":84}` (=7년 이내) |
| industry | in / not_in | `{"tags":["로봇","제조","SW"]}` |
| size | in | `{"sizes":["중소"]}` |
| revenue | lte / gte | `{"max_krw":12000000000}` |
| employees | between | `{"min":5,"max":50}` |
| founder_trait | in | `{"traits":["청년"]}` |
| certification | exists / in | `{"certs":["벤처"]}` |
| prior_award | not_in (배제) | `{"programs":["TIPS"]}` |
| ip | exists | `{"types":["특허"]}` |
| other | text_only | `{"note":"운영사 추천 필요"}` |

> `text_only`는 자동판정 불가 절 — 매칭에서 **"확인 필요"로 표시**하고 원문 링크로 위임(거짓확신 방지).

## 4. 실제 공고 매핑 검증

### (a) 기업마당 실데이터 — `[전남] 첨단로봇·AI 제조혁신`
원문: "공고일 기준 사업자등록증상 **전남도 소재 중소기업**…로봇 제조·공급·SI·SW… / 기업당 **최대 4백만원**"

```
grants: f_regions=["전남"], f_sizes=["중소"], f_industries=["로봇","제조","SW","SI"],
        support_amount={max:4000000, per:"기업"}, f_biz_age_*=null(명시 없음)
grant_criteria:
  · {dimension:region,   operator:in, value:{regions:["전남"]}, kind:required, conf:0.95, span:"전남도 소재"}
  · {dimension:size,     operator:in, value:{sizes:["중소"]},   kind:required, conf:0.9,  span:"중소기업"}
  · {dimension:industry, operator:in, value:{tags:["로봇","제조","SW","SI"]}, kind:required, conf:0.7, span:"로봇 제조·공급기업, SI·SW…", needs_review:true}
```
→ 업력은 **없으면 안 만든다**(날조 금지). industry는 표현이 느슨해 `needs_review`.

### (b) TIPS형 (구조가 풍부한 공고 예시)
"창업 **7년 이내** 기술기반 스타트업 / 운영사 추천 필수 / 기존 TIPS 선정기업 제외 / 청년·여성 대표 **우대**"

```
grant_criteria:
  · {dimension:biz_age,      operator:lte,    value:{max_months:84}, kind:required, conf:0.95, span:"창업 7년 이내"}
  · {dimension:industry,     operator:in,     value:{tags:["기술기반"]}, kind:required}
  · {dimension:prior_award,  operator:not_in, value:{programs:["TIPS"]}, kind:exclusion, span:"기존 TIPS 선정기업 제외"}
  · {dimension:other,        operator:text_only, value:{note:"운영사 추천 필요"}, kind:required, needs_review:true}
  · {dimension:founder_trait,operator:in,     value:{traits:["청년","여성"]}, kind:preferred, weight:10, span:"청년·여성 대표 우대"}
```
→ required·exclusion·preferred·text_only가 한 공고에 공존. 이 모델이면 다 표현됨.

## 5. 매칭 산출과의 접속 (`match_state` / `match_events`)

회사 프로필 × grant_criteria 평가 결과:

```
per-criterion 평가:
  required  → pass | fail | unknown(기업필드 결측 or text_only)
  exclusion → 해당 시 즉시 ineligible
  preferred → 충족 시 +weight

종합:
  eligibility = ineligible(배제/필수 fail)
              | unknown(필수에 unknown 존재, fail 없음)  ← "조건부 가능, 확인필요"
              | eligible(모든 필수 pass)
  match_score = base(eligible) + Σ(충족 preferred weight) + 기업적합·의미유사 보정 → 0~100 정규화
  rule_trace  = [{criterion_id, kind, result, source_span}]   ← 설명가능
  confidence  = f(criteria.confidence들, profile 필드 confidence들)
  ruleset_ver
```
- `unknown`을 만든 필드가 progressive profiling의 **다음 입력 유도** 대상 ("매출 입력하면 3개 더 확정").
- `rule_trace`가 "왜 추천/탈락"의 근거 = 사용자 신뢰 + 라벨링 단위.

---

## 6. 플라이휠과의 접속

- **추출 단위 라벨링**: `grant_criteria.needs_review` 또는 낮은 confidence → 라벨링 큐 → 사람 정정 → `golden_set`(추출 정답).
- **매칭 단위 라벨링**: 현재 판정은 `match_state`, 사용자 행동은 `match_events`, 명시 피드백은 `feedback`(적합/부적합·신청·선정) → `golden_set`(기업×공고 적격 정답).
- **버전 귀속**: grant은 `model/prompt_ver`, match는 `ruleset_ver/scoring_ver` → 무엇이 품질을 바꿨는지 추적.
- **backfill**: 추출 로직 개선 시 `grant_raw`(R2 원본)로 재추출 → criteria 재생성.

## 6.5 기업 측 스키마 (대칭)
매칭 입력의 나머지 절반인 **`company_profiles`(필드 단위 value/source/confidence/as_of)** 와 계정·회사·권한 모델은 `창업노트_계정권한_개인정보_설계.md`에 정의. grant_criteria와 같은 canonical 값 체계를 공유한다. (명칭은 전 문서 `company_profiles` 복수로 통일.)

## 7. 확정 결정과 남은 튜닝
1. **업종 표준 태그 체계** — KSIC + 정책태그 2층으로 확정. `industry_taxonomy` controlled vocabulary는 버전 관리하며 라벨링으로 보강.
2. **지역 정규화** — 시도+시군구 코드, 권역 전개로 확정. 본사/지사/공장 인정은 criteria의 location_type과 progressive 자가신고로 처리.
3. **업력 기준일** — 공고 명시 기준 우선, 미명시 시 개업일(개인)/법인설립일(법인). criteria.value.basis로 override.
4. **스코어 함수** — 3분해(적합도/선정가능성/가치)와 conditional 하한·상한 표현은 확정. `w_base/w_pref` 등 수치는 골든셋 확보 후 튜닝.
