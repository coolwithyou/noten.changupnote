## 3. 혜택 Taxonomy (6 패밀리 — 데이터 도출, controlled vocabulary)
| 패밀리 | 세부 type | 정량? |
|---|---|---|
| **자금(Capital)** | 보조금·사업화자금·R&D자금·인건비·바우처 | ₩ 정량 |
| **융자·보증(Loan)** | 융자·보증·이차보전 | ₩ 한도 |
| **역량(Capability)** | 컨설팅·멘토링·교육·진단 | 회차/정성 |
| **공간·인프라(Space)** | 입주·시설·보육·시제품 제작 | 면적/기간 |
| **시장(Market)** | 마케팅·판로·전시회·해외진출·바이어매칭 | 부스/건/정성 |
| **인증·IP(Cert)** | 인증취득·특허·지식재산 지원 | 정성 |
| **연결(Connection)** | 네트워킹·IR·투자연계·데모데이 | 정성 |

> 정량(₩·횟수·면적)과 정성(네트워킹 기회)을 **하이브리드**로 표현. 카드의 "보상" 아이템 = 이 type + amount/quantity + certainty.

## 4. 추출 난이도 (정직)
- **혜택은 조건보다 추출이 어렵다.** K-Startup은 구조화 필드에 혜택이 **없고**(짧은 본문에 금액 언급 **4.6%**뿐, 상세는 detl_pg_url에). 기업마당은 본문에 있으나 비정형(**금액 명시 42.9%**).
- → 혜택 `benefits[]` 추출 = 기업마당 본문/HWP + K-Startup 상세페이지에 대한 **LLM 추출**(criteria보다 비용↑). taxonomy(위)가 핵심 자산.
- **금액 커버리지 제한** → "확정 ₩X / 최대 ₩Y / 선정 시" `certainty`로 정직 표기. 절반 이상은 금액 미명시이므로 "금액 미상, 혜택 N종"으로 보여야.

## 5. 설계 반영 (이 분석 → 문서)
- `정규화스키마`: `grants.benefits[]` 신설(type·amount·unit·quantity·certainty·source_span). criteria 아님(표시·가치용).
- `데이터표준화`: 위 6-패밀리 benefit taxonomy = 매핑 자산(`benefit_vocab`, 버전 관리).
- `매칭스코어`: §4 **가치 점수**가 benefits로 실체화(자금 정량 + 혜택 종수/적합 가중).
- `인터페이스`: 카드에 "보상(혜택) 아이템 행" + 의무(자부담·정산·기간) 균형 표기.

## 6. 시퀀싱
- MVP: `support_amount`(금액)만으로 출발 가능.
- Phase 1.5~2: 혜택 풀 아이템화(LLM 추출 + benefit_vocab + 의무 추출). 골든셋으로 추출 품질 검증.
