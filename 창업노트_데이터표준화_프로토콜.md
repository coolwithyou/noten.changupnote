# 창업노트 — 데이터 표준화 프로토콜

작성일: 2026-06-19 · 역할: 수집(추출)과 매칭이 **같은 값**으로 말하게 하는 공통어. `grant_criteria` / `company_profiles`의 값 표준.

## 0. 프로토콜의 구조 (모든 차원 공통)

각 차원(dimension)은 5요소로 정의한다:
1. **Canonical** — 표준 표현(저장 형식)
2. **Company resolver** — 기업 데이터(팝빌 등) → canonical
3. **Grant extractor** — 공고 표현 → canonical (controlled list만, 자유텍스트 금지)
4. **Comparator** — 매칭 비교 규칙 (결측=unknown 처리 포함)
5. **Mapping asset** — 필요한 매핑 테이블(버전 관리, 플라이휠로 개선)

> 매핑 테이블(`industry_taxonomy`, `region_hierarchy`, `size_thresholds`…)은 코드가 아니라 **데이터 자산**이다. 버전을 달고 골든셋으로 개선한다.

---

## 1. 업종 (industry) — 가장 어려움

**문제**: 4개 체계가 따로 논다 — 국세청 업종코드 / KSIC(표준산업분류) / 중소기업기본법 업종 / 공고 자유표현("지식서비스", "딥테크", "소부장", "4차산업", "로봇").

**Canonical (2층 구조)**
- **㉠ KSIC 코드** (대분류 알파벳 + 중분류 2자리) — 객관적 backbone.
- **㉡ 정책 태그(policy tag)** — KSIC로 안 떨어지는 정책 표현의 controlled list. 예: `지식서비스`, `딥테크`, `ICT`, `소부장`, `바이오헬스`, `로봇`, `콘텐츠`, `그린/탄소`, `AI`.

**Mapping asset — `industry_taxonomy`**: `KSIC ↔ policy_tag` (다대다, 버전). 예: KSIC `C29(기타 기계·장비)` ↔ `로봇`,`소부장` / `J62(컴퓨터프로그래밍)` ↔ `ICT`,`지식서비스`,`SW`.

- **Company resolver**: 팝빌 업태·종목/업종코드 → KSIC 정규화 → taxonomy로 policy tag 확장. (기업은 KSIC + 파생 태그를 모두 보유)
- **Grant extractor**: 공고 업종 표현 → controlled list(KSIC 코드 또는 policy tag)로만 매핑. 자유텍스트 금지, 모호하면 `needs_review`.
- **Comparator**: 기업의 (KSIC ∪ 확장 태그) 와 공고 criteria(KSIC/태그)가 교집합 있으면 충족. 공고에 업종 조건 없음 = 전업종 허용.
- **Edge**: 공통 배제업종(유흥·사행 등)은 **전역 hard fail이 아니라 "주의 플래그"**로(공고별 예외 존재 + 근거 source_span 없는 전역배제는 "거짓탈락 금지" 원칙과 충돌). 공고에 배제 명시가 있을 때만 exclusion으로 승격.

## 2. 지역 (region)

**Canonical**: 행정표준코드 — **시도(2자리) + 시군구(5자리)**. 전국 = `[]`(빈값).

**Mapping asset — `region_hierarchy`**: 시군구→시도 롤업 + **권역 매핑**(수도권/비수도권, 호남·영남·충청 등) → 시도 집합 전개.

- **Company resolver**: 팝빌 주소 → 주소 파싱/지오코딩 → 시도+시군구 코드. (팝빌은 보통 **본사** 주소)
- **Grant extractor**: "전남도 소재", "비수도권", "수도권 제외" → 시도 집합 + location_type(본사/지사/공장 허용 여부)으로 추출.
- **Comparator**:
  - criteria가 시도 단위면 기업 시도와 비교, 시군구 단위면 시군구까지.
  - **"본사/지사/공장 인정"**: 기업 데이터엔 본사뿐 → 지사·공장 소재는 **progressive 자가신고**. 미입력 시 본사로만 판정하되, 근접 지역이면 `unknown`+"지사/공장 있나요?" 유도.
  - "비수도권" = 권역 전개 후 not_in 수도권.
- **Edge**: 광역 표현·복수 지역 컨소시엄 → 시도 집합으로 평탄화.

## 3. 업력 (biz_age)

**Canonical**: **개월(정수)**. 계산 기준일과 as_of를 분리 저장.

**기준일(basis) 우선순위**
1. 공고가 명시한 기준 우선 (criteria.value.basis로 포착)
2. 미명시 시 **기본값 = 개업일(개인) / 법인설립일(법인)** — 통상 "사업자등록증상 개업연월일" 기준이 다수.
- value 형태: `{max_months:84, basis:"개업일|법인설립일|사업자등록일"}`

- **Company resolver**: 팝빌 개업일자/설립일자 확보. 개인↔법인 **전환 이력**은 엣지 — 승계 여부가 공고마다 달라 `unknown`+확인.
- **Grant extractor**: "창업 7년 이내", "예비창업자", "업력 3년 미만" → `{max_months}`/예비 세그먼트 + basis 추출.
- **Comparator**:
  - ⚠ **as_of = 공고 기준일(보통 공고일/접수마감일)** 로 계산. *매칭 시점*이 아님 — "공고일 기준 N년 이내"가 표준이라 이걸 틀리면 경계 공고가 다 틀어진다.
  - 기업 개업일 결측 → `unknown`.
- **예비창업(미창업)**: 사업자번호 없음 → **별도 수기 진입경로**(예정 업종·지역·대표자속성 자가신고)로 1차 포함. 같은 매칭엔진 재사용, profile.source=`self_declared`·`is_preliminary=true`로 구분, confidence 낮게. 기창업(번호 자동보강)이 메인, 예비는 얇은 서브.

## 4. 규모 (size)

**Canonical 세그먼트**: `예비 | 소상공인 | 소기업 | 중기업 | 중견 | 대기업`.

**Mapping asset — `size_thresholds`**: 중소기업기본법 **업종별 매출 기준** + 상시근로자, 소상공인 기준(업종별 5/10인 미만). → **industry 의존**(업종마다 임계 다름).

- **Company resolver**: 팝빌 '기업규모' 필드 있으면 1차 사용. 매출·고용 결측 잦음 → 종종 `unknown`(매출·고용은 v1 CODEF 동의연동으로 확정).
- **Grant extractor**: "중소기업", "소상공인", "중견기업 이하" → 세그먼트 집합.
- **Comparator**: 기업 세그먼트 ∈ 허용 집합. 결측 시 `unknown`+"매출/고용 입력 유도".

---

## 5. 대표자 속성 (founder_trait)

**Canonical**: `청년 | 여성 | 장애인 | 시니어 | 국가유공자 | 1인창업`.
- ⚠ **청년 연령 기준이 사업마다 다름**(만34/39 등) → criteria에 `{trait:"청년", age_max:39}`로 **기준 동봉**. 기업 측은 대표자 생년만 보유 → 사업별 기준으로 판정.
- **Resolver**: 대표자 정보(자가신고 또는 등기/팝빌). 속성 대부분 **자가신고+증빙**.
- **Comparator**: 보통 `preferred`(가점)로 등장. 결측 시 가점 미적용(탈락 아님).

---

## 6. 인증 (certification)

**Canonical (controlled list)**: `벤처 | 이노비즈 | 메인비즈 | 여성기업 | 장애인기업 | 사회적기업 | 소셜벤처 | 연구소/전담부서 | ISO… `.
- **Resolver**: 자가신고 + 공개조회(벤처확인종합관리시스템 등) 하이브리드 + 증빙 업로드.
- **Comparator**: required(필수보유) 또는 preferred(가점) 둘 다 등장. 결측 → `unknown`(required일 때) / 미적용(preferred일 때).

---

## 7. 기수혜·배제 (prior_award)

**Canonical**: `{programs:[...], scope:"동일사업|중복지원|연차제한"}`.
- **Resolver**: 데이터 출처 약함 → **자가신고** + (가능 시) 정부 수혜이력. 대체로 `unknown`.
- **Comparator**: `exclusion` — 해당 시 즉시 ineligible. unknown이면 "중복수혜 확인 필요" 경고.

---

## 8. 보조: IP / 재무 (ip / revenue / employees)
- **ip**: KIPRIS(출원인=사업자번호) 조회 가능 → 특허/실용신안 보유. 주로 R&D `preferred`.
- **revenue / employees**: v1 CODEF 동의연동. 그 전엔 자가신고/`unknown`. size·정책자금 판정의 입력.

## 8.5 혜택(Benefit) Taxonomy — `benefit_vocab` (데이터 도출, 2026-06-26 전수분석 근거)
지원사업이 *주는 것*의 표준 분류. 카드 "보상 아이템"·가치 점수의 controlled vocabulary. (출처: `창업노트_공고분석_혜택taxonomy.md`)
| 패밀리 | 세부 type | 정량 |
|---|---|---|
| 자금(Capital) | 보조금·사업화자금·R&D자금·인건비·바우처 | ₩ |
| 융자·보증(Loan) | 융자·보증·이차보전 | ₩ 한도 |
| 역량(Capability) | 컨설팅·멘토링·교육·진단 | 회차/정성 |
| 공간·인프라(Space) | 입주·시설·보육·시제품 제작 | 면적/기간 |
| 시장(Market) | 마케팅·판로·전시회·해외진출·바이어매칭 | 부스/건/정성 |
| 인증·IP(Cert) | 인증취득·특허·지식재산 | 정성 |
| 연결(Connection) | 네트워킹·IR·투자연계·데모데이 | 정성 |
- **Extractor**: 본문/첨부 LLM 추출(기업마당 본문·K-Startup 상세). `certainty`(확정/최대/선정시) 필수 — 금액 명시율 낮음(K-Startup 5%·기업마당 43%).
- **짝**: `obligations`(자부담·정산·사업기간·중복제한) 동시 추출 — 혜택만 강조해 부담 숨기지 말 것.

## 9. 매핑 자산 요약 (버전 관리 대상)

| 자산 | 내용 | 개선 경로 |
|---|---|---|
| `industry_taxonomy` | KSIC ↔ 정책태그 다대다 | 추출 라벨링 시 누락 태그 발견→추가 |
| `region_hierarchy` | 시군구→시도, 권역 전개 | 정적(행정코드 기준), 거의 고정 |
| `size_thresholds` | 업종별 중소·소상공인 임계 | 법령 개정 시 갱신 |
| `default_exclusions` | 공통 배제업종 | 운영 중 보강 |
| `cert_vocab`/`trait_vocab` | controlled list | 신규 인증/속성 등장 시 추가 |

## 10. 확정 결정과 구현 중 튜닝
1. **예비창업 1차 포함** → 번호 없는 별도 수기 진입경로, 같은 엔진 재사용, 기창업 메인·예비 서브.
2. **업종 backbone = KSIC + 정책태그 2층** → `industry_taxonomy` 버전 관리.
3. **청년 등 연령 기준** → criteria에 `{trait:"청년", age_max:39}`처럼 사업별 기준 동봉.
4. **주소 지오코딩 소스 = 행안부 도로명주소 API** → 시도+시군구 코드 산출, 권역은 `region_hierarchy`에서 전개.

구현 중 튜닝할 항목은 controlled vocabulary의 초기 목록, KSIC↔정책태그 다대다 매핑, size threshold 법령 반영 주기다. 이는 결정 대기가 아니라 데이터 자산 버전업 과제다.
